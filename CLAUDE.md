# CLAUDE.md

## О проекте

Локальная POS-система для розничного магазина. На старте — одна точка, одна касса.
Охват: касса (приём товара в чек, оплата, печать) + база товаров + карточки товаров +
заявки на пополнение + приём накладных + возвраты + инвентаризация + отчёты по дню и месяцу.

Полное ТЗ — в `docs/макет-системы.txt`. Это источник истины по требованиям; при
противоречии между кодом и макетом — сверяться с макетом.

**Главное:** система крутит деньги и остатки. Баг в денежном или складском пути = недостача
и кривой склад. Это высший приоритет при любом изменении.

## Стек

- Python 3.12+
- **FastAPI** — веб-слой и API
- **HTMX + Jinja2** — UI. Минимум JavaScript. НЕ тащить React/Vue/Svelte или другой
  тяжёлый фронт без явного решения.
- **SQLModel** — ORM (поверх SQLAlchemy + Pydantic)
- **SQLite** — БД, один файл `data/store.db`
- **pywebview** — нативное окно вокруг приложения
- **python-escpos** — печать чеков и этикеток (протокол ESC/POS)

Принципы выбора, которые не нарушаем без явного решения:
- Остаёмся в Python. UI на HTMX, не на JS-фреймворках.
- Не пишем сырой SQL — только через SQLModel/SQLAlchemy. Это держит будущий переезд на
  Postgres как смену строки подключения, а не переписывание логики.

## Железные правила (деньги и данные)

1. **Деньги и количество — никогда `float`.** float даёт ошибки округления
   (0.1 + 0.2 ≠ 0.3). Разные типы под разные поля:
   - **Деньги** (цена, сумма, сдача) — целое число в минимальной единице (копейках).
     148.50 ₴ хранится как `14850`. Не Decimal, не float — обычный `int`.
   - **Количество** — `Decimal` с фиксированной точностью. Для штучных товаров целое
     (3 шт), для весовых/метражных — дробное (1.5 кг). Дробь здесь обязательна.
   - **Сумма строки чека** = цена × количество, округляется до копейки одним явным
     правилом (например ROUND_HALF_UP), результат — снова `int` в копейках.

   ⚠️ Открытый вопрос (решить при реализации кассы, не сейчас): на весовых товарах
   итоговую сумму строки оставлять с копейками или округлять до целой валюты? На
   структуру хранения это не влияет — в обоих случаях это `int` в копейках, меняется
   только правило округления.
2. **Продажа атомарна.** Сохранение чека + списание остатка + запись движения товара —
   одна транзакция БД. Либо всё целиком, либо ничего. Никаких полусписанных состояний.
3. **Валидация на границе.** Все входные данные (количество, цена, способ оплаты)
   проверяются Pydantic-моделью до попадания в логику. Количество > 0, способ оплаты —
   из перечня.
4. **Завершённый чек не редактируется.** Исправление — только через возврат (макет, разд. 15).
   Возврат возвращает товар в остаток и создаёт чек возврата.
5. **Движение товара пишется на каждое изменение остатка:** продажа, возврат, приход,
   корректировка/инвентаризация. Поля: дата, количество, пометка операции.

## Структура проекта

```
store-pos/
├── CLAUDE.md
├── README.md
├── requirements.txt
├── .gitignore
├── run.py                # точка входа: поднимает FastAPI + открывает окно pywebview
├── app/
│   ├── main.py           # сборка FastAPI-приложения, подключение роутеров
│   ├── config.py         # параметры (вместо экрана настроек — см. макет разд. 27)
│   ├── database.py       # движок и сессии SQLModel
│   ├── models/           # таблицы SQLModel: product, receipt, order, movement…
│   ├── routes/           # эндпоинты: касса, поиск, карточка, заявки, накладные…
│   ├── services/         # бизнес-логика: продажа, возврат, склад, отчёты
│   ├── hardware/         # обёртки над принтерами (ESC/POS)
│   ├── templates/        # HTML + HTMX
│   └── static/           # css, минимум js
├── data/
│   └── store.db          # файл SQLite (в .gitignore)
├── docs/
│   └── макет-системы.txt  # ТЗ, источник истины
└── tests/                # тесты денежного и складского пути
```

`services/` отделён от `routes/`: денежная и складская логика живёт отдельно от веб-обвязки,
чтобы её можно было тестировать без запуска интерфейса.

## Команды

- Установка зависимостей: `pip install -r requirements.txt`
- Запуск приложения: `python run.py` (FastAPI + окно pywebview)
- Только сервер, для отладки в браузере: `uvicorn app.main:app --reload`
- Тесты: `pytest`

## Железо

- **Сканер QR/штрихкодов** — работает как клавиатура (HID). Драйверов и интеграции не
  требует: «печатает» код в активное поле ввода и шлёт Enter. Обрабатывать как обычный
  текстовый ввод. На выбор стека не влияет.
- **Принтер чеков и принтер этикеток** — ESC/POS через python-escpos. Печать идёт из Python
  напрямую в устройство, НЕ из браузера. Браузер только дёргает локальный эндпоинт.
- **Платёжный терминал и весы — НЕ интегрированы** (макет, разд. 21, 22). Способ оплаты и вес
  продавец вводит вручную; система их только фиксирует в чеке.

## База данных

- Сейчас SQLite, один файл `data/store.db`. Для одной кассы этого достаточно: полноценные
  ACID-транзакции, бэкап = копия файла.
- Логику не привязывать к диалекту SQLite. Переход на Postgres понадобится при появлении
  нескольких касс на общую базу или второго магазина и должен сводиться к смене строки
  подключения.
- `data/store.db` — в `.gitignore`, в репозиторий не попадает.
