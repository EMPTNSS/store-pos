## 1. Что требовалось

Спроектировать и поднять хранение данных о товарах: три таблицы (`product`, `movement`,
`price_history`), корректные типы полей (деньги — `int` в копейках, количество — `Decimal`),
правило генерации числового кода от растущего счётчика и правило архивации вместо
физического удаления. Таблицы должны создаваться **существующим** механизмом `init_db()`
(этап 0.1), без своего подключения. Доменной логики (запись движений при продаже/приёме,
генерация кода) на этом этапе нет — только структура хранения.

---

## 2. Решения по реализации

1. **Счётчик кодов — отдельная таблица `product_code_counter`, а не `MAX(numeric_code)+1`.**
   ТЗ требует «постоянно растущий счётчик», чтобы номер не переиспользовался после
   архивации товара. `sqlite_sequence` не трогаем (это сырой SQL и привязка к диалекту) —
   завели явную однострочную модель SQLModel.
2. **Точность `Decimal` вынесена в `app/models/types.py`** (`QUANTITY_PRECISION=10`,
   `QUANTITY_SCALE=3`, хелпер `quantity_column()` → `Numeric(10, 3)`). Три знака после
   запятой — осознанный потолок, заданный в одном месте; `product.py` и `movement.py`
   импортируют хелпер, чтобы точность не разъезжалась по моделям.
3. **`numeric_code` без автогенерации в модели.** Поле — `str NOT NULL UNIQUE`; сама
   генерация (инкремент счётчика в транзакции) — задача сервисного слоя этапа 0.3.
   В тестах код задаётся вручную строкой (`"00001"`, `"00002"`).
4. **Засев счётчика в `init_db()` — идемпотентный.** `init_db()` зовётся при каждом старте;
   вставка `ProductCodeCounter(id=1, last_value=0)` выполняется только если строки ещё нет
   (`session.get(..., 1) is None`). Повторный старт не плодит дубль и не сбивает счётчик.
5. **`created_at` через `default_factory=datetime.now`, а не `default=datetime.now()`.**
   Со скобками значение вычислилось бы один раз при импорте модуля и все товары получили бы
   одинаковое время. `default_factory` берёт время на каждую вставку. `updated_at` не
   заводили — пока не нужен (добавлено коммитом `d366bad`).

## 5. Хранимое против вычисляемого (ТЗ §2)

Заведены **только хранимые** поля. Вычисляемые значения сознательно **не** хранятся
отдельными колонками и придут запросами на следующих этапах:

- текущий остаток — сумма `movement.quantity` (кэш в `quantity_current`, обновляется
  транзакцией движения — этапы 1, 5, 6);
- разница в деньгах и наценка в % — из `price_sell`/`price_buy` на лету;
- продано/приход/убытие — запросы к `movement`;
- чистая прибыль — из чеков (закупочная цена фиксируется в строке чека, этап 1.3).

«Нет в наличии» — не статус, а `quantity_current = 0`. Удаление товара запрещено: только
перевод в `архивный`, чтобы не рвать ссылки чеков/движений и не дать переиспользовать номер.

## 7. Соблюдение правил CLAUDE.md

- **Деньги — `int` в копейках** (`price_sell`, `price_buy`, оба поля `price_history`);
  **количество и мин. остаток — `Decimal`** (`Numeric(10, 3)`); `float` нигде нет.
- **Коды — строки.** `article`, `numeric_code`, `qr_code` — `str`; ведущие нули и буквы
  сохраняются, арифметики над ними нет.
- **Сырой SQL не писали** — только модели SQLModel; специфика SQLite не протекает в логику,
  переезд на Postgres остаётся сменой `database_url`.
- **Движение — на каждое изменение остатка:** структура `movement` заложена; сама запись
  движений — следующие этапы.
- **Завершённый чек/удаление:** товар не удаляется физически, только `архивный`.
- Код, имена, коммиты — английский; тексты ТЗ — русский.
