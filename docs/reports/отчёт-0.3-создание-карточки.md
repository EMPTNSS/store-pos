## 1. Что требовалось

Первая операция записи в базу. Минимальный способ завести товар: форма → эндпоинт →
сервис, создающий товар **одной транзакцией** вместе со всей сопутствующей историей —
числовой код, начальное движение-приход, первая точка истории цен (ТЗ разд. 1, 7).
В 0.2 были только таблицы; здесь впервые пишется бизнес-логика записи.

---

## 2. Согласованные с заказчиком решения

Перед реализацией сверили план; ключевая развилка закрыта явным решением владельца:

1. **Модель `Movement` не трогаем.** Цену закупки в движение **не** добавляем — снимок
   закупочной цены живёт только в `price_history` (ТЗ разд. 8.1). Движение хранит лишь
   дату, количество и тип операции.
2. **Дубль `qr_code`** ловится как `IntegrityError` и возвращается формой с ошибкой
   (статус 422), а не 500. Невалидная цена/количество — это `ValidationError` на границе.
3. **`now()` берётся один раз** на всю транзакцию и проставляется в `created_at`, движение
   и историю цен (одинаковый штамп времени), naive `datetime`.
4. **Тестовый счётчик** засевается через `init_db()` из 0.2 (фикстура `db`).

## 6. Критерии приёмки (раздел 11 ТЗ)

| # | Критерий | Статус |
|---|---|---|
| 1 | Валидный вызов создаёт `product` с верными полями | ✅ |
| 2 | `numeric_code` непустой, `UNIQUE`, растёт при двух подряд | ✅ |
| 3 | `status` нового товара = активный | ✅ |
| 4 | Ровно одна `movement` типа `приход`, верное `quantity`, цены в движении нет | ✅ |
| 5 | `quantity_current` равен введённому количеству | ✅ |
| 6 | Одна `price_history` с верными `price_buy`/`price_sell` (копейки) и `datetime` | ✅ |
| 7 | `quantity` = 0 / отрицательное → отказ, в БД ничего | ✅ |
| 8 | Пустое `name` → отказ | ✅ |
| 9 | `unit` не из enum → отказ | ✅ |
| 10 | Пропущенные `article`/`min_stock` → товар создан, `min_stock` = 0 | ✅ |
| 11 | Сбой любого шага сервиса → полный rollback (нет товара/движения/кода) | ✅ |
| 12 | Счётчик инкрементится в той же транзакции, берётся через `with_for_update()` | ✅ |

Сверх ТЗ: дубль `qr_code` через HTTP → 422 с формой (не 500); smoke `POST /products` → 303.

## 8. Соблюдение правил CLAUDE.md

- **Деньги — `int` в копейках** (правило 1): конвертация валюты на границе через `Kopecks`,
  float внутрь логики не проникает. **Количество — `Decimal`**.
- **Создание атомарно** (правило 2): один `commit`, любой сбой → `rollback`, нет
  полусозданных состояний и израсходованного кода.
- **Валидация на границе** (правило 3): `ProductCreate` отрабатывает до сервиса,
  `quantity` строго > 0.
- **Движение на каждое изменение остатка** (правило 5): начальный приход пишется всегда.
- Сервис отделён от роута; сырого SQL нет; диалект SQLite не торчит в логике
  (`with_for_update()` заработает на Postgres как настоящая блокировка).
- Код/имена — английский, текст формы — русский.
