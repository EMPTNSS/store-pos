---

## 1. Что требовалось

Поверх готовой корзины (1.1) дать удобные способы положить товар в чек: сканер (разд. 2),
ручной поиск по названию/коду (разд. 3), ручной ввод веса для весовых товаров (разд. 22).
Границы 1.1 сохраняются: чек — живое состояние в памяти, склад не трогается, продажа не
сохраняется (это этап 1.3).

---

## 2. Согласованные с заказчиком решения

1. **Вес весовых товаров** — отдельного шага ввода веса нет: товар добавляется с количеством 1
   и правится в inline-поле строки (механика 1.1).
2. **Организация ввода** — поиск сверху экрана (список совпадений с кнопкой «В чек»); сканер
   считывает и добавляет сразу по Enter, без нажатия кнопок.
3. **Быстрое поле/сканер** — точное совпадение по `numeric_code` **или** `qr_code`.
4. **Сканер как устройство для этапа не нужен** — HID-клавиатура, путь скана ≡ ручной ввод кода
   в поле; всё проверяется в браузере/тестах без железа.

## 6. Ключевое инженерное решение

**Поиск фильтруется в Python (`casefold`), а не через SQLite `LIKE`/`lower`.** Последние
регистронезависимы только для ASCII — «Молоко» и «молоко» через `LIKE` не совпадут. Отбор в
памяти корректен для кириллицы и не привязан к диалекту БД (CLAUDE.md). Для одной точки с
небольшим каталогом это приемлемо; при росте каталога/переезде на Postgres поиск переносится
на сторону БД — отмечено в ТЗ как точка переделки.

## 8. Граница и что дальше

Не делалось в 1.2 (по ТЗ): завершение продажи, способ оплаты, сохранение чека, списание остатка,
движение товара (всё — этап 1.3, одной транзакцией); печать (этап 2); сортировка поиска по
статусу (этап 8); поведение «товар не найден» и отрицательные остатки (отложено).

Следующий пункт roadmap — **1.3 «Завершение продажи»**: `Receipt`/`ReceiptLine`, оплата,
сохранение чека + списание остатка + движение товара одной транзакцией. Ядро денежного пути.
