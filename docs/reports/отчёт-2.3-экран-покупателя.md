# Отчёт — этап 2.3 «Экран покупателя»

> Дата: 2026-07-03. Ветка: `main`.
> ТЗ: [`docs/specs/тз-2.3-экран-покупателя.md`](../specs/тз-2.3-экран-покупателя.md).
> Roadmap: этап 2.3 — последний пункт этапа 2 «железо вокруг продажи». **Конец v1.**

---

## 1. Что требовалось

Второй экран, обращённый к покупателю (макет разд. 20): во время продажи крупно и коротко
показывать товары, количество, цену и итог (20.2, 20.3), после оплаты — «Спасибо за покупку»
(20.4). Экран — пассивное зеркало кассы: только читает состояние, ничего не меняет в корзине,
БД и складе.

---

## 2. Согласованные с заказчиком решения

1. **Обновление — HTMX-опрос ~1с** (`hx-trigger="load, every 1s"`), чистый HTMX без JS-фреймворков
   и без SSE (CLAUDE.md: минимум JS). Интервал вынесен в конфиг.
2. **Второе окно — по флагу**: роут `/customer` доступен всегда (проверяется в браузере на втором
   мониторе), окно pywebview добавлено в `run.py`, но по умолчанию выключено
   (`customer_display_window=False`) — включается там, где есть реальный второй экран.
3. **После оплаты — «Спасибо за покупку» N секунд, затем экран ожидания** (самосброс по таймауту).

## 5. Что переиспользовано (не писал заново)

- **Синглтон корзины `get_cart()` (1.1)** — экран читает `get_cart().view()` (тот же `CartView`),
  межпроцессного обмена нет: оба окна в одном процессе.
- **`format_money` / `format_quantity` (`money.py`, 2.1/2.2)** — те же фильтры для сумм и количества,
  формат «148.50» / «1.5» единый с кассой и печатью.
- **`Receipt` (1.3)** — источник данных маркера (`receipt_number`, `total`).
- **Паттерн роутера/шаблонов кассы** (`Jinja2Templates` + `TemplateResponse(request, name, ctx)`,
  фрагмент под опрос) — воспроизведён, не переписан.
- **Фикстуры `db`/`client`, `_reset_cart`, изоляция принтеров, `_make_product`** — в тестах.

## 6. Ключевое инженерное решение

**Маркер последней продажи как отдельное состояние.** Пост-продажный «успех» кассы эфемерен —
живёт только в одном HTMX-ответе (`sale_result`), а корзина после продажи очищается. Чтобы экран
покупателя показал благодарность уже после очистки, введён отдельный синглтон-маркер `LastSale`,
который ставится в кассе **после** `commit` (по образцу best-effort печати 2.1). Решение
«что показать» вынесено в **чистую** `resolve_display_state(...)` с внешним `now`: приоритет —
продажа (непустой чек перекрывает благодарность → новый покупатель), благодарность
самосбрасывается по `thanks_seconds`. Ветки таймаута тестируются подстановкой времени, без подмены
системных часов и без веба. Экран строго read-only — гонок с кассой нет (один процесс, GIL).

## 7. Критерии приёмки (выполнены)

`pytest` — **140 passed** (13 новых в `test_customer_display.py`, регрессия 1.1/1.3/2.1/2.2 цела):

- **Резолвер**: непустой чек → `sale`; пустой + свежий маркер → `thanks`; граница таймаута
  (`< thanks_seconds` → `thanks`, `>=` → `idle`); без маркера → `idle`; приоритет продажи над
  свежим маркером.
- **Маркер**: `mark_sale_completed` сохраняет `number`/`total`; `reset_last_sale` чистит.
- **HTTP**: `GET /customer` → 200, есть `hx-get="/customer/state"` и интервал `every 1000ms`;
  `/customer/state` при непустом чеке → строки + «Итого»; после `POST /cashier/complete` →
  «Спасибо за покупку»; пустой чек без продажи → приветствие.
- **Регрессия**: продажа, атомарность, печать чека и накладной не затронуты — `mark_sale_completed`
  не влияет на транзакцию и `sale_result`.

Ручная проверка: `uvicorn app.main:app --reload`, открыть `/cashier` и `/customer` в двух вкладках —
пробитие товара отражается на `/customer` в пределах ~1с, завершение продажи показывает
«Спасибо за покупку», через 8 сек — приветствие.

**v1 завершён.** Касса продаёт, печатает чек, формирует накладную и показывает покупателю
