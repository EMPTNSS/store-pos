# Отчёт — этап 2.5 «Навигационная оболочка (вкладки)»

> Дата: 2026-07-06. Ветка: `main`.
> ТЗ: [`docs/specs/тз-2.5-навигационная-оболочка.md`](../specs/тз-2.5-навигационная-оболочка.md).
> Roadmap: этап 2.5 — мост между рабочей кассой (v1) и разделами 3–7. Строим **раму**, не наполнение.

---

## 1. Что требовалось

Рама вкладок в одном окне pywebview: касса — постоянная стартовая вкладка, рабочие разделы
(Товары, Заявки, Добавить, Чеки за день) открываются во вкладки-заглушки — **одна вкладка на
раздел**, повторный клик активирует уже открытую, крестик закрывает. Переключение вкладок не
теряет введённое в поля. Тонкий слой JS только для show/hide и локального состояния; бизнес-логики
в нём нет. Разделы — пустые заглушки (наполнение на своих этапах). Бэкенд и экран покупателя не
трогаются.

---

## 2. Итог изучения кода (до реализации)

**Разворот лёгкий:** `base.html` уже был, экран кассы уже блок (`extends base.html`), а не
самостоятельная страница. Встраивание в раму свелось к выносу контента кассы в partial + include.
**Побочная находка:** окно pywebview в `run.py` открывает корень `/`, но роута `/` не было
(в `main.py` только `/products`, `/cashier`, `/customer`, `/health`) — при запуске окно упиралось
в 404. Рама заняла `/` и закрыла эту дыру.

---

## 3. Ключевые решения

1. **Сохранность состояния = show/hide, а не перезагрузка.** Требование «ушёл на вкладку и
   вернулся — введённое на месте» выполнимо только если панели **живут в DOM**, а JS переключает
   видимость классом `.active`. Перезагрузка контента на каждый клик пересоздавала бы DOM и теряла
   ввод. HTMX грузит контент раздела **один раз** при открытии, дальше — чистое переключение.
2. **Касса встроена в раму на сервере (`{% include %}`), а не по HTMX.** Постоянная вкладка без
   перезагрузки; роуты и логика кассы (`/cashier/*`) не тронуты — «встроить шаблон в раму» буквально.
3. **Вкладки сверху**, ванильный JS (не Alpine) — лишняя CDN-зависимость ради show/hide не нужна.
4. **Белый список разделов + 404** на `/panels/{key}`: заглушки — чистая презентация, изолированные id.
5. **Черновик теряется при закрытии вкладки осознанно** (roadmap): сохранность данных — сервер,
   вкладки — удобство. Автосохранение черновиков не делаем.

## 7. Критерии приёмки (выполнены)

`pytest` — **161 passed** (9 новых в `test_shell.py`, регрессия 0–2.3 цела):

- **Рама**: `GET /` → 200, содержит `#tabs`, `#launcher`, `/static/shell.js`, встроенную кассу
  (`#panel-cashier`, `#cart`), все разделы белого списка.
- **Панели**: `/panels/add` содержит `add-test-field`; `/panels/{products,orders,receipts}` — нет;
  все — заглушки «в разработке»; `/panels/bogus` → 404.
- **Статика/регрессия**: `/static/shell.js` → 200 (есть `openSection`); standalone `/cashier` → 200
  (не сломан выносом в partial).

**Ручная проверка** (uvicorn + засев 5 демо-товаров): рама на `/` рендерит рабочую кассу; поиск по
названию/коду находит товары; разделы открываются во вкладки; сценарий сохранности (ввод в тестовое
поле «Добавить» → уход на кассу → возврат → текст на месте) — глазами по логике `shell.js`. Демо-база
после проверки удалена.

**Не покрыто автотестами:** интерактивный слой на стороне `shell.js` (переключение вкладок,
сохранность поля, «одна вкладка на раздел», крестик) — TestClient JS не исполняет, Playwright в
окружении нет. Логика изолированная и простая; проверяется запуском.

## 9. Итог

Рама вкладок работает в одном окне: касса — постоянная стартовая вкладка со встроенным рабочим

минимальный роутинг под панели. Заодно закрыт 404 корня `/`, куда указывает окно pywebview.
