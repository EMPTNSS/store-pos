# Отчёт — этап 3.2 «История движений и статистика товара»

> Дата: 2026-07-06. Ветка: `main`.
> ТЗ: [`docs/specs/тз-3.2-история-движений.md`](../specs/тз-3.2-история-движений.md).
> Roadmap: этап 3.2 — история движений (разд. 5.7) и статистика товара (разд. 5.4):
> дата, количество, пометка операции; продано за всё время, приход/убытие по датам, чистая прибыль.

---

## 1. Что требовалось

Наполнить заглушку, оставленную 3.1 в карточке товара: секция с **окном истории движений** (5.7 —
дата, операция, количество) и **статистикой** (5.4 — продано за всё время, прибыло/убыло по датам,
чистая прибыль). Этап только **читает** уже копящиеся данные (`movement`, `receipt_line`,
`price_history`), хранилище не меняет.

---

## 2. Зафиксированные решения (согласованы до реализации)

1. **Себестоимость проданного — «as-of» по `price_history`.** Чистая прибыль считается по закупочной
   цене, действовавшей на момент **каждой** продажи, а не по текущей `Product.price_buy`. Правка
   закупки задним числом не искажает прибыль прошлых продаж.
2. **Секция с ленивой HTMX-подгрузкой.** История и статистика догружаются отдельным запросом
   (`hx-trigger="load"`), а не рендерятся вместе с карточкой.

---

## 3. Ключевые решения реализации

1. **Два источника данных.** Количество (продано / приход / убытие по датам, окно истории) — из
   `movement` как единого журнала изменений остатка. Деньги (чистая прибыль) — из `receipt_line`
   (выручка) + `price_history` (себестоимость as-of), потому что `Movement` не связан с чеком и денег
   не хранит.
2. **`buy_price_asof` — точка `price_history` с `datetime ≤` момента продажи.** Fallback на текущую
   `price_buy`, если точек нет (не должно случаться — первая точка пишется при создании товара).
3. **Реюз `line_total`** (`app/services/money.py`) для себестоимости строки — то же правило
   округления ROUND_HALF_UP, что и на кассе; прибыль сходится с денежным путём.
4. **Ленивый контейнер перезагружается сам.** Карточка перерисовывается `outerHTML` при корректировке
   остатка → `hx-trigger="load"` срабатывает заново, история освежается без ручного обновления.
5. **Обобщённость по типам операций.** Шаблон и статистика не хардкодят перечень: тип `возврат` (enum
   есть, пишется с 4.1) и массовый приход (6.1) отобразятся без правок этого этапа.

## 7. Критерии приёмки (выполнены)

`pytest` — **196 passed** (12 новых в `test_product_movements.py`, регрессия 0–3.1 цела):

- **История:** `product_movements` — порядок по убыванию `datetime`, содержит приход/продажу/
  инвентаризацию; товар без движений (вставлен минуя `create_product`) → пустой список.
- **Количество:** `sold_total` = Σ проданного; `by_date` — верная разбивка приход/убытие по датам
  (приход и излишек → income, продажа и недостача → outgoing).
- **Прибыль (as-of):** продажа при закупке A → правка закупки на B → вторая продажа; себестоимость
  первой по A, второй по B (ключевой тест). Нет продаж → 0; нулевая закупка → прибыль = выручке.
- **`buy_price_asof`:** последняя точка с `datetime ≤ at`; раньше всех точек и без точек → `price_buy`.
- **HTTP:** `GET /products/{id}/movements` → 200 со статистикой и историей; несуществующий id → 404;
  товар без движений → пустое состояние.

**Живая проверка (изолированная временная БД, TestClient):** товар → продажа 3 шт при закупке 40.00 →
поднятие закупки до 50.00 → продажа 2 шт → фрагмент показал **чистую прибыль 105.00 ₽** (75.00 по
первой продаже + 30.00 по второй — прямое доказательство расчёта себестоимости as-of, а не по текущей
цене), «прибыло +20 / убыло −5» по дате, три строки истории от новых к старым. Карточка содержит
ленивый контейнер `#product-movements` с `hx-trigger="load"`. На запущенном сервере роут отвечает:
несуществующий товар → 404, панель «Товары» и оболочка → 200.

**Не покрыто автотестами:** фактическое исполнение `hx-trigger="load"` браузером (TestClient JS не
исполняет). Паттерн идентичен остальному HTMX в проекте; проверен рендер самого фрагмента.

## 8. Технический долг / отложено

- **Возвраты (4.1)** — тип `возврат` в enum есть, но пока не пишется; история/статистика покажут его
  обобщённо, уменьшение прибыли на сумму возврата — доработка 4.1.
- **Графики динамики цен (5.6)** — этап 3.3 (`price_history` уже наполняется; заглушка про графики
  оставлена в карточке).
- **Пагинация истории** — для одной кассы объём мал, показываем все движения; пагинация/лимит — при
  росте объёма.
- **Миграции** — как и в 3.1, схема через `create_all` без Alembic; здесь новых полей нет, долг прежний.

## 9. Итог

Карточка товара получила историю и статистику: под паспортом ленивой подгрузкой открывается секция с
окном движений (дата · операция · количество) и сводкой (продано за всё время, приход/убытие по датам,
чистая прибыль). Прибыль считается по закупочной цене на момент каждой продажи через `price_history` —
устойчиво к последующей правке закупки. Хранилище не тронуто, денежный путь покрыт тестами; касса,
продажа, печать и карточка 3.1 не задеты.
