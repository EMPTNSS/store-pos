# Отчёт — этап 4.2 «Исправление завершённого чека через возврат»

> Дата: 2026-07-07. Ветка: `main`.
> ТЗ: [`docs/specs/тз-4.2-исправление-чека-через-возврат.md`](../specs/тз-4.2-исправление-чека-через-возврат.md).
> Roadmap: этап 4.2 (разд. 15.2–15.3) — завершённый чек напрямую не правится, только через
> возврат, привязанный к чеку-первоисточнику. Опирается на 4.1 (возврат) и 1.3 (`Receipt`).

---

## 1. Что требовалось

Дать продавцу путь **исправления завершённого чека через возврат** (разд. 15.2–15.3). Сам чек
редактировать нельзя (правило 4). Вместо правки продавец находит завершённый чек по номеру,
видит его строки **только для чтения** с пометкой «сколько ещё можно вернуть», переносит
ошибочную позицию в корзину возврата **с ценой из чека** — и дальше проверенным путём 4.1
одной транзакцией оформляет чек возврата. Новое сверх 4.1: **привязка** возврата к
первоисточнику и **защита от перевозврата** (по строке чека Σ возвращённого ≤ проданного).

---

## 2. Зафиксированные решения (согласованы до реализации)

1. **Завершённый чек не правится** (правило 4, разд. 15.2). 4.2 не добавляет редактирования
   продажного `Receipt`/`ReceiptLine`; строки чека — read-only, ошибочная позиция переносится
   в корзину возврата.
2. **Цена корректирующей строки зафиксирована по чеку** (снимок `ReceiptLine.price_sell`), не
   редактируется — в отличие от свободной строки 4.1.
3. **Без округления** (единообразно с 4.1): даже при полном возврате чека надбавка округления
   итога (`Receipt.rounding`) покупателю не возвращается.

---

## 3. Ключевые решения реализации

1. **Аддитивное расширение, свободный возврат 4.1 не тронут.** Две nullable-колонки:
   `ReturnReceipt.source_receipt_id` (FK → `receipt.id`) и `ReturnReceiptLine.source_line_id`
   (FK → `receipt_line.id`). Свободный возврат остаётся с `NULL`; продажный `Receipt` не
   мигрируется.
2. **Инвариант перевозврата — в транзакции, не только в UI.** В `complete_return` до любых
   вставок для каждой привязанной строки проверяется
   `already_returned(source_line_id) + quantity ≤ ReceiptLine.quantity`, иначе `ValueError` и
   откат. Защита от гонок и устаревшего черновика — критичный складской путь.
3. **`already_returned` суммирует в Python, не в SQL.** Σ `quantity` по `source_line_id`
   считается перебором строк (Decimal-безопасно, без зависимости от того, как БД агрегирует
   `SUM` над Decimal-типом количества).
4. **Один возврат = один чек.** Корзина держит `source_receipt_id`; `add_from_receipt_line`
   запрещает смешивать свободные и корректирующие строки и строки из разных чеков (`ValueError`),
   а свободный `add` запрещён при активном корректирующем черновике. Мёрж повторной строки —
   по `source_line_id` (один товар может стоять в чеке несколькими строками), не по `product_id`.
5. **Guard цены на двух уровнях.** `set_price` для `price_locked`-строки кидает `ValueError`;
   роут `POST /returns/items/{id}/price` ловит его и показывает «Цена по чеку не редактируется».
6. **«Доступно» учитывает и черновик.** В `POST /returns/from-receipt` доступное =
   продано − уже возвращено (в БД) − уже набрано в корзине по этой строке, чтобы UX-отказ
   совпадал с авторитетным инвариантом сервиса.

## 7. Критерии приёмки (выполнены)

`pytest` — **243 passed** (18 новых в `test_return_correction.py`, регрессия 0–4.1 цела, было 225):

- **Lookup / доступное:** `already_returned` = Σ по `source_line_id`; `returnable_lines` даёт
  `available = sold − returned`; несуществующий номер → сообщение.
- **Корзина:** `add_from_receipt_line` берёт цену из `ReceiptLine.price_sell`, ставит
  `source_line_id`/`price_locked`/снимок; повторное добавление мёржит; `set_price` по locked →
  `ValueError` (и цена не «едет» при подорожании карточки); смешивание свободных/корректирующих
  и разных чеков → `ValueError`.
- **Сервис:** пишет `source_receipt_id`/`source_line_id`, остаток `+= quantity`, движение
  `возврат`, `total = Σ`; перевозврат заблокирован (ничего не сохранено, остаток цел);
  частичный → остаток → третий сверх проданного отклонён; полный возврат чека с `rounding>0` →
  итог = `subtotal`, без надбавки.
- **Регрессия 4.1:** свободный возврат с редактируемой ценой работает; `source_*` = `NULL`.
- **HTTP:** `GET /returns/receipt` найденного/ненайденного; `POST /returns/from-receipt`
  (locked-строка / отказ при `quantity > available`); `POST /returns/complete` корректирующего
  → `ReturnReceipt` с `source_receipt_id`, баннер с № чека продажи; `POST .../price` по locked →
  сообщение, цена цела.

**Живая проверка (TestClient на реальном ASGI, временная БД):** продажа 3×15.50 (чек с
`rounding=50`) → lookup показывает строки с «Доступно» → перевозврат 5>3 отклонён при добавлении
→ перенос 2 шт с locked-ценой (🔒) → правка цены по чеку отклонена → проведение с баннером
«…к чеку продажи №1» → в БД `source_receipt_id=1`, `source_line_id=1`, `total=3100` (без
округления), остаток 50−3+2=49. Все 8 шагов зелёные.

**Не покрыто автотестами:** фактическое исполнение show/hide оверлея браузером (JS в TestClient
не исполняется) — паттерн тот же, что в 4.1/2.5.

  и аддитивны, логику 4.1 не задевают.
- **Печать чека возврата** (ESC/POS) — как и в 4.1, только сохранение документа; печать поверх,
  в изолированном `app/hardware`.
- **Список чеков / отчёт возвратов за день** — этап 7. Здесь только точечный поиск по номеру.

## 9. Итог

Появился корректный способ исправить завершённый чек, не редактируя его: продавец находит чек по
номеру, видит его строки read-only с «доступно к возврату» и переносит ошибочную позицию в
возврат с ценой из чека. Возврат навсегда связан с первоисточником (`source_receipt_id`/
`source_line_id`), а инвариант в транзакции не даёт вернуть больше проданного — критичный
складской и денежный путь защищён на уровне сервиса, а не только UI. Свободный возврат 4.1 не
тронут (nullable-колонки, регрессия цела, **243 passed**). **Пункт 4.2 закрыт**; этап 4
(возвраты и исправления) завершён.
