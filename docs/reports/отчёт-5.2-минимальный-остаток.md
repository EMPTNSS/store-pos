# Отчёт — этап 5.2 «Минимальный остаток»

> Дата: 2026-07-08. Ветка: `main`.
> ТЗ: [`docs/specs/тз-5.2-минимальный-остаток.md`](../specs/тз-5.2-минимальный-остаток.md).
> Roadmap: этап 5.2 (разд. 10) — отслеживание, уведомление при достижении, отображение в
> карточке. Опирается на 3.1 (`product_view`, карточка `_card.html`, поиск раздела «Товары»).

---

## 1. Что требовалось

Дать продавцу контролировать заканчивающиеся товары (макет разд. 10). Система отслеживает
текущий остаток относительно заданного минимума и, когда остаток достигает минимума, помечает
это — в карточке товара (разд. 10.5) и в результатах поиска раздела «Товары». Пометка =
«уведомление» разд. 10.4 в исполнении заказчика.

---

## 2. Зафиксированные решения (согласованы до реализации)

1. **Уведомление = визуальные пометки в карточке и поиске «Товары».** Макет (разд. 10.4 п.4)
   канал не задаёт; заказчик выбрал минимальный вариант **без** центра уведомлений,
   счётчика-бейджа в раме и отдельной вкладки «Ниже минимума».
2. **Список кандидатов на заказ — этап 5.3, не здесь.** Разд. 10.4 п.5 (добавление в заявки)
   закрывает 5.3; 5.2 только обнаруживает и показывает.
3. **Порог включителен (`<=`).** «Достигает минимума» покрывает равенство.
4. **`min_stock == 0` = «минимум не задан» → отслеживание выключено.** Ноль — отсутствие
   порога, а не «заказывать всегда».
5. **Флаг вычисляется на лету, не хранится** — консистентно с «наличием»/«наценкой» в
   `product_view`; правка минимума или остатка сразу переоценивает флаг без миграций.
6. **Касса (денежный путь) визуально не трогается** — контроль минимума живёт в разделе
   «Товары» и карточке.

---

## 3. Что сделано

- **Обнаружение** — новый pure-хелпер `is_low_stock(product) -> bool`
  ([app/services/product_service.py](../../app/services/product_service.py)):
  `min_stock > 0 and quantity_current <= min_stock`. Без session, сравнение `Decimal` —
  работает одинаково для штучных и весовых товаров.
- **Флаг во view** — в `product_view` добавлен ключ `"low_stock": is_low_stock(product)`.
- **Карточка** ([app/templates/products/_card.html](../../app/templates/products/_card.html)) —
  при `view.low_stock` у строки «Количество» показывается пометка
  «⚠ достигнут минимальный остаток» (разд. 10.5, 3-й пункт); «Мин. остаток» при нулевом
  пороге показывается как «не задан». Добавлен CSS бейджа `.low-stock`.
- **Поиск раздела «Товары»** — роут `search_for_card`
  ([app/routes/products.py](../../app/routes/products.py)) отдаёт пары `(product, low_stock)`
  (критерий считается в роуте, не дублируется в шаблоне); шаблон
  [_search_results.html](../../app/templates/products/_search_results.html) рисует бейдж
  «⚠ ниже минимума». CSS `.low-stock` добавлен в панель
  [_panel.html](../../app/templates/products/_panel.html).
- **Новый тест** `tests/test_min_stock.py`.

**Не менялись:** модель `product.py` (поле `min_stock` уже было с 0.2), схемы `product.py`,
касса (`cashier/*`). Новых таблиц/роутов/миграций нет.

## 4. Тесты (`tests/test_min_stock.py`)

Покрыты критерии приёмки ТЗ (разд. 8):

- **Критерий `is_low_stock`:** ниже минимума → True; равно минимуму → True (включительно);
  выше → False; `min_stock == 0` → False даже при нулевом остатке; весовой товар (`Decimal`);
  нулевой остаток при заданном минимуме → True.
- **View:** `product_view["low_stock"]` совпадает с `is_low_stock` для низкого и нормального.
- **HTTP-карточка:** низкий товар → пометка присутствует; нормальный → нет; нулевой минимум →
  «не задан» без пометки.
- **HTTP-поиск «Товары»:** низкий → бейдж «ниже минимума»; нормальный → нет.
- **Переоценка на лету:** поднятие минимума через `POST /products/{id}` → перерисованная
  карточка помечает низкий остаток.
- **Регрессия кассы:** `/cashier/search` по низкому товару не содержит «ниже минимума».

Прогон:

```
python -m pytest tests/test_min_stock.py -q   →  14 passed
python -m pytest -q                            →  266 passed
```

Регрессия зелёная: складской и денежный путь не затронуты.

## 5. Технический долг / на будущее

- **Заявки на пополнение** (разд. 11) — этап **5.3**: авто-кандидаты при достижении минимума
  (разд. 10.4 п.5) + ручное добавление, группировка по поставщику. `is_low_stock` — готовая
  точка обнаружения для формирования списка кандидатов.
- **Разделение «товар закончился» vs «ниже минимума»** (разд. 11.2) — при необходимости в
  5.3; сейчас закончившийся товар с заданным минимумом трактуется как частный случай «ниже
  минимума».
- **Порог/уведомление на кассе** — намеренно не добавлено; при появлении потребности денежный
  путь можно расширить пометкой через тот же `is_low_stock`, не меняя логики.
