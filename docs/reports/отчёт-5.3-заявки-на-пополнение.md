# Отчёт — этап 5.3 «Заявки на пополнение»

> Дата: 2026-07-08. Ветка: `main`.
> ТЗ: [`docs/specs/тз-5.3-заявки-на-пополнение.md`](../specs/тз-5.3-заявки-на-пополнение.md).
> Roadmap: этап 5.3 (разд. 11) — создание (авто-детект + вручную), редактирование, разделение
> по магазинам (на будущее), закрытие и хранение закрытых; группировка по поставщику, ручной
> проброс кандидата с выбором поставщика. Опирается на 2.5 (рама вкладок, заглушка «Заявки»),
> 3.1 (движок поиска), 5.2 (`is_low_stock`) и поставщиков (`Supplier`/`resolve_suppliers`).

---

## 1. Что требовалось

Дать продавцу фиксировать товары к заказу (макет разд. 11). Система авто-обнаруживает
кандидатов (достиг минимума или закончился, разд. 11.2); продавец вручную пробрасывает нужные
в заявку, выбирая поставщика и правя количество (разд. 11.9.5). Заявки группируются по
поставщику (одна открытая на поставщика), редактируются (11.6), закрываются при заказе (11.7)
и хранятся после закрытия (11.8). Заявка — намерение заказать, а не движение склада.

---

## 2. Зафиксированные решения (согласованы до реализации)

1. **«Авто» = авто-детект кандидатов + ручной проброс.** Roadmap пишет «создание авто», но
   макет 11.9.5 запрещает автозаброс без выбора продавца. Система строит только список
   кандидатов; в заявку товар уходит по «+» с выбором поставщика и правкой количества.
2. **Разделение по магазинам — поле `store` из config.** v1 — одна точка; поле заполняется
   `settings.store_name`, UI выбора нет. Схема готова к нескольким точкам без переписывания.
3. **Текущее кол-во и мин. остаток в заявке — живые из товара, не снимок.** Строка хранит
   только `needed_quantity` + `comment`; остаток/минимум берутся из `Product` при показе.
4. **Одна открытая заявка на поставщика.** Проброс дописывает/обновляет строку в открытой
   заявке; повторный проброс товара обновляет строку, не плодит дубль.
5. **Заявка не трогает склад** — не пишет `Movement`, не меняет `quantity_current` (приход —
   этап 6). Отдельные таблицы, как `receipt`/`return_receipt`.

## 3. Что сделано

**Модель** ([app/models/order.py](../../app/models/order.py)) — `OrderStatus` (открыта/закрыта),
`Order` (шапка: `supplier_id`, `store`, `status`, `created_at`, `closed_at`) и `OrderLine`
(`order_id`, `product_id`, `needed_quantity` через `quantity_column()`, `comment`). Таблицы
`purchase_order` / `purchase_order_line` (имя `order` зарезервировано в SQL). Регистрация в
[app/models/__init__.py](../../app/models/__init__.py); `init_db()` создаёт таблицы сам.

**Схема** ([app/schemas/order.py](../../app/schemas/order.py)) — `OrderLineInput`
(`needed_quantity: PositiveDecimal` > 0, `comment` пусто→None), приёмы из `schemas/product.py`.

**Сервис** ([app/services/order_service.py](../../app/services/order_service.py)):
- `is_order_candidate(product)` — чистая функция: `is_low_stock` (5.2) **или** `quantity_current <= 0`
  (закончился, в т.ч. случай `min_stock == 0`, отданный 5.2 → 5.3).
- `list_candidates` — кандидаты, сгруппированные по поставщику + группа «Разный / без
  поставщика»; пометка «уже в открытой заявке».
- `push_to_order` — найти/создать открытую заявку поставщика (`store` из config), добавить/
  обновить строку. Одна транзакция, склад не трогается.
- `add_manual_line` / `update_line` / `remove_line` — ручное редактирование (11.6); правки
  закрытой заявки отклоняются (`_open_order_of_line`).
- `close_order` — статус→закрыта + `closed_at`; идемпотентная защита от повторного закрытия.
- `order_view` / `list_open_orders` / `list_closed_orders` — чтение с живыми кол-вом/минимумом.

**Роуты** ([app/routes/orders.py](../../app/routes/orders.py)) — панель `GET /orders/panel`,
форма проброса `GET /orders/push-form`, `POST /orders/push` (разрешение поставщика: явный id /
имя с инлайн-созданием / единственный поставщик товара; при 0 или нескольких — форма выбора
через `HX-Retarget` на модалку), поиск для ручного добавления `GET /orders/{id}/search`,
`POST /orders/{id}/lines`, правка/удаление строки, `POST /orders/{id}/close`. Роутер подключён
в [app/main.py](../../app/main.py).

**Оболочка** ([app/routes/shell.py](../../app/routes/shell.py)) — ветка `key == "orders"` в
`section_panel` переведена с заглушки на реальную панель `orders/_panel.html`; добавлен фильтр
`quantity`. `SECTIONS` не тронут (кнопка «Заявки» уже была с 2.5).

**Шаблоны** [app/templates/orders/](../../app/templates/orders/): `_panel.html` (обёртка + CSS),
`_body.html` (модалка + 3 секции), `_candidates.html`, `_open_orders.html` + `_order.html`
(карточка заявки с правкой строк, поиском для добавления, закрытием), `_push_form.html`
(модалка выбора поставщика + количества), `_add_search_results.html`, `_archive.html`.

**Конфиг** ([app/config.py](../../app/config.py)) — `store_name: str = "Магазин"`.

**Новый тест** `tests/test_orders.py`; поправлен `tests/test_shell.py` (панель «Заявки» больше
не заглушка).

## 4. Тесты

`tests/test_orders.py` покрывает критерии приёмки ТЗ (разд. 11):

- **Детектор `is_order_candidate`:** ниже минимума → True; закончился при `min_stock == 0` →
  True; нормальный → False; весовой (`Decimal`).
- **Сервис:** проброс создаёт открытую заявку со `store` из config, склад не тронут (нет нового
  `Movement`, остаток цел); второй товар тому же поставщику → та же заявка; повторный проброс
  обновляет строку; инлайн-создание нового поставщика; правка/удаление/ручное добавление;
  закрытие + отклонение правок закрытой; граница `needed_quantity > 0`.
- **HTTP:** панель рисует кандидатов/открытые/архив; проброс с одним поставщиком → строка;
  несколько поставщиков без выбора → форма выбора, с выбором → добавлено; закрытие → архив,
  остаток цел.

Прогон:

```
python -m pytest tests/test_orders.py -q   →  15 passed
python -m pytest -q                         →  282 passed
```

**Живой смоук** (реальный сервер + `data/store.db`): товар-кандидат виден в панели → `POST
/orders/push` создал заявку (`store="Магазин"`, `needed_quantity=8`) → `POST /orders/{id}/close`
сменил статус на «закрыта»; остаток товара не изменился, движений по заявке не добавилось.
Таблицы `purchase_order` / `purchase_order_line` создаются `init_db()` автоматически.
Смоук-данные из dev-БД удалены.

Регрессия зелёная: денежный и складской путь (продажа/возврат/инвентаризация) не затронуты.

## 5. Технический долг / на будущее

- **Приём заказанного товара** (разд. 12, этап 6): возврат в остаток и движение «приход» по
  закрытой заявке. Сейчас закрытие склад не трогает — намеренно.
- **Разделение по магазинам** реально включится при появлении второй точки: модель уже несёт
  `store`, потребуется сущность «Магазин» и UI выбора.
- **Номер заявки / печать / экспорт** — не в scope 5.3; идентификатор = `id` + дата.
- **Мульти-поставщик в списке кандидатов**: товар с несколькими поставщиками показывается в
  группе каждого; проброс уточняет поставщика на месте.
