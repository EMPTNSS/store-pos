# Отчёт — этап 6.1 «Ручной приём накладной»

> Дата: 2026-07-09. Ветка: `main`.
> ТЗ: [`docs/specs/тз-6.1-приём-накладной.md`](../specs/тз-6.1-приём-накладной.md).
> Roadmap: этап 6.1 (разд. 12.3) — ручной приём через поиск карточек: приход существующих
> товаров (обновление количества и цены), создание карточек для новых. Опирается на 0.3
> (`create_product`/`ProductCreate`), 2.5 (рама вкладок, заглушка «Добавить»), 3.1 (движок
> поиска `search_products`, фрагмент `_supplier_row.html`).

---

## 1. Что требовалось

После приезда товара занести его в базу (макет 12.1) ручным приёмом (12.3) через поиск
карточек: товар есть → найти, добавить пришедшее количество, при необходимости изменить цену,
система обновляет товар; товара нет → создать карточку. Приём — складской путь: поднимает
остаток и пишет движение товара, всё одной транзакцией. Наполнить вкладку «Добавить» рамы (2.5).

---

## 2. Зафиксированные решения (согласованы до реализации)

1. **Приём — по одной позиции, без сущности-документа «накладная».** Макет 12.3 —
   пооперационная работа через карточку; ремарка 2.5 — «пока 1 товар». Аудит прихода несёт
   движение `Movement` типа «приход». Документ-накладная — будущее расширение.
2. **Приход прибавляет к остатку и пишет «приход»** (`OperationType.income`), в отличие от
   `adjust_quantity` (3.1), которая выставляет абсолютный остаток и пишет «инвентаризация».
3. **На форме приёма правятся обе цены** (закупочная и продажная), обе необязательны: пустое/
   неизменённое поле не трогает цену; при реальном изменении — одна точка `PriceHistory`.
4. **Новый товар — переиспользование `create_product` (0.3)**, без дублирования складской/
   ценовой логики; 6.1 добавляет только вход из поиска и HTMX-обёртку.
5. **Без новых таблиц и без миграции** — приём ложится на `Movement`/`PriceHistory`/`Product`.

---

## 3. Что сделано

**Схема** ([app/schemas/product.py](../../app/schemas/product.py)) — `ProductReceive`:
`received_quantity: PositiveDecimal` > 0; `price_buy` / `price_sell` — `Optional[int]` с
before-валидатором `optional_kopecks` (пусто/None → None = не менять, иначе разбор в копейки
через `_parse_kopecks`); закупка ≥ 0, продажа > 0 при наличии.

**Сервис** ([app/services/product_service.py](../../app/services/product_service.py)) —
`receive_product(session, product_id, received_quantity, price_buy=None, price_sell=None)`:
одной транзакцией `quantity_current += received_quantity`, движение «приход» (знак +), при
изменении любой цены — одна `PriceHistory` с новой парой. Кол-во > 0 гарантировано схемой,
поэтому движение пишется всегда. Новый товар логику не дублирует — вызывается `create_product`.

**Роуты** ([app/routes/receiving.py](../../app/routes/receiving.py)) — `GET /receiving/search`
(движок `search_products` + флаг `is_low_stock`), `GET /receiving/new` (форма создания,
имя предзаполнено), `POST /receiving/create` (реюз `ProductCreate`/`create_product`, дубль QR →
422), `GET /receiving/{id}/form` (форма приёма, цены предзаполнены текущими), `POST
/receiving/{id}` (`receive_product`). Статические `new`/`create` объявлены **до** динамических
`{product_id}`, иначе «create»/«new» ловятся как id. Роутер подключён в
[app/main.py](../../app/main.py).

**Оболочка** ([app/routes/shell.py](../../app/routes/shell.py)) — ветка `key == "add"` в
`section_panel` переведена с заглушки на реальную панель `add/_panel.html`; тестовое поле рамы
(2.5) снято (`test_field=False`). `SECTIONS` не тронут.

**Шаблоны** [app/templates/receiving/](../../app/templates/receiving/): `_search_results.html`
(«Оприходовать» / «Создать карточку»), `_receive_form.html` (кол-во + обе цены + контекст),
`_received.html` (подтверждение), `_create_form.html` (форма создания как фрагмент / успех).
[app/templates/add/_panel.html](../../app/templates/add/_panel.html) — поиск + слоты результатов
и формы. Общие поля формы создания вынесены в
[app/templates/products/_create_fields.html](../../app/templates/products/_create_fields.html),
включены и в `products/new.html`, и в `receiving/_create_form.html` (без дублирования разметки).

**Правки тестов** — `tests/test_shell.py` (панель «Добавить» больше не заглушка, тестового поля
нет), новый `tests/test_receiving.py`.

### Соответствие разд. 12.3 → код

| Требование макета | Где |
|---|---|
| Ручное добавление через поиск карточек | `GET /receiving/search` (`search_products`) |
| Товар есть: найти → добавить количество | `receive_product` (`+= кол-во`, движение «приход») |
| Товар есть: при необходимости изменить цену | форма приёма (обе цены) → `PriceHistory` при изменении |
| Товара нет: создать карточку и внести информацию | `GET /receiving/new` → `POST /receiving/create` → `create_product` |

## 4. Тесты

`tests/test_receiving.py` (18 тестов) покрывает критерии приёмки ТЗ (разд. 11):

- **Сервис `receive_product`:** приход поднимает остаток и пишет движение «приход» (+);
  без смены цен `price_history` не пополнена; смена закупки/продажи/обеих → ровно одна точка;
  весовой товар (`Decimal`); атомарность (остаток + движение + точка вместе).
- **Схема `ProductReceive`:** кол-во ≤ 0 → ошибка; закупка < 0 и продажа ≤ 0 → ошибка;
  None/пустые цены проходят, заданные разбираются в копейки.
- **Новый товар:** `POST /receiving/create` валидное → товар создан (через `create_product`);
  невалидная цена → 422-фрагмент.
- **HTTP:** поиск с попаданием → «Оприходовать», без попадания → «Создать карточку»; форма
  приёма + 404; приём поднимает остаток в БД и пишет движение; кол-во 0 → 422; `GET /panels/add`
  — реальная панель, тестового поля нет.

Прогон:

```
python -m pytest tests/test_receiving.py -q   →  18 passed
python -m pytest -q                            →  300 passed
```

**Живой смоук** (TestClient + сборка приложения): создан товар «Хлеб» (остаток 4) → `GET
/receiving/1/form` показал текущие цены → `POST /receiving/1` (пришло 6, закупка 20→22) поднял
остаток до 10, записал движение «приход» и точку `PriceHistory` (закупка изменилась); `GET
/receiving/new?name=Батон` отрисовал форму создания с предзаполненным именем и datalist
поставщиков.

Регрессия зелёная: денежный и складской путь (продажа/возврат/инвентаризация/заявки) не
затронуты — приём только добавляет движение «приход» и точки истории цен.

## 5. Технический долг / на будущее

- **Документ-накладная (несколько строк за проведение)** — сейчас приём по одной позиции
  (решение 2 п.1). При необходимости группировки — отдельная сущность `Receiving` + строки.
- **Автоматизированный приём по скану накладной (12.4)** — отложен, «под вопросом».
- **Проверка/рекомендуемая цена и подсветка (12.6)** — формула отложена (этап 8).
- **Связь прихода с закрытой заявкой (5.3)** — приём и заявки остаются отдельными путями;
  сшивку («закрыл заявку → оприходовал») не делали.
- **Empty-name через HTTP** для создания перехватывается FastAPI (`Form(...)`), до схемы не
  доходит; на форме имя обязательно (`required`), негативный кейс покрыт на уровне схемы в 0.3.
