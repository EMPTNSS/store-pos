# Отчёт — этап 7.1 «Документ завершения рабочего дня»

> Дата: 2026-07-11. Ветка: `main`.
> ТЗ: [`docs/specs/тз-7.1-завершение-рабочего-дня.md`](../specs/тз-7.1-завершение-рабочего-дня.md).
> Roadmap: этап 7.1 (разд. 23) — итоги по дню + детализация по товарам, сохранение.
> Опирается на 1.3 (`Receipt`/`ReceiptLine`), 4.1/4.2 (`ReturnReceipt`), 3.2 (`buy_price_asof`)
> и уже реализованную сущность смены `WorkDay`.

---

## 1. Что требовалось

При закрытии рабочего дня сформировать и сохранить документ по дню (макет разд. 23): итог
продаж, сумму чистыми, доп. финансовую информацию и потоварную детализацию. Документ
доступен для просмотра позже (23.5) и станет единицей для месячного отчёта (7.2). Кнопку
«Завершить рабочий день» не плодить — она уже есть в оболочке.

---

## 2. Зафиксированные решения (согласованы до реализации)

1. **«Чистыми» — реконструкцией себестоимости из `price_history`** через готовый
   `buy_price_asof` (3.2). Схему чека не меняли, снимок закупки в строке не заводили.
2. **Граница дня — смена `WorkDay`**, отбор операций строго по `work_day_id` (не окно по
   `datetime`). Ранний обходной путь `previous_close` из первой редакции ТЗ отменён.
3. **Возвраты смены вычитаются из итогов**; их себестоимость возвращается в расчёт прибыли.
4. **Закрытие смены + документ — одна транзакция** (правило 2 CLAUDE.md).

## 3. Что сделано

### Новые файлы
- **`app/models/day_report.py`** — `DayReport` (шапка-итоги, `work_day_id` unique — один
  документ на смену), `DayReportLine` (потоварная детализация), `DayReportNumberCounter`.
- **`app/services/day_report_service.py`** — `compute_day` (чистая агрегация по смене),
  `close_day_and_report` (атомарное закрытие + документ), `list_day_reports`,
  `get_day_report`, `day_report_lines`.
- **`app/routes/reports.py`** — `GET /day/reports` (список), `GET /day/reports/{id}` (просмотр).
- **`app/templates/reports/`** — `_document.html`, `_list.html`, `_panel.html`.
- **`tests/test_day_report.py`** — 20 тестов.

### Изменения
- **`app/routes/work_day.py`** — `POST /work-day/close` теперь зовёт `close_day_and_report`
  (вместо `work_day_service.close_day`); ответ ведёт к документу. `_render_status` переведён
  на современную сигнатуру `TemplateResponse(request, name, context)`.
- **`app/routes/shell.py`** — секция `receipts` («Чеки за день») отдаёт реальную панель со
  свежим документом наверху вместо заглушки.
- **`app/templates/shell/_day_status.html`** — после закрытия показывает «Документ дня №N
  сформирован» + кнопку «Открыть документ».
- **`app/static/shell.js`** — тонкий публичный хук `window.posShell` (`openSection`,
  `openReports`) для навигации из фрагмента статуса; бизнес-логики в нём нет.
- **`app/main.py`**, **`app/database.py`**, **`app/models/__init__.py`** — подключение
  роутера, инициализация счётчика, регистрация моделей.
- **`tests/test_shell.py`** — заглушечный тест `receipts` заменён на проверку реальной панели.

### Денежная семантика (реализована в `compute_day`)
- `net_sales = sales_total − returns_total` — «итоговая сумма продаж за день».
- Себестоимость строки — `line_total(buy_price_asof(product_id, operation.datetime), qty)`.
- `net_profit = net_sales − (cogs_sold − cogs_returned)` — «сумма чистыми».
- Инвариант: `Σ DayReportLine.sum_sold + rounding_total = sales_total`.

## 4. Тесты

`tests/test_day_report.py` покрывает критерии приёмки ТЗ (разд. 9): себестоимость на момент
продажи (в т.ч. при изменившейся закупке и весовом товаре), вычет возвратов, потоварный
агрегат по нескольким чекам, инвариант сверки, отбор строго по `work_day_id` (чужая смена и
`NULL`-чек не входят), независимость двух смен, атомарное закрытие, отказ без открытой смены,
нулевой документ пустой смены, разбивка нал/безнал и счётчики, сохранение/чтение, HTTP-слой
(`/work-day/close`, `/day/reports`, `/day/reports/{id}`, `/panels/receipts`).

```
python -m pytest tests/test_day_report.py -q   →  20 passed
python -m pytest -q                            →  335 passed
```

Дополнительно — сквозной живой прогон через `TestClient` на свежей БД: открытие смены →
продажа → закрытие формирует документ → панель «Чеки за день» показывает «Чистыми» и товар →
повторное закрытие даёт «Нет открытого рабочего дня». Регрессия зелёная: денежный/складской
путь продаж, возвратов и жизненный цикл смены не затронуты.
