# Roadmap — план разработки POS-системы

## Как пользоваться этим файлом

- Каждый пронумерованный пункт — это отдельная единица, по которой составляется своё ТЗ
  (один пункт = один разговор с чатом → один файл ТЗ в `docs/`).
- Номера в скобках (разд. N) — ссылки на соответствующие разделы `docs/макет-системы.txt`.
- Порядок выстроен по зависимостям и по принципу «сначала рабочая продажа, потом всё
  остальное». Не идти вширь, пока не закрыт минимальный цикл.

**Граница v1:** этапы 0–2. Это минимальная касса, которая реально продаёт, печатает чек
и показывает покупателю. Всё, что после — расширение уже работающей системы.

---

## Этап 0 — Фундамент

Без этого нечего продавать и не на чем строить. Центральная сущность всей системы — товар,
поэтому начинаем с него.

- **0.1 Скелет проекта + конфиг + подключение БД.** Структура папок, `config.py`,
  движок и сессии SQLModel поверх SQLite. Чистая инфраструктура.
- **0.2 Модель товара + база данных товаров** (разд. 5, 7). Все поля товара: название,
  артикул, коды, цена закупки/продажи, количество, **единица измерения** (разд. 9),
  минимальный остаток, **статус** (поле есть сразу, финальный список статусов — отложен,
  см. этап 8), доп. информация. Это структура, на которую ссылаются все остальные ТЗ.
- **0.3 Создание карточки товара** (разд. 6). Минимальный способ завести товар в базу —
  чтобы было что продавать и на чём тестировать.

## Этап 1 — Минимальный цикл продажи (ядро, чистый софт, без железа)

Здесь живёт главный риск проекта — денежный и складской путь. Весь этап тестируется
без принтера и сканера, в браузере. Денежную логику покрыть тестами.

- **1.1 Экран кассы.** Добавить товар в чек, изменить количество, убрать товар, видеть
  итоговую сумму. Сюда же — отмена/очистка незавершённого чека (разд. 1.1–1.3, 15.1).
- **1.2 Ввод товара в чек.** Поиск товара для чека вручную (разд. 3) + ввод со сканера
  (разд. 2). Сканер = клавиатурный ввод, поэтому почти бесплатен. Сюда же — ручной ввод
  веса для весовых товаров (разд. 22, весы не интегрированы).
- **1.3 Завершение продажи** (разд. 1.2–1.5). Способ оплаты как пометка (нал/безнал;
  терминал не интегрирован, разд. 21), сохранение электронного чека, списание остатка,
  запись движения товара — **всё одной транзакцией**. Ядро денежного пути.

▶ **Точка обратной связи:** после этапа 1 продажа работает на экране и корректно меняет
склад. Можно щупать руками.

---

## Этап 2 — Железо вокруг продажи

- **2.1 Печать чека** (разд. 18.1–18.4). ESC/POS из Python напрямую в принтер.
- **2.2 Формирование накладной при продаже** (разд. 1.4, 18.5, 18.6). Чекбокс
  «Распечатать накладную» при завершении покупки. Накладная собирается из завершённого
  чека и выводится в файл (data/invoices/). Выбор печатающего устройства и реальная
  печать отложены до появления принтера. Слой печати изолирован от сборки, чтобы
  подключение устройства не трогало логику формирования накладной.
- **2.3 Экран покупателя** (разд. 20). Второе окно: товары, количество, итог, после
  оплаты — «Спасибо за покупку».

▶ **Конец v1.** Полноценная касса: продаёт, печатает, показывает покупателю.

---

## Этап 3 — Карточка товара и движение

- **3.1 Полная карточка товара** (разд. 4, 5). Поиск товара для карточки, просмотр всей
  детальной информации, редактирование (цена, количество, мин. остаток, статус и пр.).
- **3.2 История движений товара** (разд. 5.7, 5.4). Дата, количество, пометка операции;
  статистика по товару.
- **3.3 Графики динамики цен** (разд. 5.6). Менее критично — можно отложить внутри этапа.

---

## Этап 4 — Возвраты и исправления

- **4.1 Возврат товара** (разд. 14). Кнопка возврата, выбор товара, возврат в остаток,
  чек возврата.
- **4.2 Исправление завершённого чека через возврат** (разд. 15.2–15.3). Завершённый чек
  напрямую не правится — только через возврат.

---

## Этап 5 — Склад и пополнение

- **5.1 Инвентаризация** (разд. 13). Ручная корректировка фактического остатка + отметка
  в движении товара.
- **5.2 Минимальный остаток** (разд. 10). Отслеживание, уведомление при достижении,
  отображение в карточке.
- **5.3 Заявки на пополнение** (разд. 11). Создание (авто при достижении мин. остатка +
  вручную), редактирование, разделение по магазинам (на будущее), закрытие и хранение
  закрытых заявок. Заявки группируются по **поставщику** (разд. 11.9): товар ссылается
  на одного и более поставщиков; проброс закончившегося товара в заявку — ручной, с
  выбором поставщика и правкой количества. Поле поставщика на форме создания товара (0.3,
  уже готова — добавляется позже) и в карточке (3.1).

---

## Этап 6 — Приём товара

- **6.1 Ручной приём накладной** (разд. 12.3). Через поиск карточек: обновление количества
  и цены существующих товаров, создание карточек для новых.
- *Автоматизированный приём по скану накладной (разд. 12.4) — отложен, статус «под вопросом».*

---

## Этап 7 — Отчёты и данные

- **7.1 Документ завершения рабочего дня** (разд. 23). Итоги по дню + детализация по товарам,
  сохранение.
- **7.2 Отчёт за месяц** (разд. 24). На основе сохранённых дней, без потоварной детализации.
- **7.3 Экспорт / импорт данных** (разд. 25). Выгрузка отчётов и базы, загрузка данных.
- **7.4 Печать этикеток / QR-кодов** (разд. 19). Отдельный принтер этикеток: код, цена,
  название.

- Правила отрицательных остатков (разд. 17)
- Формула рекомендуемой цены при приёме (разд. 12.6)
- Резервные копии (разд. 26)
- Экран настроек (разд. 27)
- Журнал действий (разд. 28)
