# ТЗ 0.1 — Скелет проекта + конфиг + подключение БД

> Источник истины по требованиям — `docs/макет-системы.txt` и `CLAUDE.md`.
> Этот файл — точечное ТЗ под этап 0.1 из `roadmap.md`. Всё, что здесь не описано
> и не следует из `CLAUDE.md`, в рамках 0.1 не делаем.

## 1. Цель

Поднять рабочий каркас приложения: структура папок, конфигурация, движок и сессии
SQLModel поверх SQLite. После 0.1 приложение **стартует, отдаёт health-эндпоинт и при
первом запуске создаёт файл БД**. Никакой доменной логики, никаких таблиц.

Это фундамент, на который ссылаются все последующие ТЗ. Цель — чистая инфраструктура,
которую можно прогнать тестом и пощупать в браузере.

## 2. Границы

**В scope:**
- Структура пакетов `app/` (пустые пакеты с `__init__.py`).
- `config.py` — параметры приложения (на `pydantic-settings`).
- `app/database.py` — движок, фабрика сессий, зависимость `get_session`, `init_db()`.
- `app/main.py` — сборка FastAPI через фабрику + `/health`.
- `run.py` — точка входа (uvicorn в потоке + окно pywebview).
- `requirements.txt`, `.gitignore`, `README.md`.
- `tests/` — smoke-тест приложения и проверка подключения к БД.

**Вне scope (не делать сейчас):**
- Модель товара и любые таблицы — это 0.2.
- Создание карточки товара — 0.3.
- Касса, поиск, шаблоны/страницы UI — этап 1+.
- Любое железо, печать, `python-escpos` — этап 2+.
- Экран настроек (разд. 27) — параметры живут в `config.py`.

## 4. Требования по файлам

### 4.1 `app/config.py`
- Класс `Settings(BaseSettings)` из `pydantic-settings`, единый экземпляр `settings`.
- Поля (имена — на английском):
  - `app_name: str = "store-pos"`
  - `debug: bool = False`
  - `db_path: Path = Path("data/store.db")`
  - `db_echo: bool = False` — лог SQL (по умолчанию off; удобно завязать на `debug`).
  - `host: str = "127.0.0.1"`
  - `port: int = 8000`
- Свойство/поле `database_url` → `f"sqlite:///{db_path}"`.
- Поддержать переопределение через переменные окружения и `.env` (префикс по желанию,
  например `STORE_`). `.env` — в `.gitignore`.
- Хардварных и валютных параметров здесь **пока нет** — добавим, когда появятся касса и
  печать.

### 4.2 `app/database.py`
- Один движок `engine = create_engine(settings.database_url, ...)`.
- `connect_args={"check_same_thread": False}` — uvicorn и поток pywebview ходят в БД из
  разных потоков, иначе SQLite ругается.
- `echo=settings.db_echo`.
- **Включить foreign keys** в SQLite (по умолчанию они выключены). Это инфраструктура
  на уровне движка, а не доменный SQL — правило «не пишем сырой SQL» сюда не относится:

  ```python
  from sqlalchemy import event

  @event.listens_for(engine, "connect")
  def _set_sqlite_pragma(dbapi_conn, _):
      cur = dbapi_conn.cursor()
      cur.execute("PRAGMA foreign_keys=ON")
      cur.execute("PRAGMA journal_mode=WAL")   # мягкая защита при конкуренции потоков
      cur.close()
  ```
- `def get_session() -> Generator[Session, None, None]` — зависимость FastAPI:
  открывает `Session(engine)`, делает `yield`, корректно закрывает. Это **единственный**
  способ получить сессию в остальном коде.
- `def init_db() -> None`:
  - создаёт каталог `settings.db_path.parent` (`mkdir(parents=True, exist_ok=True)`);
  - вызывает `SQLModel.metadata.create_all(engine)`.
  - На 0.1 таблиц нет — функция отработает вхолостую. В 0.2 модели будут импортированы
    (через `app/models`), их таблицы зарегистрируются в `metadata`, и `create_all` их
    создаст. Оставить короткий комментарий про это, чтобы в 0.2 не забыть импорт моделей.

### 4.3 `app/main.py`
- Фабрика `def create_app() -> FastAPI`.
- На старте приложения (lifespan / событие startup) вызывать `init_db()`.
- Эндпоинт `GET /health` → `{"status": "ok"}`, код 200.
- Модуль экспортирует `app = create_app()` (чтобы работало `uvicorn app.main:app`).
- Шаблоны и статику в 0.1 **не подключаем** — каталоги созданы, проводка Jinja2/StaticFiles
  приедет с первой реальной страницей (этап 1).

### 4.4 `run.py`
- Точка входа: поднять uvicorn с `app.main:app` в фоновом потоке на `settings.host:port`,
  затем открыть окно pywebview на `http://{host}:{port}`.
- Это «боевой» запуск под продавца. Для отладки в браузере используется
  `uvicorn app.main:app --reload` напрямую (без окна).

### 4.5 `requirements.txt`
Минимальный рабочий набор под 0.1 (версии — последние совместимые, можно запинить):
- `fastapi`
- `uvicorn[standard]`
- `sqlmodel`
- `pydantic-settings`
- `pywebview`
- `pytest`
- `httpx` (нужен для `TestClient`)

`python-escpos`, `jinja2` — **не добавлять** сейчас (печать — этап 2, шаблоны — этап 1).
Если установка `pywebview` на машине проблемна, допустимо временно вынести его и `run.py`
в конец и проверять всё через uvicorn — но по умолчанию ставим.

### 4.6 `.gitignore`
Минимум: `__pycache__/`, `*.pyc`, `.venv/`, `venv/`, `.env`, `data/*.db`, `data/*.db-wal`,
`data/*.db-shm`, `.pytest_cache/`. Файл `data/.gitkeep` при этом должен остаться в репозитории.

### 4.7 `README.md`
Коротко: установка (`pip install -r requirements.txt`), запуск (`python run.py`),
отладка (`uvicorn app.main:app --reload`), тесты (`pytest`). Без воды.

### 4.8 `tests/`
- `conftest.py`: фикстура, дающая **временную/отдельную БД** (temp-файл или
  `sqlite://` in-memory), чтобы тесты **не трогали** реальный `data/store.db`.
  Тестовый движок настроен теми же `connect_args` и тем же pragma-листенером.
- `test_smoke.py`: `TestClient(app)` → `GET /health` == 200 и тело `{"status": "ok"}`.
- `test_db.py`:
  - `init_db()` создаёт файл БД (или таблицы в in-memory) без ошибок;
  - через сессию выполняется `SELECT 1` (`session.exec(select(1))` или
    `session.execute(text("SELECT 1"))`) — соединение живое;
  - `PRAGMA foreign_keys` возвращает `1` (FK включены).

## 5. Чего не нарушать (из `CLAUDE.md`)

- Остаёмся в Python. Никаких JS-фреймворков.
- Сырой SQL — только инфраструктурный (pragma на движке, `SELECT 1`/`text()` в тестах).
  Доменной логики на сыром SQL быть не должно (её здесь и нет).
- Логику не привязывать к диалекту SQLite сверх необходимого: специфика SQLite допустима
  только в `database.py` (pragma, `check_same_thread`). Переезд на Postgres должен сводиться
  к смене `database_url`.
- Деньги — `int` в копейках, количество — `Decimal`. В 0.1 моделей нет, но это правило
  держим в голове для 0.2 и тестового кода ничем не нарушаем.
- Код, имена, коммиты — английский. (UI-текстов в 0.1 нет.)

## 6. Критерии приёмки (Definition of Done)

1. `pip install -r requirements.txt` проходит без ошибок.
2. `uvicorn app.main:app --reload` стартует, `GET /health` → 200 `{"status":"ok"}`.
3. При первом запуске создаётся каталог `data/` и файл `data/store.db`.
4. `python run.py` на машине разработчика открывает окно с приложением (ручная проверка).
5. `pytest` зелёный; тесты используют отдельную БД и не оставляют следов в `data/store.db`.
6. Тест подтверждает, что `PRAGMA foreign_keys = ON`.
7. `data/store.db` (и `.env`) не попадают в git; `data/.gitkeep` — попадает.
8. Пакет импортируется как `app` (абсолютные импорты, без относительных хаков).

## 8. Открытые решения (можно переиграть до старта)

- **Конфиг на `pydantic-settings`** vs голые константы в модуле. Выбран первый: валидация
  типов и переопределение через env «бесплатно», Pydantic и так в стеке. Если хочется
  предельной простоты — можно свести к обычному модулю с константами.
- **pywebview в `run.py` уже сейчас** vs отложить до этапа 2/UI. Включил сейчас: это
  валидирует весь стек целиком одним запуском. Жёсткие критерии (п. 6) при этом висят на
  uvicorn+TestClient — детерминированно и не зависит от GUI-окружения.
- **`journal_mode=WAL`** — добавлен как мягкая страховка при конкуренции потоков
  (окно + сервер). Для одной кассы можно убрать без последствий.
