# ТЗ — База данных товаров и карточка товара

> Этап **0.2** roadmap (модель товара + база данных товаров).
> Ссылки на макет: разд. 5 (карточка), 7 (БД товаров), 8 (статусы), 9 (единицы измерения), 10 (мин. остаток).
> Источник истины по требованиям — `docs/макет-системы.txt`. При противоречии — сверяться с макетом.

---

## 1. Назначение

Спроектировать хранение данных о товарах: структуру таблиц, типы полей, правила
вычисления производных значений. Это фундамент, на который ссылаются все
последующие ТЗ (касса, движения, отчёты, приём накладных).

Карточка товара (макет разд. 5) — это **представление** данных из БД, а не отдельное
хранилище. Все поля карточки берутся из таблиц, описанных ниже.

---

## 2. Ключевой принцип: хранимое против вычисляемого

Поля делятся на два класса. Путать их нельзя — иначе БД начнёт врать.

**Хранимое** — лежит физически в таблице, редактируется напрямую.

**Вычисляемое** — не хранится отдельным полем, а считается из других таблиц при
запросе. Хранить такие значения как поля опасно: они устаревают и расходятся с
реальностью при первом же изменении.

| Значение | Класс | Откуда берётся |
|---|---|---|
| Название, артикул, коды, цены, ед. изм., мин. остаток, статус | хранимое | таблица `product` |
| Текущий остаток | вычисляемое* | сумма движений (`movement`) |
| Разница в деньгах, наценка в % | вычисляемое | из двух цен товара |
| Продано за всё время, приход/убытие по датам | вычисляемое | запросы к `movement` |
| Динамика цен (для графиков) | вычисляемое | таблица `price_history` |
| Чистая прибыль за всё время | вычисляемое | из чеков (см. п. 7) |

\* остаток дополнительно кэшируется в поле `quantity_current` ради скорости — см. п. 5.

---

## 3. Таблицы

Система использует три таблицы:

1. **`product`** — карточка товара (хранимые поля).
2. **`movement`** — движения товара (приход, продажа, возврат, инвентаризация).
   Источник истины по остатку и всей складской статистике.
3. **`price_history`** — история цен для графиков динамики (макет 5.6).

---

## 4. Таблица `product`

| Поле | Тип | Обяз. | Описание |
|---|---|---|---|
| `id` | int, PK, автоинкремент | да | Внутренний технический ключ. Не показывается продавцу. |
| `name` | str | да | Название товара. |
| `article` | str | да | Артикул. **Строка, не число** — допускает буквы и ведущие нули (`0042`, `AB-12`). |
| `numeric_code` | str, UNIQUE | да | Числовой код для ручного поиска. Генерируется системой (см. п. 6). Строка, даже если внутри только цифры. |
| `qr_code` | str, UNIQUE, **nullable** | нет | QR / штрихкод. Может быть пустым при создании карточки (этикетка ещё не напечатана), в норме заполняется позже. См. п. 6. |
| `price_sell` | int | да | Текущая цена продажи, **в копейках** (148.50 ₽ = `14850`). |
| `price_buy` | int | да | Цена закупки, **в копейках**. |
| `unit` | enum | да | Единица измерения. Перечень: `шт`, `кг`, `м`, `л`, `упак`, `компл`. Не свободный текст. |
| `min_stock` | Decimal | да | Минимальный остаток. Decimal — у весовых товаров дробный (2.5 кг). |
| `status` | enum | да | Статус. Значения: `активный`, `архивный`. По умолчанию `активный`. См. п. 8. |
| `quantity_current` | Decimal | да | Кэш текущего остатка. Источник истины — `movement`; поле обновляется той же транзакцией. См. п. 5. |
| `created_at` | datetime | да | Дата и время создания карточки. Ставится автоматически при вставке через `default_factory` (не `default=datetime.now()`). Не редактируется. |

### Правила типов (из CLAUDE.md, железные)

- **Деньги — `int` в копейках.** Не float, не Decimal. Цена 148.50 ₽ хранится как `14850`.
  На экране делится на 100.
- **Количество и мин. остаток — `Decimal`.** float запрещён (ошибки округления).
  Штучные товары целые (3 шт), весовые дробные (1.5 кг).
- **Артикул и любые коды — `str`.** Над ними не делают арифметику, по ним ищут и
  сравнивают. Буквы и ведущие нули обязаны сохраняться.

## 5. Остаток: ведётся через движения

`quantity_current` **не редактируется напрямую**. Любое изменение остатка проходит
через запись в `movement` (CLAUDE.md, правило 5):

- продажа — списание (количество со знаком минус);
- приход — добавление (плюс);
- возврат — добавление (плюс);
- инвентаризация / корректировка — выравнивание до фактического.

**Почему кэш, а не чистое вычисление каждый раз:** суммировать все движения товара на
каждый показ карточки медленно, когда движений тысячи. Поэтому остаток кэшируется в
`quantity_current` и обновляется той же транзакцией, что пишет движение. Источник истины
остаётся `movement` — при сомнении кэш можно пересчитать суммой движений.

**Атомарность:** запись движения + обновление `quantity_current` — одна транзакция БД
(CLAUDE.md, правило 2). Либо всё, либо ничего. Полусписанных состояний быть не должно.

## 6. Коды товара

У товара ровно один числовой код и максимум один QR/штрихкод (двух QR одновременно не
бывает — решено на этапе проектирования). Поэтому оба кода — поля в `product`, отдельная
таблица не нужна.

### Числовой код (`numeric_code`)

- Генерируется системой при создании карточки.
- Берётся из **постоянно растущего счётчика** (последовательность), а не от
  «максимального существующего». Это исключает повторное использование номера после
  архивации товара.
- На поле стоит ограничение **UNIQUE** — БД физически не даст создать дубль.

### QR / штрихкод (`qr_code`)

- **Допускает пустое значение (nullable).** Причина: между созданием карточки и печатью
  этикетки кода физически нет, а карточку нужно мочь создать (товар уже надо продавать).
  Сделать поле обязательным = заблокировать создание товара без кода.
- В норме заполняется: заводским значением при приёмке, либо вашим напечатанным
  (макет разд. 19). Для весовых товаров штрихкод вклеивается в каталог/тетрадку — по нему
  потом ищут сканером (макет 19.3).
- На поле — **UNIQUE** (с учётом пустых значений): два товара не могут делить один
  штрихкод.

## 7. Вычисляемые значения

Не хранятся отдельными полями. Считаются при запросе.

| Значение | Формула / источник |
|---|---|
| Текущий остаток | сумма `movement.quantity` по товару (кэш в `quantity_current`) |
| Разница в деньгах | `price_sell − price_buy` |
| Наценка в % | `(price_sell − price_buy) / price_buy × 100` |
| Продано за всё время | сумма движений типа «продажа» |
| Приход по датам | движения типа «приход», сгруппированные по дате |
| Убытие по датам | движения типа «продажа», сгруппированные по дате |
| Чистая прибыль за всё время | **из чеков**, не отсюда — см. ниже |

### Наценка — пояснение по математике

Закуп 40, продажа 60. Разница в деньгах = **20 ₽**. Наценка = 20 / 40 = **50 %**.
Это два разных числа; в карточке показываются оба. Ни одно не хранится — оба выводятся
из двух цен на лету.

### Чистая прибыль — зависимость от чеков

Прибыль = по каждой продаже `(цена продажи − цена закупки на момент продажи) × количество`,
просуммировать.

**Критично:** закупочная цена берётся **на момент продажи**, а не текущая из карточки.
Иначе после прихода новой партии по другой цене вся историческая прибыль пересчитается
задним числом и станет враньём. Значит закупочная цена должна фиксироваться **в строке
чека** при продаже.

Это требование к таблице чека, а **не к таблицам этого ТЗ**. Здесь оно зафиксировано как
зависимость, чтобы не потерять при проектировании кассы (этап 1.3).

## 8. Статусы товара

- Поле `status` заводится **сразу**, финальный перечень статусов отложен (CLAUDE.md,
  этап 8 / макет 8.2).
- Рабочий минимум: `активный` (по умолчанию), `архивный` (снят с продажи, но остаётся
  в базе ради истории).
- **«Нет в наличии» — НЕ хранимый статус.** Это вычисляется из остатка: `quantity_current = 0`.
  Хранить его как статус нельзя — будет второй источник правды, разойдётся с реальным
  остатком.
- Сортировка в поиске (макет 8.4): активные → нет в наличии (по остатку) → архивные.
  Точная логика отложена.

---

## 9. Удаление товара запрещено

Карточка товара **не удаляется физически** — только переводится в статус `архивный`.

Причина: на товар ссылаются чеки, движения, история цен. Физическое удаление порвёт
историю и (в связке с генерацией кодов) может привести к тому, что новый товар
унаследует чужой номер и чужую статистику. Архивация решает обе проблемы: номер занят
навсегда, история цела.

## 10. Таблица `movement`

| Поле | Тип | Описание |
|---|---|---|
| `id` | int, PK | Технический ключ. |
| `product_id` | int, FK → `product.id` | Ссылка на товар. |
| `datetime` | datetime | Дата и время операции. |
| `quantity` | Decimal | Количество со знаком: приход/возврат `+`, продажа `−`. |
| `operation_type` | enum | `приход`, `продажа`, `возврат`, `инвентаризация`. |

Подробная логика записи движений — в ТЗ соответствующих этапов (продажа — 1.3,
инвентаризация — 5.1, приём — 6.1). Здесь фиксируется только структура хранения.

## 11. Таблица `price_history`

Для графиков динамики цен (макет 5.6).

| Поле | Тип | Описание |
|---|---|---|
| `id` | int, PK | Технический ключ. |
| `product_id` | int, FK → `product.id` | Ссылка на товар. |
| `datetime` | datetime | Когда зафиксирована цена. |
| `price_buy` | int | Цена закупки на этот момент, в копейках. |
| `price_sell` | int | Цена продажи на этот момент, в копейках. |

Запись добавляется при изменении цены (приход новой партии меняет закупку; ручная правка
в карточке меняет продажу). График строится прямой выборкой по `product_id`,
упорядоченной по `datetime`.
