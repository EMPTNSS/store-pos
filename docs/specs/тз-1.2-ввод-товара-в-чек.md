# ТЗ — Ввод товара в чек

> Этап **1.2** roadmap (ввод товара в чек).
> Макет: разд. 2 (сканер), разд. 3 (поиск товаров для чека), разд. 22 (весы, не интегрированы).
> Опирается на 1.1 (`docs/specs/тз-1.1-экран-кассы.md`): корзина в памяти (`app/services/cart.py`),
> механика add/set_quantity/remove/clear, снимок цены, округление.
> Опирается на 0.2 (`docs/specs/тз-0.2-база-товаров.md`): таблица `product` с полями
> `name`, `numeric_code`, `qr_code`, `article`, `unit`, `price_sell`, `quantity_current`, `status`.
> Источник истины — `docs/Макет системы.txt`. При противоречии — сверяться с макетом.

---

## 1. Назначение

Достроить над готовой корзиной (1.1) удобные способы **найти товар и положить его в чек**:

- **сканер** (разд. 2) — считал QR-код, товар сразу добавился в текущий чек, без нажатия кнопок;
- **ручной поиск** (разд. 3) — продавец не сканирует, а ищет товар по названию/коду, видит список
  совпадений с ценой, наличием и статусом, и добавляет выбранный товар в чек;
- **ручной ввод веса** для весовых товаров (разд. 22) — вес не приходит с весов автоматически,
  продавец правит количество строки вручную (дробное число).

Границы 1.1 сохраняются: чек — **живое состояние в памяти**, он **не сохраняется** в БД и
**не трогает остаток**. Сохранение чека, оплата, списание, движение — этап 1.3. Здесь только
способы ввода поверх существующей корзины; денежная и складская логика не меняется.

Результат: товар в чек можно добавить тремя путями (скан, быстрый ввод кода, поиск по названию);
склад при этом по-прежнему не меняется.

---

## 2. Граница этапа

### Входит
- **Быстрый ввод / сканер** (`POST /cashier/items`): точное совпадение по `numeric_code`
  **ИЛИ** `qr_code`, добавление в чек по Enter, без нажатия кнопки (сканер — HID-клавиатура,
  сам шлёт Enter). Поле в фокусе, после добавления очищается и снова получает фокус.
- **Поиск товара для чека** (`GET /cashier/search`): подстрочный поиск по названию +
  совпадение по числовому коду/артикулу; список совпадений с ценой, наличием, статусом;
  кнопка «В чек» у каждой строки (разд. 3).
- **Отображение статуса товара в поиске** (разд. 3.4 / 8.5).
- **Ручной ввод веса**: весовой товар добавляется с количеством 1 и правится в inline-поле
  строки (механика 1.1); дробное количество уже поддержано.

### НЕ входит (отложено)
- **Завершение продажи** (разд. 1.2–1.6) — этап 1.3: способ оплаты, сохранение чека, списание
  остатка, движение товара — всё одной транзакцией. Модель `Receipt`/`ReceiptLine` — там.
- **Печать чека/накладной** (разд. 18) — этап 2. **Экран покупателя** (разд. 20) — этап 2.3.
- **Сортировка поиска по статусу** (разд. 8.4) — этап 8. Здесь статус только **отображается**;
  порядок вывода — простой (см. п. 4.3).
- **Поведение «товар не найден»** (разд. 16) — отложено. Пока короткое сообщение/пустой список,
  без эффектов на БД.
- **Правила отрицательных остатков** (разд. 17) — отложено. Остаток не проверяется и не меняется;
  архивные и отсутствующие в наличии товары в поиске показываются, добавление не блокируется.
- **Автоматическая привязка весов** (разд. 22.5) — весы не интегрированы, вес вводится вручную.
- Схема БД **не меняется**; новых таблиц/полей нет.

## 3. Способы ввода

Два независимых механизма над одной корзиной (`get_cart().add(...)` из 1.1):

### 3.1 Сканер / быстрый ввод кода (разд. 2)

- Отдельное поле «Сканер / код» **в фокусе по умолчанию**. Сканер «печатает» код в это поле и
  шлёт Enter → форма отправляется, товар добавляется **без нажатия кнопок**.
- Поиск **точным совпадением**: `Product.numeric_code == code` **ИЛИ** `Product.qr_code == code`.
  Сканер печатает QR-код; продавец вручную может ввести числовой код — оба сценария через это поле.
- Найден → `get_cart().add(product)` (с объединением дублей, как в 1.1). Не найден → короткое
  сообщение во фрагменте, чек не меняется (разд. 16 отложен).
- После добавления поле очищается и снова получает фокус (готово к следующему скану).

### 3.2 Поиск по названию (разд. 3)

- Поле «Поиск» сверху экрана. По мере ввода (HTMX, debounce) под полем показывается список
  совпадений. Пустой запрос → список пуст (без обращения к чеку).
- Каждая строка результата: название, цена продажи, наличие (кол-во + единица), **статус**.
  Кнопка **«В чек»** добавляет товар в текущий чек (тот же путь `add`, объединение дублей).
- Добавление из поиска — по `numeric_code` через тот же эндпоинт добавления (код уникален),
  чтобы путь добавления в корзину был **один**.

Разделение соответствует пожеланию заказчика: поиск сверху, сканер добавляет мгновенно без кнопок.

## 4. Поиск товара (сервис)

Новый модуль `app/services/product_search.py`, функция
`search_products(session, query: str, limit: int = 20) -> list[Product]`.

### 4.1 Совпадение

- **Название** — подстрочное, **регистронезависимое** совпадение (в т.ч. кириллица).
- **Числовой код / артикул** — точное или префиксное совпадение (продавец частично набирает код).
- Пустой/пробельный запрос → пустой список.

### 4.2 Регистронезависимость и портируемость

SQLite `LIKE`/`lower()` регистронезависимы только для ASCII — «Молоко» и «молоко» через `LIKE`
не совпадут (кириллица). Чтобы не привязываться к диалекту SQLite (CLAUDE.md) и не писать сырой
SQL, фильтрация и ранжирование выполняются **в Python** (`str.casefold()`): сервис читает товары
через SQLModel и отбирает совпадения в памяти.

- Для одной точки с небольшим каталогом это корректно и портируемо (Python обрабатывает Unicode
  правильно независимо от БД).
- **Точка будущей переделки:** при росте каталога/переезде на Postgres поиск переносится на
  сторону БД (индекс/`ILIKE`/полнотекстовый). Отмечено как осознанный компромисс v1.

### 4.3 Порядок и ограничение

- Порядок: сначала точные/префиксные совпадения по коду, затем совпадения по названию (по
  алфавиту). Сортировка **по статусу** (разд. 8.4) — отложена до этапа 8; статус только показывается.
- Результатов не больше `limit` (по умолчанию 20), чтобы не рисовать весь каталог.
- Архивные и отсутствующие в наличии товары **показываются** (со своим статусом/наличием),
  добавление не блокируется (разд. 17 отложен).

## 5. Вес весовых товаров (разд. 22)

Весы не интегрированы. Отдельного шага ввода веса при добавлении **не вводим** (решение заказчика):

- Любой товар (штучный и весовой) добавляется с количеством **1**.
- Вес весового товара продавец правит в **inline-поле количества строки** (механика 1.1,
  `POST /cashier/items/{line_id}/quantity`). Дробное количество уже поддержано (`CartQuantity`,
  `PositiveDecimal`), сумма строки считается `line_total` (ROUND_HALF_UP до копейки).
- Единица измерения строки берётся из товара (`unit`) и показывается рядом с количеством.

Ничего нового в корзине и валидации для веса не требуется — переиспользуется путь 1.1.

## 6. Эндпоинты

Роут (`app/routes/cashier.py`) остаётся тонким: принять → (поиск) вызвать сервис / (добавление)
найти товар и вызвать корзину → вернуть HTML-фрагмент.

| Метод | Путь | Назначение | Ответ |
|---|---|---|---|
| `GET` | `/cashier` | Экран кассы (поиск + поле сканера + корзина) | `cashier/index.html` |
| `GET` | `/cashier/search` | Поиск по названию/коду (query `q`) | фрагмент `cashier/_search_results.html` |
| `POST` | `/cashier/items` | Добавить по коду (`numeric_code` **или** `qr_code`) | фрагмент `cashier/_cart.html` |
| `POST` | `/cashier/items/{line_id}/quantity` | Изменить количество/вес строки | фрагмент (1.1) |
| `POST` | `/cashier/items/{line_id}/delete` | Убрать строку | фрагмент (1.1) |
| `POST` | `/cashier/clear` | Очистить чек | фрагмент (1.1) |

- `POST /cashier/items` **расширяется**: точное совпадение теперь по `numeric_code` ИЛИ `qr_code`
  (было — только `numeric_code`). Кнопка «В чек» в результатах поиска шлёт `numeric_code` товара
  в этот же эндпоинт.
- `GET /cashier/search` — новый: `q` из query-строки, возвращает фрагмент списка. HTMX-триггер —
  ввод в поле поиска с задержкой (`hx-trigger="keyup changed delay:300ms"`), цель — контейнер
  результатов.

HTMX: добавление в чек возвращает фрагмент корзины (`hx-target="#cart"`); поиск — фрагмент
результатов (`hx-target="#search-results"`). Имена роутов/полей — английские, текст — русский.

## 7. Интерфейс (HTMX + Jinja2)

- `templates/cashier/index.html` (обновляется):
  - **Сверху** — поле «Поиск» (по названию/коду) + контейнер `#search-results`.
  - **Поле «Сканер / код»** с `autofocus`, добавляет по Enter без кнопки; после добавления
    очищается и снова в фокусе (`hx-on::after-request`, как в 1.1).
  - Контейнер `#cart` с фрагментом корзины (без изменений логики).
- `templates/cashier/_search_results.html` (новый): список совпадений — название, цена
  (`money`-фильтр), наличие (кол-во + `unit`), статус; кнопка «В чек» (`hx-post /cashier/items`,
  `hx-target="#cart"`, скрытое поле `numeric_code`). Пустой запрос/нет совпадений — короткая
  подсказка.
- `templates/cashier/_cart.html` — без изменений (строки, inline-правка количества/веса, итоги).
- Деньги выводятся через существующий Jinja-фильтр `money`.

Минимум JS: только HTMX, без JS-фреймворков (CLAUDE.md).

## 8. Валидация на границе (Pydantic)

- Код в `POST /cashier/items` — строка; пустая → «введите код»; ненайденный → «не найден»
  (разд. 16 отложен). Совпадение — точное по `numeric_code`/`qr_code`.
- `q` в `GET /cashier/search` — строка; после `strip()` пустая → пустой список без ошибки.
- Количество/вес — схема `CartQuantity` из 1.1 (`PositiveDecimal`, строго > 0). Без изменений.

## 9. Критерии приёмки (pytest)

Денежный/складской путь и границы покрыть тестами.

**Поиск (`search_products`)**
1. Подстрочное регистронезависимое совпадение по названию, **кириллица**: `«мол»` находит
   «Молоко», `«МОЛОКО»` тоже находит «Молоко».
2. Совпадение по числовому коду (точное/префиксное) и по артикулу.
3. Пустой/пробельный запрос → пустой список.
4. Результат ограничен `limit`.
5. В результат попадают товары независимо от статуса (архивный тоже находится); статус доступен
   для отображения.

**Быстрый ввод / сканер (`POST /cashier/items`)**
6. Добавление по `numeric_code` (совместимость с 1.1) — строка в чеке.
7. **Добавление по `qr_code`** (сканер): скан QR добавляет тот же товар в чек.
8. Несуществующий код → сообщение «не найден», корзина не падает (остаётся «Чек пуст»).
9. Повторный скан того же товара объединяет строку, количество суммируется (механика 1.1).

**HTTP-слой поиска**
10. `GET /cashier/search?q=...` → 200, фрагмент со строками результатов (название, цена, статус).
11. Пустой `q` → фрагмент без результатов (подсказка), без ошибки.
12. «В чек» из результата (`numeric_code`) добавляет товар — возвращается фрагмент корзины со строкой.

**Вес**
13. Весовой товар добавляется с количеством 1; `set_quantity` дробным весом (напр. 1.5) меняет
    количество и сумму строки верно (переиспользование пути 1.1).

**Граница**
14. **Склад не трогается**: после поиска и добавления `product.quantity_current` не изменился.

## 10. Журнал решений (зачем так)

1. **Два механизма над одной корзиной.** Сканер (мгновенное добавление по Enter, без кнопок) и
   поиск по названию (список → «В чек») — разные сценарии, но добавление идёт через единственный
   путь `cart.add`/эндпоинт `/cashier/items`. Меньше расхождений в поведении.
2. **Быстрое поле матчит `numeric_code` И `qr_code`.** Сканер печатает QR-код (разд. 2), продавец
   вручную вводит числовой код — оба через одно поле, точным совпадением.
3. **Поиск фильтруется в Python (`casefold`).** SQLite `LIKE`/`lower` регистронезависимы только для
   ASCII; кириллица требует Unicode-нормализации. Python-фильтрация корректна и не привязана к
   диалекту (CLAUDE.md). Явно помечено как компромисс v1 с точкой переделки на стороне БД.
4. **Вес — правкой строки, без отдельного шага.** Решение заказчика: весовой товар добавляется с
   количеством 1 и правится в inline-поле (механика 1.1). Ввода веса «на входе» не добавляем.
5. **Статус показываем, не сортируем по нему.** Разд. 3.4/8.5 требуют видимости статуса; логика
   сортировки по статусу (разд. 8.4) отложена до этапа 8.
6. **Склад и движение не трогаем.** Подтверждается тестом (`quantity_current` неизменен) — граница
   с 1.3 сохранена.
