# ТЗ — Завершение продажи

> Этап **1.3** roadmap (завершение продажи). Ядро денежного и складского пути.
> Макет: разд. 1.2 (способы оплаты), 1.3 (состав чека), 1.5 (деньги не привязаны к системе),
> 1.6 (электронное сохранение чека), 21 (терминал не интегрирован).
> Опирается на 1.1/1.2 (`docs/specs/тз-1.1-экран-кассы.md`, `docs/specs/тз-1.2-ввод-товара-в-чек.md`):
> корзина в памяти (`app/services/cart.py`), `CartView` с `subtotal`/`rounding`/`grand_total`,
> снимок цены, хелперы `line_total`/`round_total_up` (`app/services/money.py`).
> Опирается на 0.2 (`docs/specs/тз-0.2-база-товаров.md`): `Product` (`quantity_current`,
> `price_sell`, `unit`), `Movement` (`operation_type`, `datetime`, `quantity`).
> Источник истины — `docs/Макет системы.txt`. При противоречии — сверяться с макетом.

## 1. Назначение

Закрыть главный денежный путь: превратить **живой чек в памяти** (1.1/1.2) в
**завершённую продажу**. По кнопке «Завершить продажу» с выбранным способом оплаты система
**одной транзакцией**:

1. сохраняет электронный чек (шапка + строки) с **зафиксированными** суммами (разд. 1.6);
2. списывает остаток каждого товара (`quantity_current`);
3. пишет движение товара `продажа` на каждую строку (правило 5 CLAUDE.md);
4. очищает корзину и показывает подтверждение.

Либо всё целиком, либо ничего (правило 2 — продажа атомарна). После этого чек считается
завершённым и **не редактируется** (правило 4); исправление — только через возврат (этап 4).

Результат: продажа реально меняет склад и остаётся в базе; итог, строки и способ оплаты
сохранены на момент пробития и не «поедут» при последующей правке карточек или правил.

## 2. Граница этапа

### Входит
- Модели `Receipt` (шапка чека) и `ReceiptLine` (строки чека) — новые таблицы.
- Последовательный **номер чека** (`receipt_number`) через отдельный счётчик (по образцу
  `ProductCodeCounter`), независимый от `id`.
- Способ оплаты как пометка: `наличные` / `безналичные` (разд. 1.2, 21).
- Сервис `complete_sale(...)` — сохранение чека + списание остатка + движения `продажа`
  **в одной транзакции БД**.
- Фиксация сумм: `subtotal`, `rounding`, `total` чека и `price_sell`, `quantity`, `total`
  каждой строки записываются как снимок, а не пересчитываются при просмотре (разд. 1.6, п. 5).
- Движение `продажа` на каждую строку: `datetime`, `operation_type=sale`,
  **отрицательное** количество (−quantity, дельта остатка).
- Эндпоинт `POST /cashier/complete` + UI (выбор способа оплаты, кнопка, подтверждение).
- Очистка корзины после успешной продажи.
- Валидация способа оплаты на границе (Pydantic).

### НЕ входит (отложено)
- **Печать чека** (разд. 18.1–18.4) — этап 2.1. Здесь чек только **сохраняется**, не печатается.
- **Печать накладной при продаже** и галочка «Распечатать накладную» (разд. 1.4, 18.5–18.7) —
  этап 2.2. В 1.3 галочку не вводим.
- **Экран покупателя** (разд. 20) — этап 2.3.
- **Возврат и исправление завершённого чека** (разд. 14, 15.2–15.3) — этап 4.
- **Просмотр/история чеков, отчёт по дню** (разд. 23) — этап 7. Здесь чек пишется в БД, но
  экрана списка чеков нет.
- **Правила отрицательных остатков** (разд. 17) — отложено. Остаток при продаже **списывается
  как есть** (может уйти в минус), продажа не блокируется по наличию. Явный компромисс v1.
- **Открытие денежного ящика, управление терминалом, подтверждение оплаты** (разд. 1.5, 21.3) —
  вне системы принципиально. Способ оплаты — только пометка в чеке.

## 3. Модель данных (новые таблицы)

Продажа делает чек **постоянным**, поэтому вводим таблицы (обещано в 1.1/1.2 как «модель
`Receipt`/`ReceiptLine` — в 1.3»). Строки чека — **самодостаточный снимок**: не зависят от
последующей правки карточки товара (правило 4, снимок цены из 1.1).

### 3.1 `PaymentMethod` (enum, `app/models/receipt.py`)

| Значение | Метка (рус.) |
|---|---|
| `cash` | наличные |
| `card` | безналичные |

По образцу `UnitEnum`/`ProductStatus` (`str, Enum`, значение — русский текст для отображения).

### 3.2 `Receipt` (шапка чека)

| Поле | Тип | Примечание |
|---|---|---|
| `id` | int PK | Технический ключ. |
| `receipt_number` | int, unique | Последовательный номер чека (см. 3.4). Человекочитаемый. |
| `datetime` | datetime | Дата и время продажи (разд. 1.3). |
| `payment_method` | `PaymentMethod` | Способ оплаты (пометка, разд. 1.2/1.5). |
| `subtotal` | int (копейки) | **Зафиксированная** точная сумма строк. |
| `rounding` | int (копейки) | **Зафиксированная** надбавка округления до ₽ (≥ 0). |
| `total` | int (копейки) | **Зафиксированный** итог к оплате (subtotal + rounding). |

### 3.3 `ReceiptLine` (строка чека)

| Поле | Тип | Примечание |
|---|---|---|
| `id` | int PK | |
| `receipt_id` | int FK → `receipt.id` | |
| `product_id` | int FK → `product.id` | Ссылка на товар (для статистики/движений). |
| `name` | str | **Снимок** названия на момент продажи. |
| `unit` | `UnitEnum` | **Снимок** единицы измерения. |
| `price_sell` | int (копейки) | **Снимок** цены продажи. |
| `quantity` | Decimal (`quantity_column()`) | Проданное количество (штучное/весовое). |
| `total` | int (копейки) | **Зафиксированная** сумма строки (`line_total`). |

`quantity` — через `quantity_column()` из `app/models/types.py` (как `Product.quantity_current`),
чтобы дробный вес хранился корректно.

### 3.4 Счётчик номеров чека

Отдельная таблица-счётчик `ReceiptNumberCounter` (id=1, `last_value`), по образцу
`ProductCodeCounter` (`app/models/counter.py`). Номер получаем внутри той же транзакции
продажи с блокировкой строки (`.with_for_update()`, как в `create_product`) и инкрементом —
номера последовательны и без гонок. Инициализация записи `id=1, last_value=0` — в `init_db()`
(рядом с существующей инициализацией `ProductCodeCounter`).

> **Почему отдельный номер, а не `id`:** заказчик просил решение «на будущее». Отдельный
> `receipt_number` не зависит от внутренних id (удаления/служебные записи его не сдвигают),
> человекочитаем и переживает будущий переезд на Postgres. Дневной сброс нумерации (при
> необходимости для отчётов, этап 7) добавляется поверх сохранённого `datetime`, без ломки схемы.

## 4. Сервис продажи (`app/services/sale.py`)

`complete_sale(session: Session, cart: Cart, payment_method: PaymentMethod) -> Receipt`.

Повторяет проверенный паттерн `product_service.create_product`: `flush()` для получения id,
запись связанных сущностей, единый `commit()` в конце (правило 2 — атомарность).

Шаги (всё внутри одной транзакции):

1. `view = cart.view()`; если `view.lines` пуст → `ValueError("Чек пуст")` (продажу пустого
   чека не завершаем; БД не трогается).
2. Взять и инкрементировать `ReceiptNumberCounter` (`.with_for_update()`), получить
   `receipt_number`.
3. Создать `Receipt` с **зафиксированными** `subtotal`/`rounding`/`total` из `view`
   (значения не пересчитываются — берутся ровно те, что видел продавец), `datetime = now`,
   `payment_method`. `session.add(receipt)`, `session.flush()` → `receipt.id`.
4. Для каждой `CartLine`:
   - создать `ReceiptLine` (снимок `name`, `unit`, `price_sell`, `quantity`, `total`);
   - `product = session.get(Product, line.product_id)`; уменьшить
     `product.quantity_current -= line.quantity` (может уйти в минус — разд. 17 отложен);
   - создать `Movement(product_id, datetime=now, quantity=-line.quantity,
     operation_type=OperationType.sale)` — **отрицательная дельта** остатка.
5. `session.commit()`.
6. **Вне транзакции**, только после успешного `commit`: `cart.clear()` — очистить корзину
   кассы. Если `commit` упал — корзина сохраняется, продажа не потеряна.
7. Вернуть `receipt` (с `receipt_number`, `total`) для подтверждения в UI.

**Атомарность.** Любая ошибка до `commit` → откат: ни чек, ни строки, ни списание, ни движения
не сохранятся; корзина остаётся на экране. Никаких полусписанных состояний (правило 2).

**Один общий `now`** на весь чек: шапка, строки и движения датируются одним моментом
(как единый `now` в `create_product`).

## 5. Деньги — фиксация, не пересчёт

Суммы уже посчитаны корзиной (1.1): `line.total = line_total(price_sell, quantity)`,
`view.subtotal`, `view.rounding`, `view.grand_total = round_total_up(subtotal)`. В 1.3 они
**копируются в чек как есть** (обещание из `тз-1.1` разд. 5.2):

- Строки чека точны до копейки; итог округлён вверх до целой ₽; разница — в `rounding`
  (в чеке отображается/печатается отдельной строкой «Округление», печать — этап 2).
- `Receipt.total == Receipt.subtotal + Receipt.rounding` и `== round_total_up(subtotal)` —
  инвариант, проверяется тестом.
- Пересчёт при просмотре/печати **запрещён**: смена правила округления или правка карточки
  не должны менять уже завершённый чек (правило 4). Хелперы `line_total`/`round_total_up`
  переиспользуются только на этапе сборки корзины, не поверх сохранённого чека.

Деньги — `int` в копейках, количество — `Decimal` (правило 1). Ничего нового в `money.py`
не требуется.

## 6. Эндпоинт

Роут (`app/routes/cashier.py`) остаётся тонким: провалидировать способ оплаты → вызвать
сервис → вернуть HTML. Вся логика — в `sale.py`.

| Метод | Путь | Назначение | Ответ |
|---|---|---|---|
| `POST` | `/cashier/complete` | Завершить продажу (Form: `payment_method`) | фрагмент `cashier/_cart.html` |

- Успех → `_cart.html` с пустой корзиной и баннером подтверждения (`sale_result`: номер чека,
  итог, способ оплаты). HTMX `hx-target="#cart"`, `hx-swap="innerHTML"` — как остальные мутации.
- Пустой чек / неизвестный способ оплаты → тот же фрагмент с `error`, корзина не меняется,
  чек не создаётся.

Существующие эндпоинты кассы (1.1/1.2) не меняются.

## 7. Интерфейс (HTMX + Jinja2)

Обновляется `templates/cashier/_cart.html` (при непустом чеке, рядом с «Очистить чек»):

- Блок **«Способ оплаты»**: два варианта-радио `наличные` / `безналичные` (значения `cash`/`card`,
  метки — русские из `PaymentMethod`), по умолчанию `наличные`.
- Кнопка **«Завершить продажу»** — форма `hx-post="/cashier/complete"`, `hx-target="#cart"`,
  `hx-swap="innerHTML"`, передаёт выбранный `payment_method`.
- Блок оплаты и кнопка показываются **только при непустом чеке** (при пустом — уже есть
  «Чек пуст»).
- **Баннер подтверждения** (`sale_result`): «Продажа завершена. Чек №{{ number }},
  оплата {{ payment_method }}, итого {{ total | money }} ₽» — над пустой корзиной после успеха.

Деньги — через существующий Jinja-фильтр `money`. Минимум JS: только HTMX (CLAUDE.md).
Контекст `_cart_context` в роуте расширяется опциональным `sale_result`.

## 8. Валидация на границе (Pydantic)

Новая схема `SaleComplete` (`app/schemas/sale.py`):

- `payment_method: PaymentMethod` — значение обязано быть из перечня (`cash`/`card`); иное →
  отказ, сообщение во фрагменте, чек не создаётся (правило 3).

Проверка «чек не пуст» — в сервисе (`complete_sale` кидает `ValueError`), т.к. пустота — это
состояние корзины, а не входных данных формы. Роут ловит `ValidationError` (способ оплаты) и
`ValueError` (пустой чек) и отдаёт сообщение.

## 9. Критерии приёмки (pytest)

Денежный и складской путь — покрыть тестами (`tests/test_sale.py`; при необходимости
расширить `tests/test_cart.py` по HTTP-слою).

**Сервис `complete_sale`**
1. Создаёт `Receipt` с `receipt_number`, `datetime`, `payment_method` и **зафиксированными**
   `subtotal`/`rounding`/`total`, равными `cart.view()` на момент продажи.
2. Инвариант: `receipt.total == subtotal + rounding == round_total_up(subtotal)`.
3. Создаёт `ReceiptLine` на каждую строку корзины со снимком `name`/`unit`/`price_sell`/
   `quantity`/`total`, совпадающим с корзиной.
4. **Списание остатка**: `product.quantity_current` уменьшается на `quantity` каждой строки
   (в т.ч. дробный вес — уменьшение на `Decimal`).
5. **Движение**: на каждую строку пишется `Movement(operation_type=sale)` с **отрицательным**
   `quantity` (= −количество) и общим `datetime` чека.
6. Номера чеков **последовательны**: две продажи подряд → `receipt_number` растёт на 1.
7. **Атомарность**: при ошибке в середине (напр. подменить `session.commit`, чтобы бросить) —
   ни `Receipt`, ни `ReceiptLine`, ни `Movement` не сохранены, `quantity_current` не изменён.
8. **Пустой чек** → `ValueError`; чек/движения не создаются, склад не тронут.
9. **Снимок цены**: после `complete_sale` изменение `product.price_sell` не меняет
   `ReceiptLine.price_sell`/`total` сохранённого чека.
10. Корзина **очищена** после успешной продажи; при неудаче (пустой чек) — не очищена.

**HTTP-слой (`POST /cashier/complete`)**
11. Валидный `payment_method` при непустом чеке → 200, баннер подтверждения (номер чека,
    итог, способ оплаты), корзина пуста; в БД появился `Receipt` + строки; `quantity_current`
    уменьшился.
12. Пустой чек → сообщение об ошибке, `Receipt` в БД не создан.
13. Неизвестный/пустой `payment_method` → сообщение об ошибке, чек не создан, корзина цела.

**Граница**
14. Способ оплаты корректно различается: продажа `card` и `cash` сохраняют соответствующий
    `payment_method` (проверка, что система только фиксирует пометку, разд. 1.5).

## 10. Журнал решений (зачем так)

1. **Чек становится постоянным ровно здесь.** До 1.3 чек жил в памяти и не трогал склад
   (1.1/1.2). Модель `Receipt`/`ReceiptLine` вводится в 1.3, как и обещано — не раньше.
2. **Одна транзакция.** Чек + списание + движения = единый `commit` (правило 2), по образцу
   `create_product`. Полусписанных состояний нет; при ошибке корзина сохраняется на экране.
3. **Фиксация сумм, не пересчёт.** `subtotal`/`rounding`/`total` и снимок строк пишутся как
   есть. Завершённый чек не пересчитывается при смене правила округления или правке карточки
   (правила 1 и 4). Хелперы `money.py` работают только на сборке корзины.
4. **Движение — отрицательная дельта.** Продажа пишет `−quantity` (приход из 0.3 пишет
   `+quantity`). Сумма движений товара = его остаток; знак кодирует направление. Удобно для
   отчётов (этап 7) и возвратов (этап 4).
5. **Отдельный последовательный номер чека.** Решение «на будущее»: `receipt_number` не
   привязан к внутреннему `id`, человекочитаем, переживает переезд на Postgres; дневной сброс
   для отчётов достраивается поверх `datetime` без ломки схемы.
6. **Способ оплаты — только пометка.** Система не управляет терминалом/ящиком и не сверяет
   оплату (разд. 1.5, 21.3). Фиксируем `cash`/`card` в чеке — и всё.
7. **Остаток может уйти в минус.** Правила отрицательных остатков (разд. 17) отложены; продажа
   не блокируется по наличию, как и в 1.1/1.2. Явный компромисс v1.
8. **Печать и накладная — не здесь.** 1.3 закрывает софтовое ядро продажи; печать чека (2.1),
   накладная при продаже с галочкой (2.2) и экран покупателя (2.3) идут отдельными этапами
   поверх сохранённого чека.
