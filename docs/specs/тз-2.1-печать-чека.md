# ТЗ — Печать чека

> Этап **2.1** roadmap (печать чека). Первый пункт этапа 2 «железо вокруг продажи».
> Макет: разд. 18.1 (назначение), 18.2 (устройство), 18.3 (момент печати), 18.4 (содержание чека).
> Опирается на 1.3 (`docs/specs/тз-1.3-завершение-продажи.md`): чек уже сохранён —
> `Receipt`/`ReceiptLine` (`app/models/receipt.py`), суммы зафиксированы (`subtotal`/`rounding`/
> `total`), `complete_sale` возвращает `Receipt`. Печать строится **поверх сохранённого чека**.
> Опирается на 0.2: `UnitEnum` (единица измерения строки).
> Источник истины — `docs/Макет системы.txt`. При противоречии — сверяться с макетом.

---

## 1. Назначение

После завершения продажи (1.3) система **печатает чек** (макет 18.1, 18.3). Сейчас реального
принтера нет, поэтому чек «печатается» в файл, но по тому же пути, что и на будущем устройстве:
формирование чека отделено от транспорта, и подключение принтера сведётся к смене бэкенда в
конфиге, без правки логики формирования (CLAUDE.md, раздел «Железо»).

Печать делится на два изолированных слоя:

1. **Формирование** (`app/services/receipt_render.py`) — из сохранённого `Receipt` собирается
   текст чека шириной **48 символов** (лента 80 мм, Шрифт A). Чистая функция, без ввода-вывода,
   полностью тестируется без железа.
2. **Транспорт** (`app/hardware/receipt_printer.py`) — куда уходит сформированный чек. Сейчас
   бэкенд `file` пишет чек в `data/receipts/`; бэкенд реального ESC/POS-устройства подключается
   позже сменой параметра в `config.py`. Слой транспорта **не знает** о составе чека — получает
   готовый текст.

Печать — **после** успешного сохранения продажи и **best-effort**: сбой печати не откатывает
уже завершённую продажу (чек в БД, склад списан). Продажа важнее бумажки.

## 2. Граница этапа

### Входит
- `app/services/receipt_render.py` — `render_receipt_text(receipt, lines, width=48) -> str`:
  сборка текста чека из `Receipt` + его `ReceiptLine` по составу 18.4.
- `app/hardware/receipt_printer.py` — абстракция `ReceiptPrinter` (протокол) + бэкенды:
  - `FileReceiptPrinter` (по умолчанию) — пишет чек в `data/receipts/`;
  - фабрика `get_receipt_printer()` выбирает бэкенд по `config.py`.
- Подключение `python-escpos`: бэкенд `file` прогоняет текст через `escpos.printer.Dummy`
  и сохраняет **реальный ESC/POS-поток** в `data/receipts/чек-NNNN.escpos`, а человекочитаемый
  текст — в `data/receipts/чек-NNNN.txt`. Оба артефакта из одного сформированного текста.
- Вызов печати в `POST /cashier/complete` **после** успешного `complete_sale`, обёрнутый в
  `try/except`: сбой печати логируется, продажа считается завершённой, продавцу показывается
  успех (при желании — с пометкой «чек не напечатан»).
- Параметры печати в `config.py`: бэкенд, папка чеков, ширина строки, текст шапки/подвала.
- Тесты формирования (состав/ширина) и транспорта (файл создан, escpos-байты непусты,
  продажа не падает при ошибке печати).

### НЕ входит (отложено)
- **Реальное ESC/POS-устройство** (USB/Serial/Lan) — интерфейс и бэкенд `device` заложены,
  но включение и параметры подключения задаются позже, когда появится принтер. Сейчас
  бэкенд по умолчанию — `file`.
- **Оформление чека**: логотип-графика, штрихкод/QR на чеке, тонкая типографика, денежный
  ящик, звуковой сигнал. Сейчас — простой текстовый макет (18.7: содержание дополняется позже).
- **Печать накладной при продаже** и галочка «Распечатать накладную» (разд. 18.5–18.7) — этап 2.2.
- **Экран покупателя** (разд. 20) — этап 2.3.
- **Повторная печать / история чеков / отчёт по дню** (разд. 23) — этап 7. Здесь только печать
  в момент продажи.
- **Изменение данных чека при печати** — запрещено: печатаем ровно то, что сохранено (правило 4).

## 3. Архитектура: формирование ↔ транспорт

Ключевое требование CLAUDE.md: «Слой печати держать изолированным от сборки, чтобы смена
`файл → устройство` не задевала логику формирования». Реализуем разделением:

```
complete_sale(...) -> Receipt        # 1.3, уже есть: чек сохранён в БД
        │
route POST /cashier/complete
        │  receipt + его строки (session.get / select ReceiptLine)
        ▼
render_receipt_text(receipt, lines)  # services: ЧТО печатать (48 колонок). Чистая функция.
        │  str
        ▼
get_receipt_printer().print(receipt.receipt_number, text)   # hardware: КУДА печатать.
        └── FileReceiptPrinter → data/receipts/чек-NNNN.txt + .escpos (через python-escpos)
        └── (позже) DeviceReceiptPrinter → USB/Serial/Lan
```

- Формирование не знает, куда уйдёт чек. Транспорт не знает состав чека — получает готовый текст
  и номер (для имени файла). Это и есть изоляция: устройство меняет только `hardware/`, а
  `receipt_render.py` не трогается.
- Строки чека берём из БД (`ReceiptLine` по `receipt_id`), а **не** из корзины: корзина к моменту
  печати уже очищена (1.3), и печатать надо снимок сохранённого чека, а не живое состояние.

## 4. Формирование чека (`app/services/receipt_render.py`)

`render_receipt_text(receipt: Receipt, lines: list[ReceiptLine], width: int = 48) -> str`.

Состав по макету 18.4 (сверху вниз):

| Блок 18.4 | В чеке |
|---|---|
| блок магазина / рекламный блок | `receipt_header` из конфига (по центру), напр. «МАГАЗИН» |
| — | «Чек №{receipt_number}» |
| список товаров + сумма по каждому | по строке: название; ниже `{кол-во} {ед} × {цена}` … справа `{сумма}` |
| итоговая сумма | «Итог по строкам» (`subtotal`), «Округление» (`rounding`, если > 0), «ИТОГО» (`total`) |
| способ оплаты | «Оплата: {payment_method.value}» (наличные/безналичные) |
| дата и время покупки | `datetime` в формате `ГГГГ-ММ-ДД ЧЧ:ММ` |
| — | `receipt_footer` из конфига (по центру), напр. «Спасибо за покупку!» |

Правила формирования:
- **Ширина строго ≤ `width` (48)**. Суммы прижаты вправо, разделители — строка из `-` на всю
  ширину. Длинное название товара переносится/обрезается по ширине (название — на своей строке,
  «кол-во × цена … сумма» — на следующей, поэтому имя занимает всю ширину).
- **Деньги** форматируются из копеек в «148.50» (тот же вид, что Jinja-фильтр `money` в
  `cashier.py`). Вынести форматирование в общий хелпер (`app/services/money.py`,
  `format_money(kopecks) -> str`) и переиспользовать в фильтре роутов и в рендере, чтобы формат
  был один. Пересчёта сумм нет — берём `receipt.subtotal/rounding/total` и `line.total` как есть
  (правило 4: завершённый чек не пересчитывается).
- **Количество**: `Decimal` печатается без «хвоста» (`2`, а не `2.000`; `1.5` для веса).
- Функция **чистая**: без файлов и БД, принимает готовые объекты. Это делает состав чека
  (18.4) полностью тестируемым без железа и без веб-слоя.

Пример (48 колонок, оформление черновое, 18.7 — дополним позже):

```
                    МАГАЗИН
                   Чек №0001
------------------------------------------------
Хлеб бородинский
  2 шт x 45.00                            90.00
Молоко 3.2%
  1 шт x 79.99                            79.99
------------------------------------------------
Итог по строкам                         169.99
Округление                                0.01
ИТОГО                                   170.00
Оплата: наличные
2026-07-03 14:22
             Спасибо за покупку!
```

## 5. Транспорт печати (`app/hardware/receipt_printer.py`)

Абстракция транспорта. Формирование сюда не попадает.

- **Протокол** `ReceiptPrinter`: метод `print(receipt_number: int, text: str) -> None`.
- **`FileReceiptPrinter`** (бэкенд `file`, по умолчанию):
  1. пишет человекочитаемый текст в `data/receipts/чек-{receipt_number:04d}.txt`;
  2. прогоняет тот же текст через `python-escpos` (`escpos.printer.Dummy`): `d.text(text)`,
     `d.cut()`, и сохраняет `d.output` (реальные ESC/POS-байты) в
     `data/receipts/чек-{receipt_number:04d}.escpos`. Так конвейер ESC/POS проверяется уже
     сейчас, до появления принтера. Кириллица — через авто-кодировку python-escpos (кодовая
     страница вроде CP866); при проблемах кодировки бэкенд всё равно сохраняет `.txt`.
  Папку `data/receipts/` создаёт при первой печати (как `init_db` создаёт `data/`).
- **`DeviceReceiptPrinter`** (бэкенд `device`) — **заложен, но не активируется в 2.1**: обёртка
  над `escpos.printer.Usb/Serial/Network` (интерфейсы принтера — USB+Serial+Lan). Параметры
  подключения — в конфиге, включается при появлении железа. В 2.1 достаточно оставить
  точку расширения (заглушка/`NotImplementedError` с понятным сообщением) — логика формирования
  при этом не меняется.
- **`get_receipt_printer()`** — фабрика: по `settings.receipt_printer_backend` возвращает нужный
  бэкенд. `data/receipts/` — в `.gitignore` (как `data/store.db`; артефакты чеков в репозиторий
  не кладём).

## 6. Момент печати и обработка сбоя (`app/routes/cashier.py`)

Печать встраивается в существующий `POST /cashier/complete` **после** успешного `complete_sale`
(макет 18.3: сохранить → отправить на печать). Роут остаётся тонким:

```
receipt = complete_sale(session, get_cart(), data.payment_method)   # уже есть (1.3)
try:
    lines = session.exec(select(ReceiptLine).where(ReceiptLine.receipt_id == receipt.id)).all()
    text = render_receipt_text(receipt, lines)
    get_receipt_printer().print(receipt.receipt_number, text)
except Exception:
    log.exception("Печать чека №%s не удалась", receipt.receipt_number)   # best-effort
    printed = False
else:
    printed = True
# в любом случае — продажа завершена, показываем баннер успеха (при желании — с пометкой)
```

- **Продажа уже зафиксирована** (`complete_sale` сделал `commit`). Сбой печати **не откатывает
  чек и не трогает склад** — иначе баг печати превращался бы в потерю продажи (нарушение
  приоритета «деньги/остатки»). Печать — отдельный best-effort шаг после транзакции.
- Баннер `sale_result` (1.3) дополняется опциональным флагом `printed`, чтобы при неудаче печати
  показать «Продажа завершена, но чек не напечатан» (продавец допечатает, когда появится принтер;
  повторная печать — этап 7). При успехе — как сейчас.
- Порядок и атомарность продажи (1.3) **не меняются**. Печать не участвует в транзакции БД.

## 7. Конфигурация (`app/config.py`)

Экрана настроек нет (CLAUDE.md), параметры печати — в `config.py` (env-переопределяемы,
префикс `STORE_`):

| Параметр | Тип | По умолчанию | Назначение |
|---|---|---|---|
| `receipt_printer_backend` | str | `"file"` | `file` \| `device` \| `null` (без печати). |

Параметры реального устройства (тип подключения, путь/адрес) добавляются вместе с бэкендом
`device`, когда появится принтер. В 2.1 не заполняются.

## 8. Ширина ленты и символы

Принтер (скриншот заказчика, POScenter RP-100 USE): лента **80 мм**, ширина печати до 80 мм,
203 dpi, Шрифт A **12×24 точки**, Шрифт B **9×17 точек**.

- Печатная область 80-мм ленты ≈ 576 точек. **Шрифт A: 576 / 12 = 48 символов** в строке —
  это рабочая ширина v1 (`receipt_line_width = 48`), читаемо и стандартно для 80 мм.
- Шрифт B (9 точек) даёт 64 символа — можно уплотнить позже, если понадобится больше колонок;
  ширина вынесена в конфиг, менять — одна строка.
- Весь макет чека верстается под 48 символов; тест проверяет, что **ни одна строка не длиннее
  `width`**.

## 9. Критерии приёмки (pytest)

`tests/test_receipt_render.py` (формирование) и `tests/test_receipt_printer.py` (транспорт);
HTTP-слой — расширение `tests/test_sale.py`.

**Формирование `render_receipt_text`**
1. В тексте присутствуют все блоки 18.4: шапка магазина, список товаров, сумма по каждой
   строке, «ИТОГО», способ оплаты (рус. метка), дата и время; номер чека.
2. **Ни одна строка не длиннее `width`** (48) — на длинных названиях и больших суммах.
3. Суммы = зафиксированные значения чека (`line.total`, `receipt.subtotal/rounding/total`),
   **без пересчёта**; формат «148.50».
4. Строка «Округление» присутствует, когда `rounding > 0`, и отсутствует, когда `rounding == 0`.
5. Количество печатается без лишних нулей: `2` (шт) и `1.5` (кг).
6. Способ оплаты различается: `cash` → «наличные», `card` → «безналичные».

**Транспорт `FileReceiptPrinter`**
7. `print(number, text)` создаёт `data/receipts/чек-NNNN.txt` с этим текстом (в изолированной
   temp-папке из `receipts_dir`).
8. Создаётся `.escpos` с **непустым** ESC/POS-потоком (python-escpos через `Dummy`), поток
   содержит команду реза в конце.
9. Папка `receipts_dir` создаётся автоматически, если её не было.

**HTTP-слой `POST /cashier/complete`**
10. Успешная продажа при бэкенде `file` → в `receipts_dir` появился файл чека с номером
    завершённого чека; баннер успеха, корзина пуста (регрессия 1.3 сохранена).
11. **Сбой печати не ломает продажу**: при бэкенде, чей `print` бросает исключение, эндпоинт
    возвращает 200, `Receipt` в БД сохранён, склад списан, баннер показан (при неудаче — с
    пометкой `printed=False`).
12. Бэкенд `null` → продажа проходит, файлы не создаются, ошибок нет.

**Регрессия**
13. Все тесты 1.3 (`complete_sale`, атомарность, номера чеков) проходят без изменений —
    печать не влияет на транзакцию продажи.

## 10. Журнал решений (зачем так)

1. **Формирование отделено от транспорта.** `receipt_render.py` (что печатать) не знает про
   устройство; `hardware/receipt_printer.py` (куда) не знает про состав чека. Подключение
   принтера = смена бэкенда в конфиге, без правки логики формирования (CLAUDE.md).
2. **Печатаем сохранённый чек, не корзину.** К моменту печати корзина очищена (1.3). Источник —
   `Receipt`/`ReceiptLine` из БД: снимок на момент продажи, соответствует правилу 4.
3. **Печать — best-effort после `commit`.** Продажа уже зафиксирована; сбой принтера не
   откатывает чек и не трогает склад. Приоритет «деньги/остатки» выше печати бумаги.
4. **`python-escpos` сейчас, но вывод в файл.** Бэкенд `file` через `Dummy` генерирует реальные
   ESC/POS-байты — весь конвейер печати проверен до появления железа; `.txt` рядом даёт
   человекочитаемый чек для глаз и тестов. Реальное устройство — смена бэкенда на `device`.
5. **48 символов (Шрифт A, 80 мм).** Рабочая ширина v1; вынесена в конфиг — переход на Шрифт B
   (64) или другую ленту не трогает вёрстку.
6. **Оформление — минимальное (18.7).** Состав по 18.4 обязателен; логотип, штрихкод на чеке,
   ящик, сигнал — позже. Сейчас важны состав, ширина и изоляция слоя.
7. **Форматирование денег — общий хелпер.** `format_money` в `money.py` переиспользуется в
   Jinja-фильтре и в рендере, чтобы формат «148.50» был единым.
