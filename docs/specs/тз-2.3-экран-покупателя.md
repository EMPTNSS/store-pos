> Этап **2.3** roadmap. Последний пункт этапа 2 «железо вокруг продажи» — конец v1.
> Макет: разд. 20.1 (назначение), 20.2 (что показывать во время продажи), 20.3 (требования

> снимок цены.
> Опирается на 1.3 (`docs/specs/тз-1.3-завершение-продажи.md`): `complete_sale` возвращает
> `Receipt`, корзина очищается только после `commit`.
> Опирается на 2.1/2.2: переиспользует `format_money` (`app/services/money.py`) и паттерн
> `config.py`-параметров без экрана настроек.
> Источник истины — `docs/Макет системы.txt`. При противоречии — сверяться с макетом.

## 1. Назначение

С обратной стороны кассы — **второй экран для покупателя** (макет 20.1): покупатель видит
текущий чек и «ощущает контроль над покупкой». Во время формирования чека экран показывает
**крупно и коротко** (20.3) добавленные товары, количество, цену и итоговую сумму (20.2).
После завершения покупки — сообщение **«Спасибо за покупку»** (20.4).

Экран покупателя — **пассивное зеркало**: только читает состояние, ничего не меняет в корзине,
БД или складе. Вся денежная/складская логика уже в 1.1/1.3 и здесь не дублируется.

Результат: во втором окне видно живое состояние чека (обновляется по мере пробития товара),
а после оплаты — благодарность, затем экран ожидания следующего покупателя.

## 2. Граница этапа

### Входит
- `app/routes/customer.py` — новый роутер:
  - `GET /customer` — полная страница экрана покупателя (`customer/index.html`).
  - `GET /customer/state` — фрагмент текущего состояния (`customer/_display.html`),
    который тянется опросом.
- `app/services/customer_display.py`:
  - маркер последней завершённой продажи (`LastSale`: `number`, `total`, `at`) — синглтон
    в памяти процесса, по образцу корзины;
  - `mark_sale_completed(receipt)` — записать маркер (вызывается из кассы после успешной продажи);
  - **чистая** `resolve_display_state(cart, last_sale, now, thanks_seconds) -> DisplayState` —
    решает, что показать (продажа / благодарность / ожидание). Тестируется без времени и веба.
- Три состояния экрана: **продажа** (чек непуст), **благодарность** (чек пуст, продажа была
  ≤ N сек назад), **ожидание** (иначе).
- Опрос состояния на чистом HTMX: `hx-get="/customer/state"`, `hx-trigger="load, every {poll}"`,
  `hx-swap="innerHTML"`.
- Вызов `mark_sale_completed(receipt)` в `POST /cashier/complete` после успешного
  `complete_sale` (best-effort, вне транзакции продажи).
- Второе окно pywebview в `run.py`, **по флагу конфига** (по умолчанию выключено).
- Параметры в `config.py`: флаг окна, интервал опроса, длительность благодарности, тексты.
- Регистрация роутера в `app/main.py`.
- Тесты: `resolve_display_state` (три ветки + граница таймаута), HTTP-слой `/customer` и
  `/customer/state`, маркер после продажи, регрессия кассы.

### НЕ входит (отложено)
- **Точное позиционирование окна на второй монитор** — зависит от железа/ОС; в v1 окно
  открывается по флагу (fullscreen), раскладку по мониторам настраивает продавец/ОС. Автоматический
  выбор монитора — когда появится реальный второй экран.
- **SSE / веб-сокеты** — обновление через HTMX-опрос (решение заказчика). Мгновенная доставка
  событий отложена; на localhost опрос ~1с достаточен.
- **Отдельная тема оформления/брендинг** экрана — сейчас минимальная крупная вёрстка (20.3);
  логотип, реклама, анимации — позже.
- **Изменение чека с экрана покупателя** — экран только читающий (никаких кнопок/форм для
  покупателя).
- **Показ способа оплаты/накладной/деталей округления** — «без лишних деталей» (20.3):
  на экране только товары, количество, цена и итог; служебное — на кассе.

## 3. Архитектура: одно состояние, два окна

```
                 get_cart()  ──────────────┐   (синглтон корзины, 1.1)
                     ▲                      │
  окно КАССЫ  ──► POST /cashier/*  ──► мутации корзины
                     │                      │
  POST /cashier/complete ─► complete_sale ─► Receipt (1.3)
                     │                      │
                     └─► mark_sale_completed(receipt)   ← маркер последней продажи
                                            │
                 customer_display (синглтон: LastSale)
                     ▲                      │
  окно ПОКУПАТЕЛЯ ─► GET /customer/state (опрос every 1s)
                     │
        resolve_display_state(cart.view(), last_sale, now, thanks_seconds)
                     │
        ┌────────────┼─────────────────────┐
     "продажа"   "благодарность"        "ожидание"
   (чек непуст)  (пусто, ≤ N сек)      (пусто, > N сек)
```

- Оба окна — **один процесс**, читают тот же синглтон корзины. Экран покупателя **не пишет** в
  корзину/БД — только читает `get_cart().view()` и маркер. Гонок нет: чтение read-only, процесс
  один (uvicorn, GIL).
- Маркер продажи ставится **после** `complete_sale` (продажа уже зафиксирована), вне транзакции —
  как печать чека в 2.1. Сбой маркировки не должен ломать продажу (обернуть аккуратно), но
  маркер — просто запись в память, риск минимален.

## 4. Состояние экрана и маркер продажи (`app/services/customer_display.py`)

`LastSale` (dataclass): `number: int`, `total: int` (копейки), `at: datetime`.

Синглтон `_last_sale: LastSale | None` + `get_last_sale()` / `mark_sale_completed(receipt)`
(ставит `LastSale(receipt.receipt_number, receipt.total, datetime.now())`).

**Чистый резолвер** (ядро логики, тестируется без времени и веба):

```python
def resolve_display_state(cart: CartView, last_sale: LastSale | None,
                          now: datetime, thanks_seconds: int) -> DisplayState:
    if cart.lines:
        return DisplayState(kind="sale", cart=cart)
    if last_sale is not None and (now - last_sale.at).total_seconds() < thanks_seconds:
        return DisplayState(kind="thanks", sale=last_sale)
    return DisplayState(kind="idle")
```

- **Приоритет — «продажа»**: как только в чеке есть строки, показываем чек (даже если недавно
  была продажа — начался новый покупатель). Поэтому чистить маркер при добавлении товара не
  нужно — он естественно перекрывается непустым чеком, а по пустому — гаснет по таймауту.
- Благодарность **самосбрасывается** по `thanks_seconds` → «ожидание». Между покупателями,
  когда таймаут вышел, — экран ожидания (приветствие).
- `now` **передаётся аргументом** → ветки таймаута тестируются подстановкой времени, без
  подмены системных часов. Роут вызывает с `datetime.now()`.

## 5. Роуты (`app/routes/customer.py`)

Тонкий роутер по образцу кассы (свой `Jinja2Templates`, тот же фильтр `money`
из `format_money`).

| Метод | Путь | Назначение | Ответ |
|---|---|---|---|
| `GET` | `/customer` | Полная страница экрана покупателя | `customer/index.html` |
| `GET` | `/customer/state` | Текущее состояние (для опроса) | фрагмент `customer/_display.html` |

- `/customer/state`: `state = resolve_display_state(get_cart().view(), get_last_sale(), datetime.now(), settings.customer_display_thanks_seconds)` → рендер фрагмента.
- Регистрация: `app.include_router(customer_router)` в `app/main.py`.
- Имена роутов/полей — английские; текст на экране — русский.

## 6. Интерфейс (HTMX + Jinja2)

- `templates/customer/index.html` — **отдельная крупная вёрстка** (не наследует узкий
  `base.html` с `max-width:720px`; собственный полноэкранный layout, крупный шрифт, высокий
  контраст — 20.3). Подключает htmx (тот же CDN, что и `base.html`). Содержит контейнер опроса:

  ```html
  <div id="display"
       hx-get="/customer/state"
       hx-trigger="load, every {{ poll_ms }}ms"
       hx-swap="innerHTML">
      {% include "customer/_display.html" %}
  </div>
  ```

  `poll_ms` — из `settings.customer_display_poll_ms`.

- `templates/customer/_display.html` — три ветки по `state.kind`:
  - **`sale`**: крупный список строк (название · `{кол-во} {ед} × {цена}` · сумма строки) и
    очень крупный **«Итого: {{ cart.grand_total | money }} ₽»**. Коротко, без служебных деталей
    (без способа оплаты, без строки округления — итог уже с округлением; 20.3).
  - **`thanks`**: крупно **«Спасибо за покупку»** (текст из конфига, точно по 20.4); опционально
    мелким — сумма чека.
  - **`idle`**: экран ожидания — приветствие из конфига (напр. «Добро пожаловать»).

- Деньги — через фильтр `money` (`format_money`), формат «148.50», единый с кассой и печатью.
  Количество — как в кассе (`| string` / `format_quantity`).
- Минимум JS: только HTMX (CLAUDE.md), без JS-фреймворков.

## 7. Конфигурация (`app/config.py`)

Экрана настроек нет (CLAUDE.md) — параметры в `config.py` (env-переопределяемы, префикс `STORE_`):

| Параметр | Тип | По умолчанию | Назначение |
|---|---|---|---|
| `customer_display_window` | bool | `False` | Открывать ли второе окно pywebview в `run.py`. |
| `customer_display_poll_ms` | int | `1000` | Интервал HTMX-опроса состояния. |
| `customer_display_thanks_seconds` | int | `8` | Сколько показывать «Спасибо за покупку». |
| `customer_display_welcome` | str | `"Добро пожаловать"` | Текст экрана ожидания. |
| `customer_display_thanks_text` | str | `"Спасибо за покупку"` | Текст после оплаты (20.4). |

## 8. Второе окно (`run.py`)

`run.py` открывает второе окно pywebview **только при** `settings.customer_display_window`
(по умолчанию выключено, чтобы на одном мониторе окно покупателя не перекрывало кассу в dev):

```python
webview.create_window(settings.app_name, f"http://{host}:{port}")            # касса (как есть)
if settings.customer_display_window:
    webview.create_window("Покупатель", f"http://{host}:{port}/customer",
                          fullscreen=True)                                    # покупатель
webview.start()
```

- pywebview поддерживает несколько окон — достаточно второго `create_window` до `start()`.
- В dev/браузере экран проверяется открытием `http://127.0.0.1:8000/customer` на втором мониторе,
  флаг не нужен.
- **Точная привязка к монитору** — отложена (разд. 2): раскладку задаёт ОС/продавец; авто-выбор
  экрана — с появлением железа, логики формирования это не трогает.

## 9. Критерии приёмки (pytest)

`tests/test_customer_display.py` (резолвер + маркер), HTTP-слой — там же или в расширении
`tests/test_sale.py`.

**Резолвер `resolve_display_state`**
1. Чек непуст → `kind="sale"`, отдаёт `cart` (строки и итог).
2. Чек пуст, продажа `at` = now → `kind="thanks"`, отдаёт `last_sale`.
3. Чек пуст, `now - at` = `thanks_seconds - ε` → всё ещё `thanks`; `= thanks_seconds`/больше → `idle`.
4. Чек пуст, маркера нет → `idle`.
5. Приоритет продажи: чек непуст **и** свежий маркер → `sale` (не `thanks`).

**Маркер**
6. `mark_sale_completed(receipt)` сохраняет `number`/`total`; `get_last_sale()` их возвращает.

**HTTP-слой**
7. `GET /customer` → 200, есть контейнер опроса (`hx-get="/customer/state"`, интервал из конфига).
8. `GET /customer/state` при непустом чеке → фрагмент со строками и итогом (`grand_total`).
9. После `POST /cashier/complete` → `GET /customer/state` показывает **«Спасибо за покупку»**
   (маркер выставлен, чек пуст).
10. Пустой чек без недавней продажи → экран ожидания (приветствие).

**Регрессия**
11. Все тесты 1.1/1.3/2.1/2.2 проходят без изменений — экран покупателя ничего не меняет в
    корзине/продаже; `mark_sale_completed` не влияет на транзакцию и `sale_result`.

## 10. Журнал решений (зачем так)

1. **Экран — пассивное зеркало одного синглтона.** Оба окна в одном процессе читают
   `get_cart()`; второму окну не нужен свой стор или IPC. Read-only → без гонок с кассой.
2. **HTMX-опрос, не SSE** (решение заказчика). Чистый HTMX без доп. JS-extension (CLAUDE.md:
   минимум JS). На localhost опрос ~1с дёшев; интервал — в конфиге.
3. **Маркер последней продажи — отдельное состояние.** Пост-продажный «успех» кассы эфемерен
   (живёт в одном HTMX-ответе), а экрану нужно показать благодарность после очистки корзины.
   Маркер ставится после `commit` (вне транзакции), по образцу best-effort печати 2.1.
4. **Благодарность самосбрасывается по таймауту.** «Продажа → Спасибо N сек → ожидание» —
   привычно для клиентских дисплеев и не требует событий/чистки маркера: непустой чек его
   перекрывает, пустой — гасит по времени.
5. **Резолвер — чистая функция с внешним `now`.** Ветки таймаута тестируются подстановкой
   времени, без подмены системных часов и без веба.
6. **Второе окно — по флагу, выключено по умолчанию.** На одном мониторе fullscreen-окно
   покупателя перекрыло бы кассу и мешало dev. Роут `/customer` доступен всегда для проверки в
   браузере; окно включают там, где есть второй экран. Привязка к монитору — отложена.
7. **«Крупно и коротко» (20.3).** Отдельная крупная вёрстка (не узкий `base.html`), на экране
   только товары/количество/цена/итог; способ оплаты, округление, накладная — не показываем.
