## 1. Назначение

До этого этапа касса — отдельная страница `/cashier`. Продавцу нужно работать **в одном окне
pywebview**, переключаясь между разделами, **не теряя начатую работу**. Этот этап строит
**раму** вкладок: касса становится постоянной стартовой вкладкой, рабочие разделы (Товары,
Заявки, Добавить, Чеки за день) открываются в отдельные вкладки.

Строим **только раму, не наполнение**: содержимое разделов приходит на своих этапах (Карточка —
3, Заявки — 5.3, Добавить/приём — 6, Чеки/отчёты — 7). Здесь разделы — **пустые панели-заглушки**
без складской и денежной логики.

Разделяем жёстко (roadmap): **сохранность данных = сервер, вкладки = удобство.** Незакоммиченный
черновик при закрытии вкладки теряется осознанно — автосохранение черновиков не делаем.

**Побочно закрыта дыра:** окно pywebview в `run.py` открывает корень `/`, а роута `/` не было
(в `main.py` только `/products`, `/cashier`, `/customer`, `/health`) — при запуске окно упиралось
в 404. Рама занимает `/`.

## 2. Граница этапа

### Входит
- `app/routes/shell.py` — новый роутер рамы:
  - `GET /` — страница-рама (`shell/index.html`): ряд вкладок + панель кассы, встроенная на
    сервере. Передаёт живую корзину (`get_cart().view()`) — тот же контекст, что `/cashier`.
  - `GET /panels/{key}` — фрагмент-заглушка раздела (`shell/_stub.html`). `key` валидируется по
    белому списку `SECTIONS`; неизвестный → 404. Чистая презентация, без бизнес-логики.
- `app/templates/shell/index.html` — рама: `{% extends "base.html" %}`, ряд вкладок сверху
  (постоянная «Касса» + кнопки-лаунчеры разделов), контейнер панелей, панель кассы
  `{% include "cashier/_screen.html" %}`, подключение `/static/shell.js`.
- `app/templates/shell/_stub.html` — универсальная заглушка «Раздел «…» — в разработке»;
  у «Добавить» (`test_field`) — тестовое текстовое поле для проверки сохранности состояния.
- `app/templates/cashier/_screen.html` — **извлечённый** контент кассы (стиль + `<h1>` + формы +
  `#cart`). Разметка/логика не меняются — вынос в partial для переиспользования рамой и standalone.
- `app/static/shell.js` — ванильный JS оболочки (~110 строк): реестр открытых вкладок,
  `openSection` / `activate` / `closeSection`. Только show/hide и одноразовая HTMX-загрузка
  панели. Денежной/складской логики нет.
- `app/main.py` — регистрация `shell_router`, монтаж `StaticFiles` на `/static`.
- `app/templates/cashier/index.html` — сведён к `extends base.html` + `{% include %}`
  (standalone `/cashier` продолжает работать для отладки в браузере).

### НЕ входит (отложено на свои этапы)
- **Наполнение разделов.** Товары/Карточка — этап 3, Заявки — 5.3, Добавить (приём/создание) — 6,
  Чеки за день / отчёты — 7. Здесь — только кнопки и пустые панели.
- **Карточка товара как вкладка с подменой товара внутри** (roadmap) — этап 3. Здесь заглушка.
- **Возврат как модалка поверх** (roadmap) — этап 4. Здесь не реализуется.
- **Экран покупателя** — отдельное окно pywebview (2.3, разд. 20), оболочки вкладок не касается.
  Не трогаем.
- **Единый визуальный слой** (типографика, кнопки, таблицы) — отдельный следующий шаг. Здесь
  каркас функциональный, минимальная разметка, дизайн не вылизываем.
- **Автосохранение черновиков** — не делаем (см. разд. 1). Незакоммиченный ввод при закрытии
  вкладки теряется осознанно.
- **Изменение бэкенда** — `services/`, `models/`, `hardware/`, транзакция продажи, печать,
  логика кассы не трогаются.

## 3. Архитектура: одно окно, панели в DOM, show/hide

```
                       GET /  (рама)
                          │
        ┌─────────────────┴──────────────────────────┐
   ряд вкладок (#tabs)                     кнопки-лаунчеры (#launcher)
   [ Касса ][ Товары ✕ ] …                 [Товары][Заявки][Добавить][Чеки]
        │                                            │ клик → openSection(key)
        │ клик по вкладке → activate(key)            │
        ▼                                            ▼
   контейнер панелей (#panels): все открытые панели ЖИВУТ в DOM
   ┌──────────────┬───────────────┐
   │ #panel-cashier│ #panel-<key>  │  ← .active показывает одну (display:block),
   │ (встроена на  │ (HTMX-загрузка│     остальные display:none
   │  сервере)     │  один раз)    │
   └──────────────┴───────────────┘
```

**Ключевое решение — сохранность = show/hide, а не перезагрузка.** Требование «ушёл на другую
вкладку и вернулся — введённое на месте» выполнимо только если панели **не выгружаются**: JS
переключает видимость классом `.active` (CSS `display`). Если бы контент грузился заново на
каждый клик — DOM пересоздавался бы и введённое терялось.

- **Касса** — постоянная панель, её шаблон **включается в раму на сервере** (`{% include
  "cashier/_screen.html" %}`) при загрузке `/`. Никогда не выгружается, крестика нет. Это и есть
  «встроить шаблон кассы в раму»; роуты и логика кассы (`/cashier/*`) не тронуты.
- **Разделы** — вкладка создаётся при первом клике по лаунчеру; содержимое подгружается по HTMX
  **один раз** (`htmx.ajax('GET', '/panels/<key>', {target: panel})`), затем панель остаётся в DOM.
- **Одна вкладка на раздел:** повторный клик по лаунчеру — просто `activate` уже открытой (реестр
  `open[key]`), новую не плодит, содержимое не перезагружает.
- **Крестик** удаляет вкладку и панель из DOM; если закрыта активная — активируется касса. Повторное
  открытие грузит заново (черновик теряется осознанно).

**Оба окна — один процесс.** Экран покупателя (2.3) — отдельное окно pywebview, читает тот же
синглтон корзины; рамы вкладок не касается.

## 4. Разделы рамы (`SECTIONS`, `app/routes/shell.py`)

Белый список: `key` (англ., для id/URL) → заголовок вкладки (рус.):

| key | Заголовок | Наполнение (этап) |
|---|---|---|
| `products` | Товары | 3 |
| `orders` | Заявки | 5.3 |
| `add` | Добавить | 6 |
| `receipts` | Чеки за день | 7 |

- `GET /panels/{key}`: `title = SECTIONS.get(key)`; если `None` → 404. Иначе рендер `_stub.html`
  с `{key, title, test_field: key == "add"}`.
- Флаг `test_field` — только у «Добавить»: тестовое поле `#add-test-field` для проверки сохранности
  (уйдёт при наполнении раздела на этапе 6).
- Ремарка roadmap: вход «Добавить» через общий с «Товарами» движок поиска (разд. 4 и 12.3 — режимы
  одного поиска) — содержимое этапа 6, здесь не дублируется.

## 5. Роуты (`app/routes/shell.py`)

Тонкий роутер по образцу кассы/покупателя (свой `Jinja2Templates`, тот же фильтр `money` из
`format_money` — рама встраивает `cashier/_cart.html`, который его использует).

| Метод | Путь | Назначение | Ответ |
|---|---|---|---|
| `GET` | `/` | Страница-рама (вкладки + касса) | `shell/index.html` |
| `GET` | `/panels/{key}` | Фрагмент-заглушка раздела | `shell/_stub.html` (404 при неизвестном key) |

- `GET /` передаёт `{sections, cart, error, sale_result}` — `cart = get_cart().view()`, `error`/
  `sale_result = None`, как у standalone-роута `/cashier` (встроенный `_cart.html` читает `cart`).
- Регистрация: `app.include_router(shell_router)` в `app/main.py` (первым — раздаёт `/`).
- Имена роутов/ключей/полей — английские; заголовки вкладок и тексты — русские.

## 6. Интерфейс (HTMX + Jinja2 + ванильный JS)

- `templates/shell/index.html` наследует `base.html`. Три зоны:
  - `#tabs` — ряд открытых вкладок. Статически — постоянная «Касса» (`data-key="cashier"`, без
    крестика). Вкладки разделов добавляет JS (с кнопкой-крестиком).
  - `#launcher` — кнопки разделов из `sections`, с `data-key/-title/-url` для JS.
  - `#panels` — контейнер панелей. Статически — `#panel-cashier` (`.active`) со встроенным
    `cashier/_screen.html`. Панели разделов добавляет JS.
- `templates/shell/_stub.html` — минимальная заглушка: заголовок + «в разработке»; при `test_field`
  — `<input id="add-test-field">`.
- `static/shell.js` — ванильный (CLAUDE.md: минимум JS, без фреймворков). HTMX — для загрузки
  контента панели. Никакой денежной/складской логики.
- Деньги в встроенной кассе — через фильтр `money` (единый с кассой/печатью).

### Изоляция панелей
- Касса таргетит `#cart` / `#search-results` — эти id есть только в панели кассы. Заглушки — свои
  непересекающиеся id (`#panel-<key>`, `#add-test-field`). HTMX-обновление одной вкладки (напр.
  пробитие товара обновляет `#cart`) не задевает другие панели.

## 7. JS оболочки (`app/static/shell.js`)

IIFE, `"use strict"`. Реестр `open = { cashier: true }`.

- `activate(key)` — снять `.active` со всех вкладок и панелей, поставить на выбранную (show/hide).
- `openSection(key, title, url)` — если `open[key]` → `activate`; иначе создать вкладку
  (текст + крестик) и панель `#panel-<key>`, один раз `htmx.ajax('GET', url, {target, swap:
  'innerHTML'})`, затем `activate`. Крестик: `stopPropagation` + `closeSection`.
- `closeSection(key)` — удалить вкладку и панель; если была активной → `activate('cashier')`.
- Постоянная вкладка кассы и кнопки-лаунчеры — обработчики `click` при инициализации.

Скрипт подключён в конце `content`, после загрузки htmx в `<head>` и после элементов `#tabs`/
`#launcher`/`#panels` — на момент выполнения DOM и `window.htmx` готовы.

- `app.mount("/static", StaticFiles(directory=app/static), name="static")` — раздача `shell.js`.
  Каталог `app/static/` был заведён структурой проекта под это.
- `app.include_router(shell_router)` — до остальных роутеров (даёт `/`).
- `run.py` **не меняется** — уже открывает `/`, теперь там рама.

## 9. Критерии приёмки

**Серверный слой (автотест, TestClient)**
1. `GET /` → 200; в теле `#tabs`, «Касса», `#launcher`, `panel-cashier`, `/static/shell.js`,
   встроенный `#cart`, заголовки разделов (Товары … Чеки за день).
2. `GET /panels/add` → 200, содержит `add-test-field`.
3. `GET /panels/products` → 200, **без** `add-test-field`.
4. `GET /panels/bogus` (не из белого списка) → 404.
5. `GET /static/shell.js` → 200.
6. `GET /cashier` (standalone) → 200 — не сломан выносом в partial.
7. Регрессия: весь существующий набор тестов зелёный (бэкенд не тронут). **Факт: 152 passed.**

**Интерактивный слой (проверка запуском `uvicorn`/окна)**
8. Стартовая вкладка «Касса» с рабочим экраном кассы; кнопки разделов сверху.
9. Клик «Добавить» → открывается вкладка с панелью-заглушкой и тестовым полем.
10. **Сохранность:** ввести текст в поле → переключиться на «Касса» → вернуться на «Добавить» →
    текст на месте (панель не выгружалась).
11. **Одна вкладка на раздел:** повторный клик «Добавить» активирует уже открытую, новую не плодит.
12. Крестик закрывает вкладку; «Касса» закрыть нельзя (крестика нет).
13. **Изоляция:** пробитие товара в чек обновляет только `#cart`, панели заглушек не мигают.

## 10. Журнал решений (зачем так)

1. **Разворот лёгкий.** `base.html` уже есть, экран кассы уже блок (`extends base.html`), а не
   самостоятельная страница — встраивание в раму сводится к выносу контента в partial и include.
2. **Сохранность = show/hide, панели в DOM.** Единственный способ не терять ввод при переключении.
   HTMX грузит контент раздела один раз при открытии, дальше — чистое переключение видимости.
   Черновик теряется только при явном закрытии вкладки (осознанно; сервер — источник сохранности).
3. **Касса встроена на сервере (`include`), а не по HTMX.** Постоянная вкладка без перезагрузки;
   логика роутов кассы не тронута — «встроить шаблон в раму» буквально.
4. **Вкладки сверху.** Касса широкая; вертикальный сайдбар съел бы ширину. Верхний ряд привычен
   для POS. (Предложено в плане, одобрено.)
5. **Ванильный JS, не Alpine.** Ради show/hide и реестра вкладок лишняя CDN-зависимость не нужна.
   Бизнес-логики в JS нет (CLAUDE.md).
6. **Белый список разделов + 404.** `/panels/{key}` не открыт для произвольных ключей; заглушки —
   чистая презентация, изолированные id.
7. **Рама занимает `/`.** Заодно закрыт 404 корня, куда указывает окно pywebview.
8. **Экран покупателя не трогаем.** Отдельное окно/зритель ≠ вкладки одного продавца (roadmap).
