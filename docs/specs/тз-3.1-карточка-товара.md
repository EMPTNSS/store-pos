# ТЗ — Полная карточка товара

> Этап **3.1** roadmap (полная карточка: поиск для карточки, просмотр детальной информации,
> редактирование).
> Макет: разд. 4 (поиск товаров для карточки), 5 (карточка товара: 5.2 идентификация,
> 5.3 информация, 5.5 редактирование). Разд. 5.4 (статистика) и 5.7 (окно движений) — этап 3.2;
> разд. 5.6 (графики) — этап 3.3.
> Опирается на 0.2 (`docs/specs/тз-0.2-база-товаров.md`): таблицы `product`, `movement`,
> `price_history`, разметка хранимого/вычисляемого (п.7), правило архивации (п.9).
> Опирается на 2.5 (`docs/specs/тз-2.5-навигационная-оболочка.md`): рама вкладок, «Карточка —
> одна вкладка с подменой товара внутри», раздел «Товары» был заглушкой — здесь наполняется.
> Переиспользует: `search_products` (`app/services/product_search.py`), `resolve_suppliers` /
> `list_active_suppliers` (`app/services/supplier_service.py`), фильтр `money`
> (`format_money`, `app/services/money.py`), схемные типы `Kopecks` / `PositiveDecimal`
> (`app/schemas/product.py`), фрагмент `products/_supplier_row.html` и роут
> `/products/supplier-row`.
> Источник истины по требованиям — `docs/макет-системы.txt`. При противоречии — сверяться с макетом.

---

## 1. Назначение

До этого этапа товар можно только **создать** (0.3) и **найти для чека** (1.2). Места, где видно
**всю** информацию о товаре и где её **правят**, ещё нет: раздел «Товары» в оболочке вкладок (2.5) —
пустая заглушка.

Этот этап наполняет его сквозным путём: **поиск товара для карточки** (разд. 4) → **открытие
карточки** во вкладку с подменой товара внутри (roadmap 2.5) → **просмотр** детальной информации
(разд. 5.2, 5.3) → **редактирование** (разд. 5.5), включая точечную **корректировку количества**
через движение товара.

Карточка — это **представление** данных из БД (0.2 п.1), а не отдельное хранилище: все поля берутся
из `product`, а вычисляемые (наличие, наценка) считаются на лету.

---

## 2. Зафиксированные решения

Три развилки закрыты на этапе проектирования (сверх макета):

1. **Количество правится уже в 3.1** (не откладывается в 5.1). Правка количества — это **складской
   путь**: пишет `movement` типа «инвентаризация» атомарно и обновляет `quantity_current`. Этап 5.1
   остаётся за **полноценной** инвентаризацией (массовая сверка/лист); карточка даёт точечную
   корректировку одного товара.
2. **Обе цены редактируемы** — и закупка, и продажа. Изменение любой из цен добавляет **одну** точку
   `price_history` с новой парой (0.2 п.11). Совпадает с показом обеих цен в 5.3.
3. **Название редактируемо.** Артикул, числовой код, QR — **read-only**: коды якорят историю, чеки
   и движения (0.2 п.9), их менять нельзя; привязка QR — забота печати этикеток (этап 7.4).

## 3. Границы этого ТЗ

### Входит
- **Поиск для карточки (разд. 4)** в панели «Товары» рамы. Переиспользует `search_products` — тот же
  движок, что на кассе (roadmap 2.5: не дублировать поиск).
- **Карточка — просмотр (5.2, 5.3):** идентификация + все хранимые поля + вычисляемые (наличие,
  наценка в ₽ и %). Формулы — из 0.2 п.7.
- **Редактирование паспорта (5.5):** название, обе цены, единица, мин. остаток, статус, поставщики,
  доп. информация.
- **Корректировка количества (5.5):** отдельное действие складского пути — движение «инвентаризация»
  + обновление остатка, одной транзакцией.
- **Новое поле `Product.extra_info`** (доп. информация, 5.3/5.5).
- **Карточка как вкладка с подменой** (roadmap 2.5): один таб «Карточка», открытие другого товара
  заменяет содержимое, вкладку не плодит.

### Не входит (соседние этапы)
- **Статистика товара (5.4)** — продано за всё время, приход/убытие по датам, чистая прибыль — этап 3.2.
- **Окно истории движений (5.7)** — этап 3.2. Данные уже копятся в `movement`; окно строится там.
- **Графики динамики цен (5.6)** — этап 3.3. `price_history` уже наполняется (создание + правка цены
  здесь); график рисуется там.
- **Полноценная инвентаризация (массовая сверка, разд. 13)** — этап 5.1.
- **Отслеживание/уведомление по мин. остатку (разд. 10)** — этап 5.2. Здесь поле только правится и
  показывается.
- **Проброс закончившегося товара в заявку по поставщику (разд. 11.9)** — этап 5.3.
- **Правка артикула, числового кода, QR** — коды якорят историю (см. п.2.3).
- **Приём/изменение закупки партией (разд. 12, этап 6.1).**
- **Финальный список статусов и сортировка поиска (этап 8).** Здесь статус — рабочий минимум
  `активный`/`архивный` (0.2 п.8), поиск сортируется как есть.

## 4. Модель: новое поле `extra_info`

Доп. информация о товаре (макет 5.3, 5.5) сейчас в модели отсутствует. Добавляется в
`app/models/product.py`:

```python
extra_info: Optional[str] = Field(default=None)
```

Свободный текст, nullable. Существующие товары — `None` до первой правки в карточке. Форму создания
(0.3) не трогаем: поле необязательное, при желании добавляется туда отдельным шагом.

**Миграция.** `SQLModel.metadata.create_all` (`app/database.py`) не выполняет `ALTER TABLE`.
Механизма миграций в проекте пока нет — dev-БД `data/store.db` (в `.gitignore`) пересоздаётся, как и
при прежних добавлениях полей. Зафиксировать в отчёте как технический долг: **при первой боевой БД
нужен Alembic** (добавление колонки без потери данных).

Прочие таблицы (`movement`, `price_history`, `product_supplier_link`) уже готовы с 0.2/этапа
поставщиков — структурно не меняются.

## 5. Схемы валидации (`app/schemas/product.py`)

Валидация на границе (CLAUDE.md правило 3): вход проверяется схемой до попадания в логику.

### 5.1 `ProductEdit` — паспорт товара

| Поле | Тип | Правило |
|---|---|---|
| `name` | str | не пустое (`name_not_empty`) |
| `price_buy` | `Kopecks` | ≥ 0 (`price_buy_non_negative`) |
| `price_sell` | `Kopecks` | > 0 (`price_sell_positive`) |
| `unit` | `UnitEnum` | из перечня |
| `min_stock` | `PositiveDecimal` | ≥ 0 (`min_stock_non_negative`) |
| `status` | `ProductStatus` | `активный` / `архивный` |
| `supplier_names` | `list[str]` | обрезка/дедуп (`clean_supplier_names`) |
| `extra_info` | `Optional[str]` | свободный текст |

Переиспользует существующие типы `Kopecks` / `PositiveDecimal` и валидаторы из `ProductCreate`.
**Нет** поля `quantity` и **нет** кодов (`article`, `numeric_code`, `qr_code`) — они не редактируются
паспортной формой.

### 5.2 `QuantityAdjust` — корректировка склада

| Поле | Тип | Правило |
|---|---|---|
| `quantity` | `PositiveDecimal` | ≥ 0 |

`0` допустим (обнуление остатка). Отрицательное новое количество запрещено — правила отрицательных
остатков отложены (этап 8, макет разд. 17).

## 6. Сервис (`app/services/product_service.py`)

Дописывается к существующему `create_product`.

### 6.1 `get_product(session, product_id) -> Optional[Product]`
Карточка по id. `None` → роут отдаёт 404.

### 6.2 `product_view(session, product) -> dict`
Вычисляемые значения для показа (0.2 п.7), нигде не хранятся:

| Ключ | Формула |
|---|---|
| `in_stock` | `quantity_current > 0` (наличие; «нет в наличии» — не статус, а остаток, 0.2 п.8) |
| `margin_abs` | `price_sell − price_buy` (разница в деньгах, копейки) |
| `margin_pct` | `(price_sell − price_buy) / price_buy × 100`; при `price_buy == 0` → `None` («—») |
| `suppliers` | поставщики товара через `ProductSupplierLink` |

### 6.3 `update_product(session, product_id, data: ProductEdit) -> Product`
Правка паспорта. **Одна транзакция:**
1. обновить `name, price_buy, price_sell, unit, min_stock, status, extra_info`;
2. если `price_buy` **или** `price_sell` изменились против текущих — добавить **одну**
   `PriceHistory(product_id, datetime=now, price_buy, price_sell)` с **новой** парой (0.2 п.11);
   не изменились — точку не писать;
3. заменить связи поставщиков: удалить `ProductSupplierLink` товара, заново вставить
   `resolve_suppliers(data.supplier_names, session)` (реюз `supplier_service`; дедуп/инлайн-создание
   уже внутри);
4. `commit`.

Не трогаются: `article`, `numeric_code`, `qr_code`, `created_at`, `quantity_current`.

### 6.4 `adjust_quantity(session, product_id, new_quantity: Decimal) -> Product`
Складской путь (CLAUDE.md правила 2, 5). **Одна транзакция:**
1. `delta = new_quantity − product.quantity_current`;
2. `delta == 0` → ничего не писать (движение только на реальное изменение остатка);
3. иначе: `Movement(product_id, datetime=now, quantity=delta, operation_type=inventory)`
   (знак: недостача `−`, излишек `+`) + `product.quantity_current = new_quantity`;
4. `commit`. Либо всё, либо ничего — полусписанных состояний нет.

**Почему количество — отдельным действием, а не внутри `update_product`.** Правка количества — это
складской путь: движение, атомарность, высокая цена ошибки. Паспортные поля — простое обновление
строки. Разделив их, мы (а) не пишем `movement` инвентаризации при каждой правке поставщика или
описания, (б) держим денежно-складскую транзакцию изолированной и тестируемой отдельно от веб-обвязки
(принцип CLAUDE.md: `services/` отделён от `routes/`).

## 7. Роуты (`app/routes/products.py`)

Дописываются к существующему роутеру. У него уже свой `Jinja2Templates`; добавить фильтр `money`
(как в кассе/оболочке — единый формат денег).

| Метод | Путь | Назначение | Ответ |
|---|---|---|---|
| `GET` | `/products/search?q=` | Поиск товара для карточки (панель «Товары») | `products/_search_results.html` |
| `GET` | `/products/{id}/card` | Полная карточка (просмотр + инлайн-формы) | `products/_card.html` (404 если нет) |
| `POST` | `/products/{id}` | Сохранить паспорт | `products/_card.html` (422 при ошибке, перерисовка) |
| `POST` | `/products/{id}/quantity` | Корректировка количества (инвентаризация) | `products/_card.html` (422 при ошибке) |

- **Поиск:** `search_products(session, q)`; фрагмент показывает по 4.4 — есть ли товар в системе,
  наличие (остаток > 0), цену, доп. информацию/статус; в каждой строке кнопка «Открыть карточку».
- **`POST`-роуты:** валидация схемой до логики; `ValidationError` → **422** + перерисовка карточки с
  ошибками (паттерн из `create_product_route`: собрать `err["msg"]`, вернуть тот же фрагмент со
  статусом).
- **Поставщики в форме правки** переиспользуют роут `/products/supplier-row` и фрагмент
  `products/_supplier_row.html` (уже существуют для формы создания).
- **`/panels/products`** (сейчас заглушка в `app/routes/shell.py`) переводится на реальную панель:
  отдаёт `products/_panel.html`. Ключ `products` уходит из «заглушечного» пути `_stub.html`.
  Остальные ключи (`orders`, `add`, `receipts`) остаются заглушками до своих этапов.

## 8. Интерфейс (HTMX + Jinja2, минимум JS)

### 8.1 Шаблоны
- **`products/_panel.html`** — наполнение вкладки «Товары»: поле поиска
  (`hx-get="/products/search"`, `hx-trigger="input changed delay:200ms"`, цель —
  `#product-search-results`) + пустой контейнер результатов. По образцу поиска кассы
  (`cashier/_screen.html` + `_search_results.html`).
- **`products/_search_results.html`** — таблица результатов; в строке кнопка «Открыть карточку» с
  `data-product-id="{{ p.id }}"` (JS перехватывает клик → `openCard(id)`).
- **`products/_card.html`** — карточка целиком; рендерится и при открытии, и при сохранении (HTMX-swap
  самой себя, `hx-swap="outerHTML"`):
  - **Идентификация (5.2), read-only:** артикул, числовой код, QR, дата создания (название правится в
    форме ниже).
  - **Информация (5.3):** наличие, цена продажи, цена закупки, наценка ₽ и %, количество + единица,
    мин. остаток, статус, поставщики, доп. информация. Деньги — через фильтр `money`.
  - **Форма паспорта (5.5):** `hx-post="/products/{id}"`, цель — сама карточка. Поля: название, обе
    цены, единица (`select` из `UnitEnum`), мин. остаток, статус (`select` `активный`/`архивный`),
    поставщики (ряды `_supplier_row.html` + «+ добавить поставщика»), доп. информация (`textarea`).
  - **Форма количества (5.5, склад):** отдельная, визуально отделённая форма
    `hx-post="/products/{id}/quantity"`, подпись «Корректировка остатка (инвентаризация)».
  - **Место под движения (3.2) и график (3.3)** — не делаем; максимум заглушечная подпись «появится
    на этапе 3.2/3.3».

### 8.2 JS оболочки (`app/static/shell.js`) — `openCard`
Добавить к существующему тонкому слою вкладок (2.5). Бизнес-логики в JS нет — только show/hide и
HTMX-загрузка (CLAUDE.md).
- `openCard(productId)`:
  - вкладки `card` нет → создать вкладку «Карточка» (с крестиком) и панель `#panel-card`,
    `htmx.ajax('GET', '/products/'+id+'/card', {target:'#panel-card', swap:'innerHTML'})`,
    затем `activate('card')`;
  - вкладка `card` есть → заново загрузить карточку **нового** товара в ту же панель (**подмена
    содержимого**), `activate('card')`. Одна вкладка на все карточки (roadmap 2.5), не по вкладке на
    товар.
- Делегированный обработчик клика по `[data-product-id]` внутри результатов поиска.

### 8.3 `app/templates/shell/index.html`
Панель `#panel-products` перестаёт быть stub-заглушкой: её содержимое грузится с `/panels/products`
(реальная панель поиска). Механику вкладок (2.5) не переписываем — только источник содержимого
раздела «Товары».

### 8.4 Изоляция панелей (наследие 2.5)
Карточка таргетит свои id (`#panel-card`, `#product-search-results`), непересекающиеся с кассой
(`#cart`) и заглушками. HTMX-обновление карточки не задевает другие вкладки.

## 9. Критерии приёмки

**Сервисный слой (автотест, складской/ценовой путь — приоритет):**
1. `adjust_quantity`: излишек (`+delta`) и недостача (`−delta`) — движение с верным знаком и типом
   `inventory`, `quantity_current` обновлён.
2. `adjust_quantity`: `delta == 0` → движения не создаётся, остаток не меняется.
3. `QuantityAdjust`: отрицательное количество отклонено (422 на роуте).
4. `update_product`: изменение цены → ровно одна новая `PriceHistory` с новой парой; цены не менялись
   → новых точек нет.
5. `update_product`: замена поставщиков (добавление/удаление/дедуп через `resolve_suppliers`);
   архивация статусом (товар физически не удалён).
6. `update_product`: `article`, `numeric_code`, `qr_code`, `created_at`, `quantity_current` не тронуты.
7. `product_view`: наценка ₽ и %, `price_buy == 0` → без деления; `in_stock` по остатку.

**Роуты (TestClient):**
8. `GET /products/{id}/card` → 200, содержит поля карточки; несуществующий id → 404.
9. `GET /products/search?q=` → фрагмент с «Открыть карточку».
10. `POST /products/{id}` валидное → 200 с обновлённой карточкой; невалидное (пустое имя, цена
    продажи ≤ 0) → 422 + ошибки.
11. `POST /products/{id}/quantity` валидное → 200 + движение в БД; отрицательное → 422.
12. `GET /panels/products` → панель поиска (не текст «в разработке»).
13. **Регрессия:** весь существующий набор тестов зелёный.

**Интерактивный слой (запуск `uvicorn`/окна):**
14. Вкладка «Товары» → поиск → «Открыть карточку» → карточка во вкладке со всей информацией.
15. Открыть карточку другого товара из поиска — содержимое **подменилось**, вкладка одна.
16. Правка паспорта (цена/поставщик/статус/доп.инфо) → карточка перерисовалась, значения на месте.
17. Корректировка количества → остаток изменился, в `movement` строка «инвентаризация» с верным
    знаком.

## 10. Журнал решений (зачем так)

1. **Количество правится в 3.1, но отдельным действием.** Макет 5.5 требует правку количества в
   карточке; сделали, но через складской путь (движение «инвентаризация» + атомарность), не трогая
   `quantity_current` напрямую (0.2 п.5). Отделили от паспортной формы, чтобы движение писалось только
   на реальную корректировку остатка, а не на каждую правку описания.
2. **Обе цены редактируемы, точка `price_history` — только на изменение.** Ручная правка цены —
   документированный канал изменения цены продажи (0.2 п.11); закупку обычно меняет приёмка (6.1), но
   ручную правку тоже разрешили. Точка истории пишется лишь когда цена реально изменилась — иначе
   график (3.3) засорялся бы дублями.
3. **Название правим, коды — нет.** Переименование безопасно (имя не якорь). Коды (`numeric_code`,
   `qr_code`, `article`) якорят чеки, движения, историю — их правка порвала бы связи (0.2 п.9).
4. **Поиск и поставщики — переиспользование, не дублирование.** `search_products` — тот же движок, что
   на кассе (roadmap 2.5). Поставщики — существующие `resolve_suppliers` / `_supplier_row.html` /
   `/products/supplier-row`.
5. **Карточка — одна вкладка с подменой** (roadmap 2.5): не плодим вкладку на товар; JS остаётся
   тонким (show/hide + HTMX), бизнес-логики в нём нет.
6. **Статистика, движения и графики — не здесь.** 5.4/5.7 → 3.2, 5.6 → 3.3. Данные для них уже копятся
