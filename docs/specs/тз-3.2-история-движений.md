# ТЗ — История движений и статистика товара

> Этап **3.2** roadmap (история движений: дата, количество, пометка операции; статистика по товару).
> Макет: разд. **5.7** (окно истории движений товара) и **5.4** (история и статистика товара).
> Опирается на 0.2 (`docs/specs/тз-0.2-база-товаров.md`): таблицы `movement`, `price_history`,
> `receipt` / `receipt_line`; разметка хранимого/вычисляемого.
> Опирается на 3.1 (`docs/specs/тз-3.1-карточка-товара.md`): карточка как вкладка с подменой,
> `_card_context` / `product_view`, перерисовка `#product-card` целиком (`hx-swap="outerHTML"`);
> в `_card.html` уже стоит заглушка под этот этап.
> Переиспользует: `line_total` (`app/services/money.py`) — «цена × количество → копейки,
> ROUND_HALF_UP»; фильтры `money` / `quantity` (уже подключены в `app/routes/products.py`);
> `get_product` (`app/services/product_service.py`).
> Источник истины по требованиям — `docs/Макет системы.txt`. При противоречии — сверяться с макетом.

## 1. Назначение

Карточка (3.1) показывает **текущее** состояние товара и правит его паспорт. Не хватает **истории**:
что и когда происходило с остатком и сколько товар заработал. Данные для этого уже копятся —
`movement` пишется на каждое изменение остатка (приход при создании, продажа, корректировка), а
`price_history` — на каждое изменение цены. Этот этап их **читает и показывает**, ничего в хранилище
не меняя.

Наполняется заглушка, оставленная 3.1 в `app/templates/products/_card.html` («История движений и
графики… появятся на этапах 3.2 и 3.3»): вместо неё — секция с **окном истории движений** (5.7) и
**статистикой товара** (5.4).

---

## 2. Зафиксированные решения

Две развилки закрыты на этапе проектирования (сверх макета):

1. **Себестоимость проданного — «as-of» по `price_history`, не текущая цена.** Чистая прибыль (5.4)
   считается по закупочной цене, **действовавшей на момент каждой продажи** (последняя точка
   `price_history` с `datetime ≤` момента чека), а не по текущей `Product.price_buy`. Правка закупочной
   цены задним числом не искажает прибыль прошлых продаж. Это возможно потому, что первая точка
   `price_history` пишется при создании товара — у любого товара есть цена «as-of» до первой продажи.
2. **Секция в карточке с ленивой подгрузкой.** История и статистика — отдельная секция, которая
   **догружается своим HTMX-запросом** (`hx-trigger="load"`), а не рендерится вместе с карточкой. Так
   открытие карточки быстрое, а после корректировки остатка (карточка перерисовывается `outerHTML`)
   секция сама перезагружает свежую историю.

## 3. Границы этого ТЗ

### Входит
- **Окно истории движений (5.7):** таблица всех движений товара — дата, пометка операции, количество
  со знаком и единицей. Порядок — от новых к старым.
- **Статистика товара (5.4):** продано за всё время; прибыло и убыло по датам; чистая прибыль за всё
  время.
- **Секция в карточке** на месте заглушки 3.1, с ленивой HTMX-подгрузкой (отдельный роут-фрагмент).

### Не входит (соседние этапы)
- **Графики динамики цен (5.6)** — этап 3.3. `price_history` уже наполняется (создание + правка цены в
  3.1); график рисуется там. Заглушку про графики в карточке **оставляем**.
- **Возвраты (разд. 14)** — этап 4.1. Тип операции `возврат` объявлен в `OperationType`, но пока
  нигде не пишется. История и статистика обрабатывают типы **обобщённо** — возврат отобразится без
  доработки, когда 4.1 начнёт его писать; уменьшение прибыли на сумму возврата — задача 4.1.
- **Массовый приход по накладной (разд. 12)** — этап 6.1. Приход сейчас пишется только при создании
  товара (0.3); этап 3.2 показывает его как есть.
- **Пагинация истории.** Для одной кассы объём мал — показываем все движения. Возможная пагинация —
  техдолг на случай роста объёма (см. разд. 9).
- **Изменение структуры `movement` / `receipt` / `price_history`** — только чтение, схема не трогается.

## 4. Модель — без изменений

Новых полей и таблиц нет. Всё берётся из существующих (0.2):

- `Movement(product_id, datetime, quantity, operation_type)` — `quantity` хранится знаковой дельтой
  остатка: приход/излишек `+`, продажа/недостача `−`. Источник **окна истории** и **количественной**
  статистики.
- `ReceiptLine(receipt_id, product_id, price_sell, quantity, total)` + `Receipt.datetime` — источник
  **выручки** по продажам (деньги на движении не хранятся).
- `PriceHistory(product_id, datetime, price_buy, price_sell)` — источник **себестоимости as-of**.

**Замечание по данным.** `Movement` не связан с чеком, поэтому денежную часть (прибыль) считаем не по
движениям продажи, а по строкам чеков (`ReceiptLine`), где есть и количество, и выручка, и — через
`Receipt.datetime` — момент для лукапа закупочной цены. Количественную часть (продано / приход /
убытие по датам) считаем по `Movement` как единому источнику изменений остатка.

## 5. Сервис (`app/services/product_service.py`)

Дописывается к существующим функциям карточки (`get_product`, `product_view`, `update_product`,
`adjust_quantity`).

### 5.1 `product_movements(session, product_id) -> list[Movement]`
Все движения товара, `order_by(Movement.datetime.desc(), Movement.id.desc())` — от новых к старым
(вторичная сортировка по `id` разводит движения одной секунды, напр. несколько строк одного чека).
Источник окна истории (5.7).

### 5.2 `buy_price_asof(session, product_id, at: datetime) -> int`
Последняя `PriceHistory` товара с `datetime <= at` (`order_by datetime desc, limit 1`); если точек нет
— fallback `product.price_buy`. Возвращает закупочную цену в копейках, действовавшую на момент `at`.

### 5.3 `product_stats(session, product_id) -> dict`
Статистика товара (5.4). Ключи:

| Ключ | Тип | Смысл / формула |
|---|---|---|
| `sold_total` | `Decimal` | Продано за всё время: Σ `|quantity|` по движениям типа `продажа`. |
| `by_date` | `list[dict]` | По датам: группировка движений по `date(datetime)`; на каждую дату `income` = Σ положительных дельт (прибыло), `outgoing` = Σ `−`отрицательных (убыло). Порядок — от новых дат к старым. |
| `net_profit` | `int` (копейки) | Чистая прибыль за всё время: по строкам чеков товара `revenue += line.total`; `cost += line_total(buy_price_asof(receipt.datetime), line.quantity)`; результат `revenue − cost`. |

- **`net_profit`** считается по `select` из `ReceiptLine` c `join` на `Receipt` (за `datetime`), фильтр
  `ReceiptLine.product_id == product_id`. Себестоимость строки — реюз `line_total` (то же правило
  округления, что в кассе). Прибыль считается **только по продажам**; недостача/излишек
  (инвентаризация) деньгами не оценивается. Нет продаж → `0`.
- **`by_date`** закрывает оба пункта макета «сколько прибыло по датам» и «сколько убыло по датам»
  одной таблицей: знак дельты определяет колонку. Приход и излишек инвентаризации попадают в
  `income`, продажа и недостача — в `outgoing`.

Разделение источников (количество ← `movement`, деньги ← `receipt_line` + `price_history`)
зафиксировано в разд. 4; тестируется в сервисном слое отдельно от веб-обвязки (CLAUDE.md: `services/`
отделён от `routes/`).

## 7. Интерфейс (HTMX + Jinja2, минимум JS)

### 7.1 `products/_movements.html` (новый фрагмент)
- **Статистика (5.4):**
  - «Продано за всё время»: `stats.sold_total | quantity` + единица товара.
  - «Чистая прибыль»: `stats.net_profit | money` ₽.
  - Таблица «Движение по датам»: строки `stats.by_date` — дата · прибыло (`income | quantity`) ·
    убыло (`outgoing | quantity`).
- **Окно истории движений (5.7):** таблица `movements` — колонки: дата (`m.datetime`, формат
  `дд.мм.гггг`), операция (`m.operation_type.value`: приход/продажа/возврат/инвентаризация),
  количество (`m.quantity | quantity` со знаком) + единица товара.
- **Пустое состояние:** движений нет → строка «Движений по товару пока нет».
- Деньги — фильтр `money`, количество — фильтр `quantity` (без хвостовых нулей).

### 7.2 `products/_card.html` — замена заглушки
Строки 129-130 (заметка `.todo-note` про 3.2/3.3) заменяются контейнером ленивой подгрузки:

```html
<div id="product-movements"
     hx-get="/products/{{ product.id }}/movements"
     hx-trigger="load" hx-swap="innerHTML">
    <p class="muted">Загрузка истории…</p>
</div>
```

Отдельная заметка про графики динамики цен (этап 3.3) — **остаётся**. Т.к. карточка перерисовывается
`outerHTML` при корректировке остатка, контейнер отрабатывает `load` заново и показывает свежую
историю после инвентаризации — без ручного обновления.

### 7.3 Изоляция (наследие 2.5/3.1)
Секция таргетит свой id `#product-movements` внутри `#panel-card`; на другие вкладки (`#cart`,
заглушки) не влияет. Бизнес-логики в JS нет — только декларативный HTMX-триггер.

## 8. Критерии приёмки

**Сервисный слой (автотест, приоритет):**
1. `product_movements`: возвращает движения товара по убыванию `datetime`; включает приход (от
   создания товара), продажу и инвентаризацию.
2. `product_stats.sold_total`: после продажи N единиц равен N (сумма по движениям типа `продажа`).
3. `product_stats.by_date`: верная разбивка `income` / `outgoing` по датам (приход и излишек → income,
   продажа и недостача → outgoing).
4. **`net_profit` через `price_history` as-of (ключевой тест решения):** продажа при закупке A →
   правка закупочной цены на B (3.1) → вторая продажа; себестоимость первой продажи считается по A,
   второй — по B; прибыль = Σ выручки − Σ себестоимости.
5. `net_profit`: нет продаж → `0`; `price_buy == 0` → себестоимость `0`, прибыль = выручке.
6. `buy_price_asof`: возвращает цену последней точки с `datetime ≤ at`; нет точек → `product.price_buy`.

**Роуты (TestClient):**
7. `GET /products/{id}/movements` → 200, фрагмент содержит статистику и строки истории; несуществующий
   id → 404.
8. Товар без движений → фрагмент с пустым состоянием («Движений по товару пока нет»).

**Регрессия:**
9. Весь существующий набор тестов зелёный; касса, продажа, печать, карточка (3.1) не тронуты.

**Интерактивный слой (запуск `uvicorn`/окна):**
10. Открыть карточку товара → секция истории догружается под паспортом: статистика + таблица движений.
11. Корректировка остатка в карточке → секция перезагрузилась, новая строка «инвентаризация» видна
    с верным знаком.

## 9. Журнал решений (зачем так)

1. **Себестоимость as-of, а не текущая цена.** Прибыль прошлых продаж не должна меняться при
   сегодняшней правке закупочной цены. `price_history` уже хранит закупку по датам (первая точка — при
   создании товара), поэтому точный лукап «на момент продажи» возможен без изменения хранилища.
2. **Деньги — из строк чеков, количество — из движений.** `Movement` не связан с чеком и не хранит
   денег; выручка живёт в `ReceiptLine`. Прибыль считаем там, где есть и количество, и деньги, и
   момент (`Receipt.datetime`). Количественную статистику и окно истории — по `Movement` как единому
   журналу изменений остатка (CLAUDE.md правило 5).
3. **Реюз `line_total` для себестоимости.** То же правило округления «цена × количество → копейки,
   ROUND_HALF_UP», что и на кассе, — прибыль сходится с денежным путём.
4. **Ленивая секция, а не рендер с карточкой.** История может расти; не грузим её при каждом открытии
   карточки. Перерисовка карточки `outerHTML` (3.1) заново триггерит `load` — история сама
   освежается после корректировки остатка.
5. **Возврат — обобщённо.** Тип операции уже в enum; история и статистика не хардкодят перечень
   операций, поэтому возврат (4.1) отобразится без правок этого этапа.
6. **Без пагинации в v1.** Одна касса — малый объём. Если история вырастет, добавить пагинацию/лимит
   в `product_movements` и фрагмент — техдолг, не блокер.
