# ТЗ — Графики динамики цен

> Этап **3.3** roadmap (графики динамики цены закупки и цены продажи; менее критично внутри этапа 3).
> Макет: разд. **5.6** (динамика цен) в контексте разд. **5.3** (карточка показывает динамику
> цены закупки и продажи).
> Опирается на 0.2 (`docs/specs/тз-0.2-база-товаров.md`): таблица `price_history`
> (`product_id, datetime, price_buy, price_sell` — цены в копейках `int`).

> создании товара и при правке цены (`update_product`, только если цена изменилась); карточка —
> вкладка с подменой, перерисовка `#product-card` целиком (`hx-swap="outerHTML"`).
> Опирается на 3.2 (`docs/specs/тз-3.2-история-движений.md`): паттерн ленивой секции карточки
> (`hx-trigger="load"` → отдельный роут-фрагмент); реюзит `buy_price_asof` как сосед по чтению
> `price_history`. Фильтр `money` (`app/services/money.py`) уже подключён в `app/routes/products.py`.
> Источник истины по требованиям — `docs/Макет системы.txt`. При противоречии — сверяться с макетом.

## 1. Назначение

Карточка (3.1) показывает **текущие** цены и наценку, история движений (3.2) — количественный
путь. Не хватает **временнóй динамики цен**: как менялись закупочная и розничная цена за время
жизни товара. Данные уже копятся — `price_history` пишет точку при создании товара (0.3) и при
каждой правке цены в карточке (3.1, только если цена реально изменилась). Этот этап их **читает
и рисует**: секция с графиком двух кривых на месте заглушки, оставленной 3.1/3.2 в `_card.html`.

Хранилище не трогается — только чтение существующих точек `price_history`.

---

## 2. Зафиксированные решения

Три развилки закрыты на этапе проектирования (сверх лаконичного макета 5.6):

1. **Серверный inline-SVG, без JS-библиотек.** График строится как `<svg>` прямо в Jinja-partial
   из точек `price_history`. Ноль нового JavaScript, ноль внешних CDN — в русле архитектуры
   (весь UI — HTMX + серверный HTML, единственная внешняя зависимость — htmx; CLAUDE.md запрещает
   тяжёлый фронт без явного решения). Геометрия (нормализация точек в координаты SVG) считается
   **чистой функцией в сервисном слое** и тестируется без веба; partial только отрисовывает готовые
   координаты.
2. **Ступенчатая кривая (step).** Между двумя соседними точками цена держится **горизонтально**,
   в момент следующей точки — вертикальный скачок. Это честно: между правками цена не менялась,
   линейная интерполяция рисовала бы несуществующий плавный тренд. Последний сегмент тянется до
   правого края (текущая цена действует «до сих пор»).
3. **Один график, две кривые.** Закупка и продажа — на общих осях, различаются цветом, с легендой.
   Видна и динамика каждой цены, и зазор между ними (наценка) на одном полотне. Ось X — реальное
   время (`datetime` точек), ось Y — цена в рублях, общий диапазон на обе кривые.

## 3. Границы этого ТЗ

### Входит
- **Секция «Динамика цен»** в карточке на месте заглушки 3.3, с ленивой HTMX-подгрузкой (отдельный
  роут-фрагмент, как секция движений в 3.2).
- **График двух ступенчатых кривых** (закупка, продажа) в серверном inline-SVG: оси, подписи
  крайних значений цены и дат, легенда, маркеры в точках изменения.
- **Чтение ряда** `price_history` товара и **чистый геом-хелпер** нормализации точек в координаты SVG.

### Не входит (соседние этапы / вне scope)
- **Изменение структуры `price_history`** — только чтение, схема не трогается.
- **Запись новых точек цены** — уже делают 0.3 (создание) и 3.1 (правка цены). Приход по накладной
  (этап 6.1) со временем добавит новые точки; график отрисует их без доработки.
- **Интерактивность графика** (зум, tooltip при наведении, выбор диапазона) — не требуется макетом;
  статичный SVG. Возможное улучшение — техдолг (см. разд. 9).
- **Экспорт графика в файл/печать** — вне scope (этап 7.x про экспорт/этикетки).
- **Разница в процентах / наценка отдельным графиком** — на карточке уже есть числовая наценка
  (3.1); зазор между кривыми виден визуально, отдельная кривая наценки не нужна.

## 4. Модель — без изменений

Новых полей и таблиц нет. Всё берётся из существующей (0.2):

- `PriceHistory(product_id, datetime, price_buy, price_sell)` — одна строка = одна «точка» обеих
  кривых на момент `datetime`; `price_buy` / `price_sell` — **копейки `int`**. Первая точка есть у
  любого товара (пишется при создании), новая — только когда цена реально изменилась (3.1).
  Единственный источник данных графика.

Единица измерения товара к ценам отношения не имеет (цена за единицу); ось Y — деньги в рублях.

## 5. Сервис (`app/services/product_service.py`)

Дописывается к существующим функциям карточки; соседствует с `buy_price_asof` (3.2), который уже
читает `price_history`.

### 5.1 `price_history(session, product_id) -> list[PriceHistory]`
Все точки цены товара, `order_by(PriceHistory.datetime.asc(), PriceHistory.id.asc())` — от старых
к новым (для графика нужен хронологический порядок; вторичная сортировка по `id` разводит точки
одной секунды). Источник ряда для графика.

### 5.2 `build_price_chart(points, *, width, height, pad) -> dict` — чистая функция (без session)
Нормализует список точек в геометрию SVG. Тестируется в изоляции от БД и веба (принимает готовый
список `PriceHistory` или лёгких кортежей `(datetime, price_buy, price_sell)`). Возвращает словарь:

| Ключ | Тип | Смысл |
|---|---|---|
| `has_data` | `bool` | Есть ли хоть одна точка. |
| `single` | `bool` | Ровно одна точка (цена ни разу не менялась) — рисуем плоскую линию во всю ширину. |
| `buy_line` | `str` | `points`-строка `polyline` для ступенчатой кривой закупки: `"x0,y0 x1,y0 x1,y1 …"` + дотяжка последнего сегмента до `width−pad`. |
| `sell_line` | `str` | То же для кривой продажи. |
| `buy_dots` / `sell_dots` | `list[(x, y)]` | Координаты маркеров в точках изменения. |
| `y_labels` | `list[(y, text)]` | Подписи оси Y: минимум и максимум цены (через фильтр `money`, в рублях). |
| `x_labels` | `list[(x, text)]` | Подписи оси X: дата первой и последней точки (`дд.мм.гггг`). |

Правила нормализации:
- **X** — линейно по реальному времени: `x = pad + (t − t_min)/(t_max − t_min) · (width − 2·pad)`.
  При `t_max == t_min` (все точки одномоментны → одна точка / один день) → `single`, кривая плоская
  на всю ширину.
- **Y** — линейно по цене (копейки): бóльшая цена — выше (инверсия оси SVG: `y` растёт вниз).
  Общий диапазон `[y_min_price, y_max_price]` **на обе кривые** (единый масштаб, чтобы зазор был
  правдив). При `y_max_price == y_min_price` (цена одна на всём ряду) — паддинг диапазона (±10 %
  или ±1 ₽), чтобы плоская линия шла по центру, а не делила на ноль.
- **Ступень**: между точками i и i+1 добавляется промежуточная вершина `(x[i+1], y[i])` — линия
  идёт горизонтально до момента изменения, затем вертикально к новому значению.

Деньги остаются в копейках `int` внутри геометрии; в рубли переводятся только для текстовых
подписей (реюз фильтра `money`). Никакого `float` в денежных значениях — только в координатах
пикселей SVG (это не деньги, округление некритично).

Дописывается к существующему роутеру карточки; фильтр `money` уже подключён.

| Метод | Путь | Назначение | Ответ |
|---|---|---|---|
| `GET` | `/products/{id}/prices` | Секция динамики цен (ленивая подгрузка) | `products/_prices.html` (404 если товара нет) |

## 7. Интерфейс (HTMX + Jinja2, серверный SVG, минимум JS)

### 7.1 `products/_prices.html` (новый фрагмент)
Самодостаточный partial: один корневой `<div class="price-chart">` + локальный `<style>` + `<svg>`
с фиксированным `viewBox` (напр. `0 0 640 260`), масштабируется в ширину контейнера (`width:100%`,
`height:auto`). Оформление в стиле `_movements.html` (заголовок `<h2>` с `border-top`, приглушённые
подписи `.muted`).

- **Заголовок:** `<h2>Динамика цен</h2>`.
- **Легенда:** два маркера цветом — «Закупка» и «Продажа». Цвета согласованы с палитрой карточки
  (закупка — сдержанный синий/серый, продажа — акцентный зелёный `#1a7f37`, уже используется в
  `_movements.html` для положительных значений; конкретные цвета — на реализации, но стабильные
  и различимые).
- **График:** две `<polyline fill="none" stroke=…>` из `chart.buy_line` / `chart.sell_line`
  (ступенчатые), `<circle>`-маркеры из `*_dots`, оси/сетка тонкими линиями, подписи `x_labels`
  (даты снизу) и `y_labels` (цены слева).
- **Состояния:**
  - `chart.has_data == false` → «Нет данных о ценах» (теоретически недостижимо: точка есть с
    создания товара, но защищаемся).
  - `chart.single == true` → плоская линия(и) во всю ширину + подпись «Цена не менялась».
- Цена в подписях — фильтр `money` + « ₽».

### 7.2 `products/_card.html` — замена заглушки
Строки 138-139 (заметка `.todo-note` про графики 3.3) заменяются контейнером ленивой подгрузки —
тем же паттерном, что секция движений 3.2:

```html
<!-- Графики динамики цен (5.6) — ленивая подгрузка секции (hx-trigger load). -->
<div id="product-prices"
     hx-get="/products/{{ product.id }}/prices"
     hx-trigger="load" hx-swap="innerHTML">
    <p class="muted">Загрузка графика…</p>
</div>
```

Т.к. карточка перерисовывается `outerHTML` при сохранении паспорта (в т.ч. изменение цены, 3.1),
контейнер отрабатывает `load` заново — после правки цены секция сама показывает свежую ступень без
ручного обновления.

### 7.3 Изоляция (наследие 2.5 / 3.1 / 3.2)
Секция таргетит свой id `#product-prices` внутри `#panel-card`; на `#product-movements`, `#cart` и
заглушки других вкладок не влияет. JS-логики нет — только декларативный HTMX-триггер и статичный SVG.

## 8. Критерии приёмки

**Сервисный слой (автотест, приоритет — геометрия тестируема без БД):**
1. `price_history`: возвращает точки товара по возрастанию `datetime`; после создания товара — одна
   точка, после правки цены (3.1) — две, в хронологическом порядке.
2. `build_price_chart` — **ступенька**: для двух точек `buy_line` содержит промежуточную вершину
   `(x1, y0)` (горизонталь до момента изменения, затем вертикаль); последний сегмент дотянут до
   правого края.
3. `build_price_chart` — **единый масштаб Y** на обе кривые: точка с максимальной ценой (среди buy и
   sell) ложится на верхнюю границу, с минимальной — на нижнюю.
4. `build_price_chart` — **одна точка** (`single`): плоская линия во всю ширину, без деления на ноль
   по X.
5. `build_price_chart` — **цена не менялась** (`y_max == y_min` на всём ряду): диапазон Y паддится,
   линия по центру, без деления на ноль по Y.
6. `build_price_chart` — пустой ряд: `has_data == false`, безопасно (пустые линии).

**Роуты (TestClient):**
7. `GET /products/{id}/prices` → 200, фрагмент содержит `<svg>`, легенду и подписи цен;
   несуществующий id → 404.
8. Товар с одной точкой → фрагмент с состоянием «Цена не менялась».

**Регрессия:**
9. Весь существующий набор тестов зелёный; касса, продажа, печать, карточка (3.1), история (3.2) не
   тронуты. Заглушка 3.3 в `_card.html` заменена, секция движений и её контейнер не задеты.

**Интерактивный слой (запуск `uvicorn` / окна):**
10. Открыть карточку товара → под секцией движений догружается график с двумя кривыми и легендой.
11. Изменить цену в карточке (3.1) → карточка перерисовалась → на графике появилась новая ступень
    (скачок в момент правки), без ручного обновления.

## 9. Журнал решений (зачем так)

1. **Серверный SVG, а не JS-библиотека.** Весь UI проекта — HTMX + серверный HTML, единственная
   внешняя зависимость — htmx (CLAUDE.md запрещает тяжёлый фронт без явного решения). График из
   нескольких десятков точек прекрасно строится статичным `<svg>` на сервере; тащить chart-библиотеку
   и JS-слой ради двух кривых — против архитектуры и лишняя зависимость.
2. **Ступенчатая кривая, а не линейная.** Цена меняется дискретными событиями (создание, правка) и
   держится постоянной между ними. Ступенька это отражает; линейная интерполяция рисовала бы плавный
   тренд, которого не было.
3. **Один график, две кривые, общий масштаб Y.** Динамика обеих цен и зазор между ними (наценка)
   читаются на одном полотне. Единый масштаб делает зазор правдивым; раздельные оси исказили бы
   визуальное сравнение.
4. **Геометрия — чистая функция в сервисе.** Нормализация точек в координаты SVG тестируется без БД и
   без веба (CLAUDE.md: `services/` отделён от `routes/`). Partial только отрисовывает готовые
   координаты — в шаблоне нет вычислений.
5. **Ленивая секция, а не рендер с карточкой.** Единый паттерн с историей движений (3.2): не грузим
   график при каждом открытии карточки. Перерисовка карточки `outerHTML` при правке цены заново
   триггерит `load` — график сам освежается.
6. **Ось X по реальному времени.** «Динамика» — про время: цена, продержавшаяся полгода, даёт длинную
   полку, недавняя правка — короткий хвост. Равномерная раскладка по индексу исказила бы темп изменений.
7. **Без интерактива в v1.** Макет требует «график / кривую», не зум/tooltip. Статичный SVG закрывает
   требование; интерактив — техдолг на случай реальной необходимости.
