# ТЗ — Возврат товара

> Этап **4.1** roadmap (возврат товара). Складской путь + новый документ «чек возврата».
> Макет: разд. 14 (возврат: назначение, кнопка, сценарий, документ возврата), 15.1 (отмена
> незавершённого — как аналог отмены незавершённого возврата).
> Опирается на 1.1/1.2 (`app/services/cart.py`, движок поиска `app/services/product_search.py`,
> сканер = ввод кода): паттерн корзины и добавления товара переиспользуется.
> Опирается на 1.3 (`docs/specs/тз-1.3-завершение-продажи.md`): `complete_sale`, счётчик
> номеров, атомарная транзакция, снимок строк — образец для возврата.
> Опирается на 0.2 (`тз-0.2-база-товаров.md`): `Product` (`quantity_current`, `price_sell`,
> `unit`), `Movement` (`operation_type`, `datetime`, `quantity`) — тип `return_ = "возврат"`
> в `OperationType` **уже есть**.
> Опирается на 2.5 (`тз-2.5-навигационная-оболочка.md`): возврат — **модалка поверх**, не вкладка.
> Источник истины — `docs/Макет системы.txt`. При противоречии — сверяться с макетом.

## 1. Назначение

Дать продавцу **отдельную функцию возврата** (разд. 14.1–14.2). По кнопке «Возврат» открывается
модалка; продавец набирает **корзину возврата** (сканер/поиск, несколько позиций), при
необходимости правит цену и количество, и проводит возврат. Система **одной транзакцией**:

1. сохраняет **чек возврата** (шапка + строки) со снимком названий/единиц и указанными суммами;
2. **возвращает остаток** каждого товара (`quantity_current += quantity`, разд. 14.3 п. 4);
3. пишет движение `возврат` на каждую позицию (правило 5 CLAUDE.md), **положительная дельта**;
4. очищает корзину возврата и показывает подтверждение (номер чека возврата, итог).

Либо всё целиком, либо ничего (правило 2 — атомарность). Чек возврата **сохраняется в системе**
(разд. 14.4) и, как завершённый чек, не редактируется (правило 4).

Результат: возврат реально возвращает товар на склад и остаётся отдельным документом в базе;
суммы зафиксированы на момент оформления.

## 2. Граница этапа

### Входит
- Модели `ReturnReceipt` (шапка) и `ReturnReceiptLine` (строки) — новые таблицы, отдельные от
  продажного `Receipt`.
- Отдельный последовательный **номер чека возврата** (`return_number`) через свой счётчик
  `ReturnNumberCounter` (по образцу `ReceiptNumberCounter` из 1.3). Своя нумерация, чтобы
  возвраты не смешивались с продажами в будущих отчётах (этап 7).
- **Корзина возврата** `ReturnCart` (`app/services/return_cart.py`) — синглтон в памяти, по
  образцу `Cart` (1.1), но с **редактируемой ценой** позиции.
- Набор позиций: добавление сканером/кодом и через поиск (переиспользуют `search_products` и
  паттерн `/cashier/items`), правка количества, **правка цены**, удаление строки, очистка.
- Сервис `complete_return(...)` — сохранение чека возврата + возврат остатка + движения
  `возврат` **в одной транзакции БД**.
- Движение `возврат` на каждую строку: `datetime`, `operation_type=return_`, **положительное**
  количество (+quantity, дельта остатка).
- Способ возврата денег как пометка: `наличные` / `безналичные` (переиспользуем `PaymentMethod`).
- Эндпоинты `/returns/*` + модалка (поиск, корзина возврата, правка, проведение, подтверждение).
- Кнопка «Возврат» в оболочке (2.5) + модалка поверх (тонкий JS show/hide, без бизнес-логики).
- Валидация на границе (Pydantic): количество > 0, цена ≥ 0, способ возврата из перечня.
- Инициализация счётчика возврата в `init_db()`.

### НЕ входит (отложено)
- **Привязка возврата к чеку-первоисточнику** и **исправление завершённого чека** (разд. 15.2–15.3)
  — этап **4.2**. В 4.1 возврат самостоятельный: цену/количество вводит продавец, чек продажи не
  ищется и не помечается.
- **Печать чека возврата** (ESC/POS). Как и продажа в 1.3, здесь чек возврата только
  **сохраняется** в системе (разд. 14.4). Реальная печать — отдельным шагом поверх сохранённого
  документа (слой печати уже изолирован, `app/hardware`), не в 4.1.
- **Экран покупателя** для возврата (разд. 20) — не трогаем.
- **Отчёт/список чеков возврата за день** (этап 7). Здесь чек возврата пишется в БД, но экрана
  списка нет; движение `возврат` уже видно в истории карточки (3.2).
- **Правила отрицательных остатков** (разд. 17) — отложены. Возврат остаток только увеличивает,
  так что здесь неактуально.
- Проверка «а этот товар вообще продавался/в каком количестве» — вне 4.1 (нет привязки к чеку).

## 3. Модель данных (новые таблицы, `app/models/return_receipt.py`)

Чек возврата — **постоянный самодостаточный документ**, отдельный от продажи (решение
заказчика). Строки — снимок на момент оформления (как `ReceiptLine`, правило 4).

### 3.1 `ReturnReceipt` (шапка чека возврата)

| Поле | Тип | Примечание |
|---|---|---|
| `id` | int PK | Технический ключ. |
| `return_number` | int, unique | Последовательный номер чека возврата (см. 3.3). Человекочитаемый. |
| `datetime` | datetime | Дата и время оформления возврата (разд. 14, движение — разд. 5.7). |
| `payment_method` | `PaymentMethod` | Способ возврата денег (пометка): `наличные`/`безналичные`. |
| `total` | int (копейки) | **Зафиксированный** итог возврата = Σ строк. Без округления вверх до ₽. |

`PaymentMethod` переиспользуется из `app/models/receipt.py` (не дублировать enum).

### 3.2 `ReturnReceiptLine` (строка чека возврата)

| Поле | Тип | Примечание |
|---|---|---|
| `id` | int PK | |
| `return_receipt_id` | int FK → `return_receipt.id` | |
| `product_id` | int FK → `product.id` | Ссылка на товар (для движения/статистики). |
| `name` | str | **Снимок** названия на момент возврата. |
| `unit` | `UnitEnum` | **Снимок** единицы измерения. |
| `price` | int (копейки) | Цена возврата за единицу — **указанная продавцом** (по умолчанию `price_sell`). |
| `quantity` | Decimal (`quantity_column()`) | Возвращаемое количество (штучное/весовое). |
| `total` | int (копейки) | **Зафиксированная** сумма строки (`line_total(price, quantity)`). |

`quantity` — через `quantity_column()` (`app/models/types.py`), как в `ReceiptLine`.

### 3.3 Счётчик номеров возврата

Отдельная таблица `ReturnNumberCounter` (id=1, `last_value`), по образцу `ReceiptNumberCounter`
(1.3). Номер берётся внутри той же транзакции с `.with_for_update()` и инкрементом — номера
последовательны и без гонок. Инициализация записи `id=1, last_value=0` — в `init_db()`, рядом с
`ReceiptNumberCounter`.

> **Почему отдельная нумерация возвратов:** возврат — иной документ, чем продажа. Своя
> последовательность держит журналы чистыми (в отчётах этапа 7 продажи и возвраты считаются
> раздельно, «выручка минус возвраты»), человекочитаема и переживает переезд на Postgres.

## 4. Корзина возврата (`app/services/return_cart.py`)

Синглтон в памяти по образцу `Cart` (1.1), но с **редактируемой ценой** позиции. Отдельный класс,
чтобы не менять протестированный продажный `Cart` (безопасность денежного пути).

Структура (зеркало `Cart`/`CartLine`/`CartView`):

- `ReturnCartLine`: `line_id`, `product_id`, `name`, `unit`, `price` (int, копейки — редактируемая),
  `quantity` (Decimal); `total` = `line_total(price, quantity)`.
- `ReturnCartView`: `lines`, `total` = Σ строк. **Без** `rounding`/`grand_total` — возврат не
  округляется вверх (магазин возвращает ровно указанную сумму, решение заказчика).
- `ReturnCart`:
  - `add(product, quantity=1)` — при повторном добавлении товара увеличивает количество строки
    (как `Cart.add`); `price` при добавлении снимается с `product.price_sell`.
  - `set_quantity(line_id, quantity)` — заменить количество (> 0, иначе `ValueError`).
  - `set_price(line_id, price)` — **новое**: заменить цену возврата строки (копейки, ≥ 0).
  - `remove(line_id)`, `clear()`, `view()`.
- `get_return_cart()` — единственный активный черновик возврата.

Черновик возврата — «разовое действие» (2.5): при отмене/закрытии модалки корзина возврата
очищается явно (`clear`), автосохранения нет. БД до проведения не трогается.

## 5. Сервис возврата (`app/services/return_service.py`)

`complete_return(session, cart: ReturnCart, payment_method: PaymentMethod) -> ReturnReceipt`.

Повторяет проверенный паттерн `complete_sale` (1.3), зеркально по знаку остатка/движения.

Шаги (всё внутри одной транзакции):

1. `view = cart.view()`; если `view.lines` пуст → `ValueError("Возврат пуст")` (пустой возврат не
   проводим; БД не трогается).
2. Взять и инкрементировать `ReturnNumberCounter` (`.with_for_update()`), получить `return_number`.
3. Создать `ReturnReceipt` (`return_number`, `datetime=now`, `payment_method`, `total=view.total`),
   `add`, `flush()` → `id`.
4. Для каждой `ReturnCartLine`:
   - создать `ReturnReceiptLine` (снимок `name`, `unit`, `price`, `quantity`, `total`);
   - `product = session.get(Product, line.product_id)`; **увеличить**
     `product.quantity_current += line.quantity` (разд. 14.3 п. 4 — товар возвращается в остаток);
   - создать `Movement(product_id, datetime=now, quantity=+line.quantity,
     operation_type=OperationType.return_)` — **положительная дельта** остатка.
5. `session.commit()`; `session.refresh(receipt)`.
6. **Вне транзакции**, только после успешного commit: `cart.clear()`.
7. Вернуть `receipt` (с `return_number`, `total`) для подтверждения в UI.

**Атомарность.** Любая ошибка до commit → откат: ни чек возврата, ни строки, ни возврат остатка,
ни движения не сохранятся; корзина возврата остаётся (правило 2). **Один общий `now`** на весь
чек (шапка, строки, движения).

## 6. Деньги — фиксация указанной суммы

- Строка возврата: `total = line_total(price, quantity)` (тот же хелпер `app/services/money.py`,
  ROUND_HALF_UP до копейки, результат — `int` копеек, правило 1).
- Итог возврата: `total = Σ строк` — **без** `round_total_up` (в отличие от продажи). Магазин
  возвращает ровно набранную сумму; переплата покупателю недопустима.
- `ReturnReceipt.total == Σ ReturnReceiptLine.total` — инвариант, проверяется тестом.
- Пересчёт при просмотре запрещён (правило 4): сохранённый чек возврата не «поедет» при правке
  карточки. Деньги — `int` копейки, количество — `Decimal`. Нового в `money.py` не требуется.

## 7. Эндпоинты (`app/routes/returns.py`)

Роутер тонкий: провалидировать вход → дёрнуть сервис/корзину → вернуть HTML-фрагмент. Логика —
в `return_cart.py`/`return_service.py`. Зеркалит `/cashier/*` (1.1–1.3).

| Метод | Путь | Назначение | Ответ |
|---|---|---|---|
| `GET` | `/returns/modal` | Тело модалки: поиск + пустая/текущая корзина возврата | `returns/_modal.html` |
| `GET` | `/returns/search` | Поиск товара (Query `q`) — тот же движок | `returns/_search_results.html` |
| `POST` | `/returns/items` | Добавить по коду/сканеру (Form `numeric_code`) | `returns/_return_cart.html` |
| `POST` | `/returns/items/{line_id}/quantity` | Изменить количество (Form `quantity`) | `returns/_return_cart.html` |
| `POST` | `/returns/items/{line_id}/price` | Изменить цену возврата (Form `price`) | `returns/_return_cart.html` |
| `POST` | `/returns/items/{line_id}/delete` | Убрать строку | `returns/_return_cart.html` |
| `POST` | `/returns/clear` | Очистить корзину возврата (отмена/закрытие) | `returns/_return_cart.html` |
| `POST` | `/returns/complete` | Провести возврат (Form `payment_method`) | `returns/_return_cart.html` |

- Успех проведения → `_return_cart.html` с пустой корзиной и баннером `return_result` (номер чека
  возврата, итог, способ). HTMX `hx-target="#return-cart"`, `hx-swap="innerHTML"`.
- Пустой возврат / неизвестный способ → тот же фрагмент с `error`, корзина не меняется, чек не
  создаётся.
- Эндпоинты кассы (1.1–1.3) и продажный чек не затрагиваются.

## 8. Интерфейс (HTMX + Jinja2 + модалка)

**Кнопка «Возврат»** — в оболочке (`templates/shell/index.html`), рядом с лаунчером разделов, но
помечена как открывающая **модалку**, а не вкладку (2.5). По клику тонкий JS показывает оверлей и
подгружает `/returns/modal` в его тело (HTMX, один раз за открытие).

**Модалка** (`returns/_modal.html`): оверлей поверх, внутри —
- строка **поиска** (`hx-get="/returns/search"`, как на кассе) + поле **сканера/кода**
  (`hx-post="/returns/items"`), результаты поиска с кнопкой «В возврат» (`data`-атрибут кода);
- блок **`#return-cart`** — фрагмент `returns/_return_cart.html`;
- кнопки «Отмена»/крестик закрывают модалку и шлют `POST /returns/clear` (черновик — разовый).

**`returns/_return_cart.html`**: таблица строк (название, единица, **редактируемая цена**,
количество, сумма), у каждой — правка количества/цены (`hx-post`), удаление; итог возврата
(`total | money`); блок **«Способ возврата»** (радио `наличные`/`безналичные`, по умолчанию
`наличные`); кнопка **«Оформить возврат»** (`hx-post="/returns/complete"`, `hx-target="#return-cart"`).
Блок способа и кнопка — только при непустой корзине. После успеха — **баннер** `return_result`:
«Возврат оформлен. Чек возврата №{{ number }}, {{ payment_method }}, к возврату {{ total | money }} ₽»
над пустой корзиной.

Деньги — через фильтр `money` (регистрируется в роутере, как в `cashier.py`/`shell.py`). Минимум
JS: только HTMX + тонкий show/hide оверлея в `shell.js` (аналог `openCard`, но модалка). Бизнес-
логики в JS нет (CLAUDE.md, 2.5).

## 9. Валидация на границе (Pydantic, `app/schemas/return_.py`)

- `ReturnComplete`: `payment_method: PaymentMethod` — из перечня (`cash`/`card`); иначе отказ,
  сообщение во фрагменте, чек не создаётся (правило 3).
- `ReturnLinePrice`: `price` — парсится в копейки (переиспользовать парсер денег из схем товара,
  `app/schemas/product.py`), должна быть ≥ 0.
- Количество — переиспользовать `CartQuantity` (`app/schemas/cart.py`): число > 0.
- «Возврат не пуст» — проверяет сервис (`complete_return` кидает `ValueError`), т.к. это состояние
  корзины, а не поля формы. Роут ловит `ValidationError` и `ValueError` и отдаёт сообщение.

## 10. Критерии приёмки (pytest, `tests/test_return.py`)

**Сервис `complete_return`**
1. Создаёт `ReturnReceipt` с `return_number`, `datetime`, `payment_method` и `total = view.total`.
2. Инвариант: `receipt.total == Σ ReturnReceiptLine.total`; округления вверх до ₽ **нет**
   (напр. итог с копейками сохраняется как есть, а не поднимается до рубля).
3. Создаёт `ReturnReceiptLine` на каждую строку со снимком `name`/`unit`/`price`/`quantity`/`total`.
4. **Возврат остатка**: `product.quantity_current` **увеличивается** на `quantity` каждой строки
   (в т.ч. дробный вес — увеличение на `Decimal`).
5. **Движение**: на каждую строку пишется `Movement(operation_type=return_)` с **положительным**
   `quantity` (= +количество) и общим `datetime` чека.
6. Номера возвратов **последовательны**: два возврата подряд → `return_number` растёт на 1;
   нумерация **независима** от продажных `receipt_number` (свой счётчик).
7. **Атомарность**: при ошибке в середине (подменить `session.commit`) — ни `ReturnReceipt`, ни
   строки, ни `Movement` не сохранены, `quantity_current` не изменён.
8. **Пустой возврат** → `ValueError`; чек/движения не создаются, склад не тронут.
9. **Указанная цена**: строка с ценой, отличной от текущей `price_sell` (продавец правил), пишет
   именно указанную цену; последующее изменение `product.price_sell` не меняет сохранённый чек.
10. Корзина возврата **очищена** после успеха; при неудаче (пустой возврат) — не очищена.

**Корзина возврата `ReturnCart`**
11. `add` дважды один товар → одна строка, количество суммируется; `price` = снимок `price_sell`.
12. `set_price` меняет цену строки и `total` пересчитывается; `set_quantity(<=0)` → `ValueError`.

**HTTP-слой (`/returns/*`)**
13. `POST /returns/items` валидным кодом → строка в корзине возврата (фрагмент `_return_cart`).
14. Валидный `payment_method` при непустой корзине → 200, баннер `return_result`, корзина пуста;
    в БД появился `ReturnReceipt` + строки; `quantity_current` увеличился; есть движение `возврат`.
15. Пустой возврат → сообщение об ошибке, `ReturnReceipt` не создан.
16. Неизвестный `payment_method` → сообщение об ошибке, чек не создан, корзина цела.
17. `POST /returns/clear` очищает корзину (отмена/закрытие модалки).

## 11. Журнал решений (зачем так)

1. **Отдельные таблицы возврата.** Продажа и возврат — разные документы. Отдельные
   `ReturnReceipt`/`ReturnReceiptLine` не трогают продажный `Receipt` и его инвариант
   `total == round_total_up(subtotal)` (протестирован в 1.3). Чисто и без миграции продаж.
2. **Своя нумерация возвратов.** Отдельный счётчик держит журналы раздельными; в отчётах дня
   (этап 7) «продажи − возвраты» считаются по разным последовательностям.
3. **Корзина возврата — зеркало кассовой, но с ценой.** Продавец «сканирует товары» (решение
   заказчика) — тот же паттерн, что на кассе (1.2), плюс правка цены и суммы («та, что написана»).
   Отдельный `ReturnCart` не рискует протестированным `Cart`.
4. **Движение — положительная дельта.** Возврат пишет `+quantity` (продажа — `−quantity`). Сумма
   движений = остаток; знак кодирует направление, `by_date` в статистике (3.2) уже это учитывает.
5. **Итог без округления вверх.** Магазин возвращает ровно указанную сумму; поднимать до рубля
   (как в продаже) нельзя — это переплата покупателю. Поэтому у возврата нет `rounding`.
6. **Самостоятельный возврат, без привязки к чеку.** Привязка к завершённому чеку и исправление
   (разд. 15.2–15.3) — этап 4.2. В 4.1 продавец сам задаёт товар/количество/цену.
7. **Возврат — модалка, не вкладка** (2.5). Разовое действие, не откладывают наполовину;
   черновик при закрытии осознанно теряется (очистка корзины).
8. **Печать не здесь.** 4.1 сохраняет электронный чек возврата (разд. 14.4), как 1.3 сохранял
   продажу без печати. Реальная печать — поверх, в изолированном слое `app/hardware`.
