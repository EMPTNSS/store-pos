# ТЗ — Исправление завершённого чека через возврат

> Этап **4.2** roadmap. Завершённый чек напрямую не правится — исправление ошибочной
> позиции выполняется через **возврат, привязанный к чеку-первоисточнику**.
> Макет: разд. 15.2 (после завершения чек не исправляется, только через возврат),
> 15.3 (сценарий исправления), 14 (возврат: остаток, чек возврата).
> Опирается на **4.1** (`docs/specs/тз-4.1-возврат-товара.md`): `ReturnReceipt`/
> `ReturnReceiptLine`/`ReturnNumberCounter`, `ReturnCart` (`app/services/return_cart.py`),
> `complete_return` (`app/services/return_service.py`), модалка возврата и `/returns/*` —
> **всё переиспользуется и расширяется**, свободный возврат 4.1 не ломается.
> Опирается на 1.3: `Receipt`/`ReceiptLine` (`app/models/receipt.py`) — снимок продажи,
> источник данных для исправления (не мутируется, правило 4).
> Источник истины — `docs/Макет системы.txt`. При противоречии — сверяться с макетом.

---

## Контекст (зачем этот этап)

В 4.1 появился **самостоятельный** возврат: продавец вручную набирает товар, цену и
количество, чек продажи при этом не ищется и никак не связывается. Но макет (разд. 15.2–15.3)
требует отдельный путь: **исправление ошибки в уже завершённом чеке**. Завершённый чек
редактировать нельзя (правило 4 CLAUDE.md, разд. 15.2) — вместо правки продавец находит чек
продажи, указывает ошибочную позицию, и система оформляет **возврат, привязанный к этому чеку**.
Это даёт две вещи, которых нет в 4.1:
1. **Достоверность** — цена и количество берутся из реальной продажи (снимок `ReceiptLine`),
   а не вводятся заново; исправление возвращает ровно уплаченное.
2. **Защиту склада/денег** — нельзя вернуть больше, чем было продано по этой строке чека
   (учитывая уже оформленные ранее возвраты). Это критичный складской путь.

Результат: у продавца есть корректный способ исправить завершённый чек, сам чек остаётся
неизменным, а возврат навсегда связан с первоисточником.

### Зафиксированные решения (согласованы до реализации)
1. **Завершённый чек не правится** (правило 4, разд. 15.2). 4.2 не добавляет никакого
   редактирования продажного `Receipt`/`ReceiptLine`; строки чека показываются **только для
   чтения**, ошибочная позиция переносится в корзину возврата.
2. **Цена корректирующей строки зафиксирована по чеку** (снимок `ReceiptLine.price_sell`),
   **не редактируется** (в отличие от свободной строки 4.1 с редактируемой ценой).
3. **Возврат по строкам, без округления** — даже при полном возврате всех позиций чека
   надбавка округления итога (`Receipt.rounding`) покупателю не возвращается (как переплата
   ≤ 1 ₽ при продаже; единообразно с 4.1, у возврата нет `round_total_up`).

## 1. Назначение

Дать продавцу путь **исправления завершённого чека через возврат** (разд. 15.2–15.3). Продавец
находит завершённый чек по номеру, видит его строки (read-only) с пометкой «сколько ещё можно
вернуть», выбирает ошибочную позицию и количество — она попадает в **корзину возврата** с ценой
из чека. Дальше — проверенный путь 4.1: **одной транзакцией** сохраняется чек возврата, товар
возвращается в остаток, пишется движение `возврат`. Новое сверх 4.1:

1. **Привязка** чека возврата к чеку-первоисточнику (шапка и строки).
2. **Защита от перевозврата**: по каждой строке чека Σ возвращённого ≤ проданного.

Сам завершённый чек не изменяется (правило 4). Чек возврата, как и в 4.1, — самодостаточный
документ; связь с продажей хранится ссылками, а не пересчётом.

## 2. Граница этапа

### Входит
- **Поиск завершённого чека по номеру** (`receipt_number`) — минимальный lookup (полного списка
  чеков пока нет, это этап 7).
- **Показ строк чека read-only** + по каждой строке: продано, **уже возвращено**, **доступно к
  возврату** = продано − уже возвращено.
- **Перенос выбранной строки чека в корзину возврата** с ценой из чека (снимок
  `ReceiptLine.price_sell`, зафиксирована), привязкой к строке чека и количеством ≤ доступного.
- **Привязка возврата к первоисточнику**: `ReturnReceipt.source_receipt_id`,
  `ReturnReceiptLine.source_line_id` — оба **nullable** (свободный возврат 4.1 остаётся с `NULL`).
- **Защита от перевозврата** в сервисе `complete_return`: авторитетная проверка внутри
  транзакции — Σ возвращённого по каждой строке чека ≤ проданного.
- **Цена корректирующей строки не редактируется** (зафиксирована по чеку); `set_price` для такой
  строки и `POST .../price` — отклоняются.
- **Один корректирующий возврат привязан ровно к одному чеку**: в одном черновике возврата не
  смешиваются свободные (4.1) и корректирующие строки, и не смешиваются строки из разных чеков
  (`source_receipt_id` — одно значение на чек возврата).
- UI: в **существующей модалке возврата** — блок «Возврат по чеку» (поиск по номеру, read-only
  строки, кнопки «в возврат»). Свободный возврат 4.1 в той же модалке остаётся.
- Валидация на границе (Pydantic): номер чека — целое; количество > 0; строка-первоисточник
  существует.

### НЕ входит (отложено / запрещено)
- **Редактирование завершённого чека** — запрещено концептуально (правило 4, разд. 15.2), не
  делаем ни сейчас, ни потом.
- **Печать чека возврата** (ESC/POS) — как и в 4.1: только сохранение документа. Печать — поверх,
  в изолированном `app/hardware`.
- **Список чеков / отчёт возвратов за день** — этап 7. Здесь только точечный поиск по номеру.
- **Экран покупателя** для возврата (разд. 20) — не трогаем.
- **Правила отрицательных остатков** (разд. 17) — неактуальны: возврат остаток только
  увеличивает; перевозврат блокируется отдельным инвариантом (не остатком).

## 3. Модель данных (расширение существующих таблиц, `app/models/return_receipt.py`)

Аддитивно, обе колонки **nullable** — свободный возврат 4.1 продолжает работать с `NULL`.
Продажный `Receipt`/`ReceiptLine` **не меняется** (правило 4).

### 3.1 `ReturnReceipt` — новое поле
| Поле | Тип | Примечание |
|---|---|---|
| `source_receipt_id` | `Optional[int]`, FK → `receipt.id`, nullable | Чек-первоисточник. `NULL` для свободного возврата (4.1). |

### 3.2 `ReturnReceiptLine` — новое поле
| Поле | Тип | Примечание |
|---|---|---|
| `source_line_id` | `Optional[int]`, FK → `receipt_line.id`, nullable | Строка чека-первоисточника. По ней считается «уже возвращено» и проверяется перевозврат. `NULL` для свободной строки. |

> **Долг миграции (прежний).** Схема поднимается через `SQLModel.metadata.create_all` без
> Alembic; `create_all` не добавит колонки в уже существующую таблицу. Для dev-БД — пересоздание
> файла (как отмечено в отчётах 4.1). Колонки nullable и аддитивны, логику 4.1 не задевают.

## 4. Доступное к возврату (`app/services/return_service.py` или `return_cart.py`)

Хелпер `already_returned(session, source_line_id) -> Decimal` — сумма `ReturnReceiptLine.quantity`
по строкам с данным `source_line_id` (учитывает все ранее оформленные возвраты).

Хелпер/представление для lookup: по `Receipt` вернуть список строк с полями `line`, `sold`
(= `ReceiptLine.quantity`), `returned` (= `already_returned`), `available` = `sold − returned`.
Строки с `available <= 0` показываются, но недоступны к добавлению.

## 5. Корзина возврата — расширение (`app/services/return_cart.py`)

Зеркалит подход 4.1, минимально расширяя протестированную корзину.

- `ReturnCartLine` — новые поля: `source_line_id: Optional[int] = None`,
  `price_locked: bool = False` (цена зафиксирована по чеку, не редактируется).
- `ReturnCart` — новый атрибут `source_receipt_id: Optional[int] = None` (чек-первоисточник
  текущего корректирующего черновика; `None` — свободный/пустой).
- `add_from_receipt_line(receipt_line: ReceiptLine, quantity: Decimal)` — **новое**:
  - создаёт строку со снимком `name`/`unit` из `receipt_line`, `price = receipt_line.price_sell`,
    `source_line_id = receipt_line.id`, `price_locked = True`;
  - повторное добавление той же строки-первоисточника — **мёрж по `source_line_id`** (не по
    `product_id`: один товар может стоять в чеке несколькими строками);
  - выставляет/сверяет `cart.source_receipt_id` (см. правила смешивания ниже).
- **Правила смешивания** (когерентность документа — один возврат = один чек):
  - `add_from_receipt_line` при наличии в корзине свободных строк (`source_line_id is None`) или
    строк другого чека → `ValueError` («Очистите корзину: смешивать нельзя»).
  - Свободный `add(product)` (4.1, скан/поиск) при `source_receipt_id is not None` → `ValueError`.
- `set_price(line_id, ...)` для строки с `price_locked=True` → `ValueError` («Цена по чеку не
  редактируется»). Для свободных строк 4.1 — как было.
- `set_quantity` — как в 4.1 (> 0); верхнюю границу (≤ доступного) проверяет UI и **авторитетно**
  сервис при проведении.
- `clear()` сбрасывает и `source_receipt_id = None`.

## 6. Сервис возврата — расширение (`app/services/return_service.py`)

Сигнатура `complete_return(session, cart, payment_method)` **не меняется** — `source_receipt_id`
берётся из корзины. Внутри той же одной транзакции добавляется:

1. В шапку `ReturnReceipt` пишется `source_receipt_id = cart.source_receipt_id`.
2. В каждую `ReturnReceiptLine` пишется `source_line_id = line.source_line_id`.
3. **Инвариант перевозврата** (для строк с `source_line_id`): для каждой строки
   `already_returned(session, source_line_id) + line.quantity <= ReceiptLine.quantity`.
   Иначе → `ValueError("Нельзя вернуть больше, чем продано")`, транзакция откатывается
   (ни чек возврата, ни строки, ни возврат остатка, ни движения не сохранятся).
   Проверка — внутри транзакции (не только в UI): защита от гонок и устаревшего черновика.

Остальное — как в 4.1 без изменений: единый `now`, номер из `ReturnNumberCounter`
(`.with_for_update()`), остаток `+= quantity`, движение `OperationType.return_` с положительной
дельтой, `total = Σ строк` **без** `round_total_up`, очистка корзины только после успешного commit.

## 7. Деньги — по строкам чека, без округления

- Строка возврата: `total = line_total(price, quantity)` — цена = снимок `ReceiptLine.price_sell`
  (тот же хелпер `app/services/money.py`, правило 1).
- Итог возврата = Σ строк, **без** `round_total_up`. При полном возврате чека надбавка
  `Receipt.rounding` не возвращается (переплата ≤ 1 ₽ в пользу магазина, как при продаже).
- Пересчёт при просмотре запрещён (правило 4): изменение текущей цены товара или продажного чека
  не «двигает» сохранённый чек возврата. Нового в `money.py` не требуется.

## 8. Эндпоинты (`app/routes/returns.py`, дополнение к 4.1)

| Метод | Путь | Назначение | Ответ |
|---|---|---|---|
| `GET` | `/returns/receipt` | Найти завершённый чек по номеру (Query `number`), показать строки read-only с «доступно к возврату» | `returns/_receipt_lookup.html` |
| `POST` | `/returns/from-receipt` | Перенести строку чека в возврат (Form `source_line_id`, `quantity`) | `returns/_return_cart.html` |

- `GET /returns/receipt`: чек не найден / неверный номер → фрагмент с сообщением, корзина цела.
- `POST /returns/from-receipt`: валидация (строка существует, `0 < quantity ≤ available`, правила
  смешивания) → `add_from_receipt_line` → `_return_cart.html`; при нарушении — сообщение, корзина
  не меняется.
- Существующие `/returns/*` (4.1) переиспользуются; `POST /returns/items/{id}/price` теперь
  отклоняет строки с `price_locked` (сообщение «Цена по чеку не редактируется»).
- Успешное проведение → баннер `return_result` дополняется ссылкой на чек продажи:
  «…к чеку продажи №{source_number}» (если возврат корректирующий).

## 9. Интерфейс (модалка возврата, дополнение)

`returns/_modal.html` получает **блок «Возврат по чеку»** (над/рядом со свободным вводом 4.1):
- поле **номера чека** (`hx-get="/returns/receipt"`, `hx-target="#receipt-lookup"`);
- `#receipt-lookup` → фрагмент `returns/_receipt_lookup.html`: шапка чека (№, дата, итог) и
  таблица строк **только для чтения** (товар, цена по чеку, продано, уже возвращено, доступно),
  у доступных строк — поле «вернуть» (кол-во ≤ доступного) и кнопка «В возврат»
  (`hx-post="/returns/from-receipt"`, `hx-target="#return-cart"`).

`returns/_return_cart.html`: для строк с `price_locked` цена показывается **текстом** (без формы
правки), количество редактируемо как прежде; свободные строки 4.1 — с редактируемой ценой.
Баннер `return_result` при корректирующем возврате упоминает № чека продажи.

Минимум JS: только HTMX. Бизнес-логики в шаблонах нет (CLAUDE.md, 2.5). Модалка та же (2.5).

## 11. Критерии приёмки (pytest, `tests/test_return_correction.py`)

**Доступное к возврату / lookup**
1. `already_returned` = Σ количеств по `source_line_id`; для непокрытой строки = 0.
2. Lookup по номеру возвращает строки чека с `available = sold − returned`; несуществующий
   номер → пусто/сообщение.

**Корзина (корректирующие строки)**
3. `add_from_receipt_line` берёт `price` из `ReceiptLine.price_sell`, ставит `source_line_id`,
   `price_locked=True`, снимок `name`/`unit`; выставляет `cart.source_receipt_id`.
4. Повторное добавление той же строки чека — мёрж (одна строка, количество суммируется).
5. `set_price` по locked-строке → `ValueError`; изменение `product.price_sell` после не влияет
   на цену в корзине и в сохранённом чеке.
6. Смешивание запрещено: свободный `add` при корректирующем черновике → `ValueError`;
   `add_from_receipt_line` из другого чека / поверх свободных строк → `ValueError`.

**Сервис `complete_return` (корректирующий)**
7. Пишет `ReturnReceipt.source_receipt_id` и `ReturnReceiptLine.source_line_id`; остаток
   `+= quantity`; движение `возврат` с `+quantity`; `total = Σ строк`.
8. **Перевозврат заблокирован**: попытка вернуть `quantity > available` → `ValueError`, ничего
   не сохранено (ни чек, ни строки, ни движение), остаток не изменён.
9. **Частичный возврат, затем остаток**: вернуть часть строки, потом остаток — оба проходят;
   третий возврат сверх проданного → `ValueError`.
10. **Полный возврат чека**: итог = Σ строк по ценам чека, **без** надбавки округления (чек
    с `rounding>0` → возврат = `subtotal`, а не `total`).

**Регрессия 4.1**
11. Свободный возврат (скан/поиск, редактируемая цена) работает как раньше;
    `source_receipt_id`/`source_line_id` = `NULL`; итог без округления; движение `возврат`.

**HTTP (`/returns/*`)**
12. `GET /returns/receipt?number=…` найденного чека → строки с «доступно»; ненайденного →
    сообщение.
13. `POST /returns/from-receipt` валидной строкой → строка в корзине (locked-цена в
    `_return_cart`); `quantity > available` → сообщение, корзина цела.
14. `POST /returns/complete` корректирующего возврата → `ReturnReceipt` с `source_receipt_id` в
    БД, строки с `source_line_id`, остаток вырос, движение `возврат`; баннер с № чека продажи.
15. `POST /returns/items/{id}/price` по locked-строке → сообщение, цена не изменена.

## 12. Журнал решений (зачем так)

1. **Чек не правится — только возврат** (правило 4, разд. 15.2). 4.2 не добавляет
   редактирования `Receipt`; строки чека read-only, исправление живёт в отдельном документе
   возврата, связанном ссылкой.
2. **Привязка через nullable FK** (`source_receipt_id`/`source_line_id`). Аддитивно, свободный
   возврат 4.1 остаётся с `NULL` и не ломается; продажный чек не мигрируется.
3. **Защита от перевозврата — инвариант в транзакции.** Складской/денежный путь: нельзя вернуть
   больше проданного (с учётом прошлых возвратов). Проверка внутри `complete_return`, а не только
   в UI, — против гонок и устаревшего черновика.
4. **Цена зафиксирована по чеку.** Исправление = вернуть уплаченное; редактирование сломало бы
   сверку «продано ↔ возвращено» и смысл коррекции. Свободный возврат 4.1 (редактируемая цена)
   остаётся для случаев без чека.
5. **Без округления** (решение заказчика). Единообразно с 4.1; надбавка ≤ 1 ₽ при полном
   возврате не возвращается — как переплата в пользу магазина при продаже.
6. **Один возврат — один чек.** Запрет смешивать свободные/корректирующие строки и разные чеки
   держит `source_receipt_id` однозначным и документ когерентным.
