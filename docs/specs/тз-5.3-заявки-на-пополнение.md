# ТЗ — Заявки на пополнение

> Этап **5.3** roadmap (заявки: создание — авто-детект + вручную, редактирование, разделение
> по магазинам на будущее, закрытие и хранение закрытых; группировка по поставщику, ручной
> проброс кандидата с выбором поставщика).
> Макет: разд. 11 целиком (назначение, источник, хранение, магазины, содержание, редактирование,
> закрытие, хранение закрытых, поставщики 11.9).
> **Ключевой факт:** поставщики уже реализованы (`app/models/supplier.py` —
> `Supplier`, `ProductSupplierLink`; `app/services/supplier_service.py` —
> `supplier_key`, `list_active_suppliers`, `resolve_suppliers`; поле поставщика на форме
> создания 0.3 и в карточке 3.1). Детектор минимума `is_low_stock` реализован в 5.2.
> Опирается на 2.5 (рама вкладок, заглушка «Заявки»), 3.1 (движок поиска), 5.2 (`is_low_stock`).
> Источник истины — `docs/Макет системы.txt`. При противоречии — сверяться с макетом.

## 1. Назначение

Дать продавцу фиксировать товары, которые нужно заказать (макет разд. 11.1). Система
**автоматически обнаруживает кандидатов** (товар достиг минимума или закончился, разд. 11.2),
показывает их списком; продавец **вручную пробрасывает** нужные в заявку, выбирая поставщика
и корректируя количество (разд. 11.9.5). Заявки группируются **по поставщику** (разд. 11.9.4):
одна открытая заявка на поставщика. Заявку можно редактировать (разд. 11.6), закрыть при
заказе (разд. 11.7) и просмотреть после закрытия (разд. 11.8).

Заявка — это **намерение заказать**, а не движение склада. Создание, правка и закрытие заявки
**не меняют остаток** и **не пишут движение товара**. Возврат товара в остаток при приезде
заказа — это приём накладной (этап 6), отдельный путь.

## 2. Зафиксированные решения (развилки 5.3)

1. **«Авто» = авто-детект кандидатов + ручной проброс.** Roadmap пишет «создание авто при
   достижении мин. остатка + вручную», но макет 11.9.5 прямо запрещает автозаброс без выбора
   продавца («Автоматического заброса без выбора продавца не происходит»). Разрешаем
   противоречие в пользу макета: система **автоматически строит список кандидатов**
   (детект — `is_low_stock` из 5.2 + «закончился»), но в заявку товар попадает **только**
   по кнопке «+» с выбором поставщика и правкой количества. Автоматически создаваемых/
   наполняемых заявок нет.
2. **Разделение по магазинам (11.4) — поле `store` с дефолтом из config.** v1 — одна точка,
   UI выбора магазина нет. Поле `store` в шапке заявки заполняется единственным магазином
   `settings.store_name`. Схема готова к нескольким точкам (переезд как смена конфигурации,
   по духу CLAUDE.md), без отдельной сущности «Магазин» сейчас.
3. **Текущее кол-во и мин. остаток в заявке — живые из товара, не снимок.** Строка заявки
   хранит только `needed_quantity` и `comment`; «текущее количество» и «минимальный остаток»
   (разд. 11.5) подтягиваются из `Product` на момент показа. Компромисс: закрытая заявка
   показывает актуальные, а не исторические цифры остатка — принято осознанно (см. журнал
   решений п. 7). Персистентное намерение — это `needed_quantity`.
4. **Одна открытая заявка на поставщика.** «Заявка формируется по поставщику» (11.9.1):
   проброс кандидата к поставщику X дописывает строку в **открытую** заявку поставщика X
   (создаёт, если открытой нет). Повторный проброс того же товара в ту же заявку **обновляет**
   строку, а не плодит дубль.
5. **Заявка всегда с поставщиком.** При пробросе поставщик выбирается всегда (11.9.5 пп. 2–4):
   один — уходит к нему; несколько — модалка выбора; нет — продавец вводит/выбирает на месте
   (инлайн-создание через `resolve_suppliers`). Группа «Разный / без поставщика» (11.9.4) —
   это раздел **списка кандидатов** (товары без поставщика), а не заявка без поставщика.

## 3. Границы этапа

### Входит
- **Модель заявок:** `Order` (шапка) + `OrderLine` (строка). Регистрация в metadata.
- **Детектор кандидатов:** `is_order_candidate(product)` = `is_low_stock` (5.2) **или**
  «закончился» (`quantity_current <= 0`). Чистая функция, без session.
- **Панель «Заявки»** во вкладке рамы (замена заглушки): список кандидатов (сгруппирован,
  «+» на каждом), открытые заявки по поставщикам, архив закрытых.
- **Проброс кандидата** (11.9.5): «+» → определение поставщика (1 / несколько / нет) →
  правка нужного количества → добавление строки. Модалка выбора поставщика при неоднозначности.
- **Редактирование заявки** (11.6): добавить произвольный товар через поиск, задать/изменить
  нужное количество, добавить комментарий, удалить строку.
- **Закрытие заявки** (11.7): статус → «закрыта», `closed_at`; остаток и движения **не трогаются**.
- **Хранение закрытых** (11.8): закрытая заявка не удаляется, доступна в архиве только для чтения.
- **Поле `store`** из `settings.store_name` (11.4, на будущее).
- **Схемы валидации границы:** нужное количество > 0, имя поставщика непустое (реюз приёмов
  `schemas/product.py`).
- **Профильный тест** `tests/test_orders.py`.

### Не входит
- **Автоматическое создание/наполнение заявок без продавца** — отклонено (разд. 2 п.1, макет 11.9.5).
- **Отдельная сущность «Магазин», UI выбора магазина** — только поле `store` из config (11.4 «на будущее»).
- **Снимок остатка/минимума в строке заявки** — живые значения (разд. 2 п.3).
- **Приём заказанного товара, возврат в остаток, движение «приход» по заявке** — этап 6
  (приём накладных). Закрытие заявки склад не трогает.
- **Печать/экспорт заявки, номер-счётчик заявки** — не в этом этапе (идентификатор = `id` + дата).
- **Карточка поставщика, редактор поставщиков** — поставщик остаётся минимальной сущностью (11.9.2).
- **Правила отрицательных остатков** (разд. 17) — этап 8.

## 4. Модель данных

Новый модуль `app/models/order.py`. Отдельные таблицы (по образцу `receipt`/`return_receipt`),
чтобы не трогать протестированные инварианты продажи/склада.

```python
class OrderStatus(str, Enum):
    open = "открыта"
    closed = "закрыта"

class Order(SQLModel, table=True):          # шапка заявки (макет 11.5)
    id: Optional[int] = PK
    supplier_id: int = FK supplier.id        # заявка всегда по поставщику (11.9.1)
    store: str                               # магазин (11.4); из settings.store_name
    status: OrderStatus = open
    created_at: datetime                     # дата создания (11.5)
    closed_at: Optional[datetime] = None     # проставляется при закрытии (11.7)

class OrderLine(SQLModel, table=True):       # строка заявки
    id: Optional[int] = PK
    order_id: int = FK order.id
    product_id: int = FK product.id
    needed_quantity: Decimal (quantity_column())   # нужное кол-во для заказа (11.5/11.6)
    comment: Optional[str] = None                   # примечание к позиции (11.5/11.6)
```

- **Кол-во — `Decimal`** через `quantity_column()` (правило CLAUDE.md 1: количество не float).
  Штучные — целое, весовые — дробное.
- **«Текущее количество» и «минимальный остаток» не хранятся** — берутся из `Product` при
  показе (разд. 2 п.3). Строка заявки ссылается на товар по `product_id`.
- **Одна открытая заявка на пару (`supplier_id`, `store`)** — обеспечивается логикой сервиса
  (найти открытую → дописать / создать). В открытой заявке **не более одной строки на товар**
  (повторный проброс обновляет `needed_quantity`).
- Заявки **не пишут `Movement`** и **не меняют `Product.quantity_current`** (правило CLAUDE.md 5
  про движение — только на изменение остатка; заявка остаток не меняет).

Регистрация: импорт в `app/models/__init__.py` (как `Supplier`); `init_db()` уже подтягивает
`app.models`, отдельный счётчик-строка не нужен (номер заявки — `id`).

## 5. Схемы (границе — Pydantic, правило CLAUDE.md 3)

Новый `app/schemas/order.py`, приёмы из `schemas/product.py` (`PositiveDecimal`, валидаторы):

- `OrderLineInput`: `needed_quantity: PositiveDecimal` (> 0), `comment: Optional[str]` (пусто → None).
- Выбор поставщика при пробросе: `supplier_id` (существующий) **или** введённое имя (инлайн-
  создание через `resolve_suppliers`) — реюз, без новой логики поставщиков.
- Нужное количество `> 0` — валидатор (как `quantity_positive`); пустое/невалидное → ошибка на
  границе, до сервиса.

## 6. Сервис (`app/services/order_service.py`)

Складская/заявочная логика вне веб-обвязки (по структуре проекта). Основное:

- `is_order_candidate(product) -> bool` — чистая функция: `is_low_stock(product) or
  product.quantity_current <= 0`. Покрывает оба источника заявки (11.2): достиг минимума
  **или** закончился. (Случай «закончился при `min_stock == 0`», который 5.2 отдал 5.3, ловится
  вторым условием.)
- `list_candidates(session)` — товары-кандидаты, сгруппированные для показа: по поставщику(ам)
  товара + группа «Разный / без поставщика» (11.9.4); отметка «уже в открытой заявке».
- `push_to_order(session, product_id, supplier, needed_quantity, comment)` — проброс (11.9.5):
  найти/создать **открытую** заявку поставщика (store из config), добавить строку или обновить
  существующую по товару. Одна транзакция. Остаток/движение не трогаются.
- `add_manual_line`, `update_line`, `remove_line` — ручное редактирование (11.6).
- `close_order(session, order_id)` — статус → закрыта, `closed_at = now`. Идемпотентная защита:
  закрытую не закрывать/не править повторно.
- `list_open_orders(session)`, `list_closed_orders(session)`, `get_order(session, id)` — чтение
  для панели и архива; вычисляемый показ строки (товар, живые кол-во/минимум, нужное, комментарий).

Реюз: `is_low_stock`, `resolve_suppliers`, `list_active_suppliers`, `product_suppliers`,
`search_products` — ничего из этого не дублируется.

## 10. Журнал решений (зачем так)

1. **Авто = только детект кандидатов, проброс ручной.** Макет 11.9.5 сильнее формулировки
   roadmap; продавец подтверждает каждую позицию и поставщика.
2. **Одна открытая заявка на поставщика.** «Заявка формируется по поставщику» (11.9.1);
   проброс дописывает строку в открытую заявку, повторный по товару — обновляет, не дублирует.
3. **Заявка не трогает склад.** Намерение заказать ≠ приход. Остаток и движение меняет только
   приём (этап 6). Так заявки нельзя перепутать с денежным/складским путём.
4. **Живые кол-во/минимум, не снимок (решение заказчика).** Строка хранит только `needed_quantity`
   и `comment`; остаток/минимум читаются из товара. Проще (нет дублирования полей), но закрытая
   заявка показывает актуальные цифры — принято осознанно; исторический смысл несёт `needed_quantity`.
5. **Магазин — поле из config, без сущности.** 11.4 требует «предусмотреть»; закладываем поле,
   не раздувая v1 отдельной таблицей магазинов.
6. **Поставщик выбирается всегда при пробросе.** 11.9.5 пп. 2–4; «без поставщика» — группа
   кандидатов, а не заявка. Инлайн-создание реюзит `resolve_suppliers`.
7. **Отдельные таблицы заявок.** Как `receipt`/`return_receipt`: не трогаем протестированные
   инварианты продажи и склада.

## 11. Критерии приёмки (`tests/test_orders.py`)

**Детектор (`is_order_candidate`):**
1. Ниже минимума (`min>0`, `qty<=min`) → `True`.
2. Закончился (`qty<=0`) даже при `min_stock == 0` → `True`.
3. Нормальный (`qty>min>0`) → `False`.
4. Весовой товар — сравнение `Decimal` корректно.

**Сервис (проброс/редактирование/закрытие):**
5. `push_to_order` создаёт **открытую** заявку поставщика со `store` из config, строку с нужным
   кол-вом; `Product.quantity_current` не изменён, `Movement` не создан.
6. Второй товар тому же поставщику → **та же** открытая заявка (одна на поставщика), 2 строки.
7. Повторный проброс того же товара → строка **обновлена**, дубля нет.
8. Проброс товара без поставщика с вводом нового имени → поставщик создан инлайн
   (`resolve_suppliers`), заявка привязана к нему.
9. Ручное редактирование: изменить `needed_quantity` и `comment`; удалить строку; добавить
   произвольный товар (не кандидата).
10. `close_order` → статус «закрыта», `closed_at` задан; остаток/движения не изменены; заявка
    остаётся в БД. Правка/повторное закрытие закрытой заявки отклонены.
11. `needed_quantity <= 0` → `ValidationError` (граница).

**HTTP (TestClient):**
12. `GET` панели «Заявки» → кандидаты (низкий/закончился) сгруппированы, «+» присутствует;
    открытые заявки и архив отрисованы.
13. Проброс кандидата с одним поставщиком → строка в заявке (200, перерисовка).
14. Проброс товара с несколькими поставщиками без выбора → форма выбора; с выбором → добавлено.
15. `POST` закрытия → заявка ушла в архив, остаток товара цел.

**Регрессия:**
16. Весь существующий набор тестов зелёный; денежный и складской путь (продажа/возврат/
    инвентаризация) не затронут — заявки не пишут `Movement` и не меняют `quantity_current`.
