# ТЗ — Ручной приём накладной

> Этап **6.1** roadmap (ручной приём: приход существующих товаров через поиск карточек —
> обновление количества и цены; создание карточек для новых).
> Макет: разд. 12.3 (ручной вариант добавления из накладных). Разд. 12.4 (автоматизированный
> приём по скану) — отложен, «под вопросом». Разд. 12.6 (проверка/рекомендуемая цена) — формула
> отложена (этап 8).
> **Зафиксировано (развилки 6.1):** приём — **по одной позиции**, без сущности-документа
> «накладная» (макет 12.3 описывает пооперационную работу с карточкой; ремарка 2.5 — «пока 1
> товар»). На форме приёма правятся **обе цены** (закупочная и продажная).
> Опирается на 0.3 (`create_product`, `ProductCreate`, форма создания) — путь «нового товара»;
> 2.5 (рама вкладок, «Добавить» была заглушкой — здесь наполняется); 3.1 (движок поиска
> `search_products`, фрагмент `_supplier_row.html`, роут `/products/supplier-row`).
> Переиспользует: `search_products` (`app/services/product_search.py`), `create_product` /
> `ProductCreate` (0.3), `Movement`/`OperationType.income`, `PriceHistory`, типы
> `Kopecks`/`PositiveDecimal` (`app/schemas/product.py`), фильтры `money`/`quantity`
> (`app/services/money.py`).
> Источник истины по требованиям — `docs/Макет системы.txt`. При противоречии — сверяться с макетом.

---

## 1. Назначение

После приезда товара его надо занести в базу (макет 12.1). Ручной приём (12.3) идёт **через
поиск карточек**:

- **товар уже в базе** → продавец находит карточку, вводит пришедшее количество, при
  необходимости меняет цену, система обновляет товар (12.3, «Если товар уже есть»);
- **товара нет в базе** → продавец создаёт новую карточку (12.3, «Если товара нет»).

Приём — это **складской путь**: он поднимает остаток и обязан писать движение товара
(CLAUDE.md правило 5). Поднятие остатка + движение + возможная правка цены — **одна
транзакция** (правило 2). Наполняет вкладку «Добавить» рамы (2.5), которая до сих пор была
заглушкой.

## 2. Зафиксированные решения (развилки 6.1)

1. **Приём — по одной позиции, без сущности-документа «накладная».** Макет 12.3 описывает
   пооперационную работу через карточку («ищет → открывает → добавляет количество → меняет
   цену → система обновляет»), а ремарка roadmap 2.5 прямо ограничивает — «пока 1 товар».
   Отдельная таблица «Накладная» + строки **не заводится**. Аудит прихода несёт движение
   товара (`Movement` типа «приход») с датой и количеством — этого достаточно (правило 5).
   Группировка в документ-накладную — возможное будущее расширение (не в этом этапе).
2. **Приход — additive, отдельно от инвентаризации.** Приход **прибавляет** к остатку
   (`quantity_current += пришедшее`) и пишет движение **«приход»** (`OperationType.income`).
   Это не то же, что корректировка из карточки (3.1, `adjust_quantity`): та **выставляет
   абсолютный** остаток и пишет движение «инвентаризация». Приём = «прибыло N», инвентаризация
   = «фактически стало N». Разные операции — разные типы движения.
3. **На форме приёма правятся обе цены (закупочная и продажная).** Макет 12.3 п.5 — «при
   необходимости изменяет цену»; заказчик выбрал обе. Обе цены на форме **необязательны**:
   поле пустое → цена не меняется. Если хотя бы одна цена реально изменилась против текущей —
   пишется **одна** точка `PriceHistory` с новой парой (как в `update_product`, 3.1/0.2 п.11).
   Формула рекомендуемой цены (12.6) — отложена (этап 8); подсветку «ниже рекомендуемой» не
   делаем.
4. **Новый товар — переиспользование 0.3, без дублирования.** Ветка «товара нет» вызывает
   уже готовый `create_product` (0.3), который сам атомарно пишет движение «приход» + первую
   точку `PriceHistory` + связи поставщиков. 6.1 добавляет только точку входа из результата
   поиска и HTMX-обёртку (фрагмент вместо редиректа).

## 3. Границы этапа

### Входит
- **Панель «Добавить»** во вкладке рамы (замена заглушки): поиск товара (движок общий с
  «Товары»/кассой — `search_products`) с результатами под приём.
- **Приход существующего товара** (12.3): форма приёма (пришедшее количество + обе цены,
  предзаполнены текущими), сервис `receive_product` — атомарно: `+= кол-во`, движение
  «приход», при изменении цены — точка `PriceHistory`.
- **Создание карточки нового товара** (12.3) из ветки «не найдено»: переиспользует
  `create_product`/`ProductCreate` (0.3) в HTMX-фрагменте, имя предзаполнено запросом.
- **Схема границы** `ProductReceive` (кол-во > 0; цены — необязательные, при наличии
  проверяются как в `ProductCreate`).
- **Роуты приёма** `app/routes/receiving.py` + шаблоны `app/templates/add/` + `receiving/`.
- **Профильный тест** `tests/test_receiving.py`.

### Не входит
- **Сущность-документ «накладная», приём несколькими строками сразу** — решение 2 п.1
  (по одной позиции). Возможное будущее расширение.
- **Автоматизированный приём по скану накладной (12.4)** — отложен, «под вопросом».
- **Проверка/рекомендуемая цена, подсветка позиции (12.6)** — формула отложена (этап 8).
- **Правка кодов/артикула/поставщиков в форме приёма** — приём меняет только количество и
  цены существующего товара; паспорт правится в карточке (3.1), новый товар заводится формой
  создания (0.3).
- **Правила отрицательных остатков (разд. 17), финальные статусы (разд. 8)** — этап 8.
- **Связь прихода с закрытой заявкой (5.3)** — приём и заявки остаются отдельными путями
  (закрытие заявки склад не трогает, приём трогает); сшивку не делаем.

## 4. Модель данных

**Новых таблиц нет.** Приём переиспользует существующее хранилище:

- `Movement` (`OperationType.income` уже в перечне) — движение «приход» на пришедшее
  количество (`+`).
- `PriceHistory` — точка при изменении цены (та же схема, что при создании/правке).
- `Product.quantity_current`, `price_buy`, `price_sell` — обновляются приходом.

Следствие: **миграции не требуется** (в отличие от 3.1, где добавлялось `extra_info`). Схема
БД не меняется.

## 5. Схема границы (`app/schemas/product.py`)

Валидация до логики (CLAUDE.md правило 3). Новый класс рядом с `ProductCreate`/`ProductEdit`,
переиспользует типы `Kopecks`/`PositiveDecimal` и те же правила:

```python
class ProductReceive(BaseModel):
    """Приход партии товара (макет 12.3). Кол-во обязательно, цены — по желанию.

    Пустое ценовое поле формы → None → цена не меняется. При наличии — те же правила,
    что у ProductCreate: закупка ≥ 0, продажа > 0.
    """
    received_quantity: PositiveDecimal          # пришедшее кол-во, > 0
    price_buy: Optional[Kopecks] = None          # новая закупка (или None = не менять)
    price_sell: Optional[Kopecks] = None         # новая продажа (или None = не менять)

    @field_validator("received_quantity")  → > 0  («Количество должно быть больше 0»)
    @field_validator("price_buy")   → None пропускается, иначе ≥ 0
    @field_validator("price_sell")  → None пропускается, иначе > 0
```

Роут передаёт `price_buy=None`/`price_sell=None`, когда соответствующее поле пустое (пустая
строка → None до схемы, по образцу обработки `min_stock`/`article` в `create_product_route`).

## 6. Сервис (`app/services/product_service.py`)

Дописывается к существующему модулю (рядом с `create_product`/`adjust_quantity`).

### 6.1 `receive_product(session, product_id, received_quantity, price_buy=None, price_sell=None) -> Product`

Приход существующего товара (макет 12.3). **Одна транзакция** (CLAUDE.md правила 2, 5):

1. `product.quantity_current += received_quantity` (приход всегда прибавляет; кол-во > 0
   гарантировано схемой, поэтому движение пишется всегда — в отличие от `adjust_quantity`,
   где дельта может быть 0);
2. `Movement(product_id, datetime=now, quantity=received_quantity, operation_type=income)`
   (знак `+`);
3. `new_buy = price_buy if price_buy is not None else product.price_buy`; аналогично `new_sell`;
   `price_changed = new_buy != product.price_buy or new_sell != product.price_sell`;
4. обновить `product.price_buy/price_sell`; если `price_changed` — добавить **одну**
   `PriceHistory(product_id, now, new_buy, new_sell)` (0.2 п.11);
5. `commit`; `refresh`. Либо всё, либо ничего — полусписанных/полуоприходованных состояний нет.

Отличие от `adjust_quantity` (3.1): та выставляет абсолютный остаток и пишет «инвентаризация»;
здесь — прибавляем и пишем «приход». Это осознанно разные складские операции (решение 2 п.2).

### Новый товар — без нового сервиса
Ветка «товара нет» переиспользует **готовый** `create_product(ProductCreate, session)` (0.3):
он уже пишет приход-движение, первую точку `PriceHistory`, связи поставщиков — атомарно.
6.1 не дублирует эту логику, только вызывает её из HTMX-роута.

## 7. Роуты и шаблоны

### 7.1 Новый роутер `app/routes/receiving.py` (реестр в `app/main.py`)
Свой `Jinja2Templates` с фильтрами `money`/`quantity` (как в `products.py`). Эндпоинты —
HTMX-фрагменты:

| Метод | Путь | Назначение | Ответ |
|---|---|---|---|
| `GET` | `/receiving/search?q=` | Поиск товара для приёма (движок `search_products`) | `receiving/_search_results.html` |
| `GET` | `/receiving/{id}/form` | Инлайн-форма приёма (кол-во + обе цены, предзаполнены) | `receiving/_receive_form.html` (404 если нет) |
| `POST` | `/receiving/{id}` | Оприходовать → `receive_product` | `receiving/_received.html` (успех) / форма с ошибками (422) |
| `GET` | `/receiving/new?name=` | Форма создания карточки (фрагмент, имя предзаполнено) | `receiving/_create_form.html` |
| `POST` | `/receiving/create` | Создать товар → `create_product` | фрагмент успеха / форма с ошибками (422) |

- **Поиск:** `search_products(session, q)`; результат показывает — есть ли товар, остаток,
  цены; в строке кнопка **«Оприходовать»** (`hx-get /receiving/{id}/form`). При пустом
  результате — кнопка **«Создать карточку»** (`hx-get /receiving/new?name={q}`).
- **`POST`-роуты:** валидация схемой (`ProductReceive`/`ProductCreate`) до логики;
  `ValidationError` → **422** + перерисовка формы с ошибками (паттерн `create_product_route`:
  собрать `err["msg"]`, вернуть фрагмент со статусом). `IntegrityError` (дубль QR при
  создании) → 422 с сообщением, как в 0.3.

### 7.2 Панель «Добавить» (`app/routes/shell.py`)
В `section_panel` добавить ветку `key == "add"` → `add/_panel.html` (реальная панель поиска
приёма) вместо заглушки. Убрать флаг `test_field` для «add» (тестовое поле уходит с
наполнением раздела — так и помечено в `_stub.html`/2.5). Ключ `receipts` остаётся заглушкой
до этапа 7. `SECTIONS` не трогаем.

### 7.3 Шаблоны
`app/templates/add/_panel.html` — наполнение вкладки «Добавить»: поле поиска
(`hx-get /receiving/search`, `hx-target #receiving-results`, `hx-trigger keyup changed
delay:300ms`) + контейнер результатов + контейнер формы. По образцу `products/_panel.html`.

`app/templates/receiving/`:
- `_search_results.html` — таблица результатов; в строке «Оприходовать»
  (`hx-get /receiving/{{ p.id }}/form`, цель — общий слот формы). При пустом результате —
  блок «Ничего не найдено» + «Создать карточку» (`hx-get /receiving/new?name={{ query }}`).
- `_receive_form.html` — форма приёма одного товара: пришедшее количество (обязательно),
  закупочная и продажная цены (предзаполнены текущими, необязательны),
  `hx-post /receiving/{{ product.id }}`. Показывает текущий остаток/цены для контекста.
- `_received.html` — подтверждение: новый остаток, действующие цены, «Оприходовать ещё» /
  вернуться к поиску.
- `_create_form.html` — форма создания карточки нового товара **как фрагмент**
  (`hx-post /receiving/create`), имя предзаполнено запросом. Переиспользует ряд поставщика
  `products/_supplier_row.html`, роут `/products/supplier-row`, datalist поставщиков.

**Переиспользование полей создания.** Чтобы не расходиться с формой 0.3, вынести общие поля
создания в partial `products/_create_fields.html` и включать его и в `products/new.html`, и в
`receiving/_create_form.html` (`{% include %}`). Разница только в обёртке (страница+редирект
vs фрагмент+HTMX-ответ) и `action`. Лёгкий рефактор 0.3 без изменения логики.

### 7.4 JS оболочки — не трогаем
Панель «Добавить» самодостаточна на HTMX (поиск → форма → подтверждение — всё внутри панели,
свои id `#receiving-results`, `#receiving-form`). Отдельного слоя в `shell.js` (как `openCard`
для карточки) **не требуется** — «Оприходовать»/«Создать» это обычные `hx-get/hx-post`, не
делегированный клик. Изоляция панелей (2.5): id приёма не пересекаются с кассой/товарами.

## 10. Журнал решений (зачем так)

1. **По одной позиции, без документа-накладной.** Макет 12.3 — пооперационная работа через
   карточку; ремарка 2.5 — «пока 1 товар». Аудит прихода несёт `Movement` (приход). Документ —
   будущее расширение, не раздуваем v1.
2. **Приход прибавляет и пишет «приход»; инвентаризация выставляет и пишет «инвентаризация».**
   Разные складские смыслы — разные типы движения. Не переиспользуем `adjust_quantity`, чтобы
   не смешивать «прибыло» и «стало».
3. **Обе цены на форме, необязательны, история — только при изменении.** Заказчик выбрал обе;
   пустое поле не трогает цену; точка `PriceHistory` пишется лишь на реальное изменение — иначе
   график (3.3) засорялся бы дублями. Формула рекомендуемой цены (12.6) отложена — подсветки нет.
4. **Новый товар — переиспользование 0.3.** `create_product` уже атомарен и покрыт тестами;
   6.1 добавляет вход из поиска и HTMX-обёртку, не дублируя складскую/ценовую логику.
5. **Без новых таблиц и без миграции.** Приём ложится на `Movement`/`PriceHistory`/`Product`;
   схема БД не меняется (проще и безопаснее, чем 3.1).
6. **Панель «Добавить» самодостаточна на HTMX.** Никакой бизнес-логики в JS (CLAUDE.md/2.5);
   `shell.js` не расширяем.

## 11. Критерии приёмки (`tests/test_receiving.py`)

**Сервис (`receive_product`, складской/ценовой путь — приоритет):**
1. Приход поднимает `quantity_current` на пришедшее количество; создано движение «приход»
   (`income`) со знаком `+` и верным кол-вом.
2. Приход без смены цен (обе `None`) → остаток и движение есть, `price_history` **не** пополнена,
   цены не изменились.
3. Приход со сменой закупки → цена обновлена, **одна** новая точка `PriceHistory` с новой парой.
4. Приход со сменой продажи (или обеих) → одна точка `PriceHistory`; смена обеих не плодит две.
5. Весовой товар: пришедшее дробное количество (`Decimal`) прибавляется корректно.
6. Атомарность: остаток, движение и точка истории появляются вместе (одна транзакция).

**Схема границы (`ProductReceive`):**
7. `received_quantity <= 0` → `ValidationError`.
8. `price_buy < 0` → `ValidationError`; `price_sell <= 0` (при наличии) → `ValidationError`;
   `None` цены проходят (не меняем).

**Новый товар (переиспользование 0.3):**
9. `POST /receiving/create` валидное → товар создан, есть движение «приход» и точка
   `PriceHistory` (проверяем, что путь идёт через `create_product`); невалидное → 422 фрагмент.

**HTTP (TestClient):**

14. Весь существующий набор тестов зелёный; денежный/складской путь (продажа/возврат/
