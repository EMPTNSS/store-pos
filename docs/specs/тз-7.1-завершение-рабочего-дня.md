# ТЗ — Документ завершения рабочего дня

> Этап **7.1** roadmap (документ завершения рабочего дня: итоги по дню + детализация по
> товарам, сохранение).
> Макет: разд. 23 (кнопка «Завершить рабочий день», содержание документа, детализация,
> сохранение). Соседний разд. 24 (отчёт за месяц) — этап **7.2**, здесь не делаем, но
> модель данных дня строим так, чтобы месяц суммировал сохранённые дни.
>
> **Контекст: сущность «Рабочий день» (WorkDay) уже реализована** (коммит
> `work-day: shift lifecycle + cashier lock without an open day`), до этого ТЗ. Есть:
> `app/models/work_day.py` (`WorkDay`: `id`, `status`, `opened_at`, `closed_at`;
> `WorkDayStatus` open/closed), `app/services/work_day_service.py`
> (`get_open_day`/`open_day`/`close_day`, инвариант «ровно одна открытая смена» на сервисе),
> `app/routes/work_day.py` (`POST /work-day/open`, `POST /work-day/close`) и строка статуса
> `app/templates/shell/_day_status.html` с кнопками **«Начать рабочий день»/«Завершить
> рабочий день»**. `Receipt.work_day_id` и `ReturnReceipt.work_day_id` (nullable FK →
> `work_day.id`, `index=True`) помечают каждый чек и возврат его сменой; guard в
> `complete_sale`/`complete_return` не даёт пробить без открытой смены. **`close_day` смену
> закрывает, но документа НЕ формирует — это 7.1.**
>
> **Ключевое переиспользование:** себестоимость на момент операции уже реконструируется —
> `buy_price_asof(session, product_id, at)` (`app/services/product_service.py`, этап 3.2).
> 7.1 считает «сумму чистыми» тем же механизмом, без снимка закупки в строке чека.
> Опирается на 1.3 (`Receipt`/`ReceiptLine`), 4.1/4.2 (`ReturnReceipt`) и WorkDay.
> Источник истины — `docs/Макет системы.txt`. При противоречии — сверяться с макетом.

## 1. Назначение

Дать продавцу закрыть рабочий день и получить сохранённый документ по дню (макет разд.
23.1–23.2): сколько продано, сколько заработано чистыми, доп. финансовая информация и
потоварная детализация. Документ **сохраняется** и доступен для просмотра позже (разд. 23.5)
— он же станет единицей для месячного отчёта (7.2).

Документ считается из уже зафиксированных данных денежного пути: завершённых чеков продажи
(1.3) и чеков возврата (4.1/4.2) **той смены, которая закрывается** (отбор по `work_day_id`).
Новых источников денег/склада 7.1 не вводит — только агрегирует и сохраняет снимок.

**Смена уже есть.** 7.1 достраивает недостающий шаг: при закрытии смены (`close_day`, макет
23.1) сформировать и сохранить документ (23.2–23.5) — **одним согласованным действием**, той
же кнопкой «Завершить рабочий день», что уже стоит в оболочке. Вторую кнопку не заводим.

## 2. Зафиксированные решения (развилки 7.1)

Развилки согласованы с заказчиком. Три трогают денежный путь — фиксируем явно.

### 2.1 «Сумма чистыми» = выручка − себестоимость на момент операции, из `price_history`

Строка чека (`ReceiptLine`) снимает только `price_sell`, закупку не хранит. **Схему чека не
меняем.** Себестоимость проданного берём реконструкцией `buy_price_asof` — последняя точка
`price_history` с `datetime <= момент операции` (fallback на текущую `product.price_buy`,
если точек нет). Тот же механизм, которым 3.2 считает чистую прибыль в истории движений, —
переиспользуем, не дублируем снимок в строке.

Отклонено: снимок `price_buy` в `ReceiptLine`. Дал бы иммутабельную прибыль, но правит
протестированный путь продажи ради данных, уже восстановимых из `price_history`.

### 2.2 Граница дня — по смене (WorkDay), отбор операций по `work_day_id`

Рабочий день = **смена `WorkDay`**: `opened_at` (открытие продавцом) → `closed_at`
(закрытие). В документ входят чеки и возвраты **этой смены** — отбор строго по
`Receipt.work_day_id == day.id` / `ReturnReceipt.work_day_id == day.id`, а не окном по
`datetime`. Каждый чек уже помечен сменой guard'ом продажи/возврата, так что отбор точный и
не зависит от календаря/полуночи.

> Ранняя редакция этого ТЗ (до появления WorkDay) выводила границу через `previous_close` и
> окно `period_start < datetime <= period_end`. **Это было обходным путём на время
> отсутствия таблицы смены и теперь отменено** — граница берётся из WorkDay, отбор из
> `work_day_id`. Функции `previous_close`/оконного отбора в 7.1 нет.

`period_start`/`period_end` документа — это `WorkDay.opened_at`/`WorkDay.closed_at` смены.

### 2.3 Возвраты смены вычитаются из итогов

Возвраты, оформленные в ту же смену, **уменьшают** итог продаж и «чистыми»; их сумма и
количество показываются отдельно в доп. финансовой информации. Итог = реальные деньги в
кассе. Товар возврата вернулся на склад — его себестоимость возвращается в расчёт прибыли
(вычитается из себестоимости проданного). См. денежную семантику (разд. 5).

## 3. Границы этапа

### Входит
- Модель хранимого документа дня: `DayReport` (шапка-итоги, привязана к смене
  `work_day_id`) + `DayReportLine` (потоварная детализация продаж) + счётчик номеров
  `DayReportNumberCounter`.
- Сервис `day_report_service`: агрегирование продаж/возвратов смены по `work_day_id`,
  реконструкция себестоимости, **атомарное закрытие смены + сохранение документа одной
  транзакцией**, чтение сохранённых документов.
- Встраивание в **существующую** кнопку «Завершить рабочий день»: `POST /work-day/close`
  теперь закрывает смену И формирует документ (согласованное действие). **Новую кнопку в
  лаунчере не создаём.**
- Наполнение секции **«Чеки за день»** (`receipts`, до сих пор заглушка 2.5): список
  сохранённых документов дня + просмотр любого (разд. 23.5, «доступен позже»).
- Тесты `tests/test_day_report.py`: денежная и складская арифметика дня, отбор по смене,
  атомарность закрытия, сохранение/чтение.

### Не входит
- **Отчёт за месяц** (разд. 24) — этап 7.2; строится поверх сохранённых `DayReport`.
- **Жизненный цикл смены** (открытие/закрытие, guard кассы, инвариант единственной смены) —
  **уже сделан** до 7.1; не переписываем, только достраиваем формирование документа при
  закрытии.
- **Снимок закупки в строке чека** — отклонено (разд. 2.1).
- **Печать документа дня** на устройство — слой печати ESC/POS вне 7.1; документ рендерится
  в UI и сохраняется в БД. Реальная печать — когда появится железо (как в 2.2 для накладной).
- **Экспорт/импорт** документа (разд. 25) — этап 7.3.
- **Правки/удаление сохранённого документа**, повторное формирование по уже закрытой смене —
  документ иммутабелен, как чек (правило 4 CLAUDE.md).

## 4. Модель данных

Новые таблицы. Суммы — `int` в копейках, количество — `Decimal` (железные правила CLAUDE.md).
Значения **зафиксированы на момент закрытия** и не пересчитываются (как `Receipt`).

### 4.1 `DayReport` (`app/models/day_report.py`) — шапка документа
| Поле | Тип | Смысл |
|---|---|---|
| `id` | int PK | |
| `report_number` | int unique | человекочитаемый номер (счётчик, как у чека) |
| `work_day_id` | int unique, FK → `work_day.id` | смена документа (один документ на смену) |
| `opened_at` | datetime | снимок `WorkDay.opened_at` — начало смены |
| `closed_at` | datetime | снимок `WorkDay.closed_at` — момент закрытия |
| `sales_total` | int | Σ `Receipt.total` смены (валовые продажи, реальные деньги, вкл. округление) |
| `returns_total` | int | Σ `ReturnReceipt.total` смены |
| `net_sales` | int | `sales_total − returns_total` — **«итоговая сумма продаж за день»** |
| `cogs_sold` | int | себестоимость проданного (Σ по строкам чеков) |
| `cogs_returned` | int | себестоимость возвращённого (Σ по строкам возвратов) |
| `net_profit` | int | `net_sales − (cogs_sold − cogs_returned)` — **«сумма чистыми»** |
| `rounding_total` | int | Σ `Receipt.rounding` смены (доп. фин. инфо) |
| `sales_count` | int | число чеков продажи |
| `returns_count` | int | число чеков возврата |
| `cash_sales` / `card_sales` | int | продажи по способу оплаты (Σ `Receipt.total`) |
| `cash_returns` / `card_returns` | int | возвраты по способу |

`work_day_id` — `unique`: одна смена → один документ (повторное формирование запрещено,
разд. 3). `opened_at`/`closed_at` — снимок из `WorkDay` для самодостаточности документа
(читается позже без джойна), значения совпадают со сменой (после закрытия смена не меняется).

### 4.2 `DayReportLine` — потоварная детализация продаж (разд. 23.3)
Снимок агрегата по товару за смену; хранится, чтобы документ читался позже без пересчёта.
| Поле | Тип | Смысл |
|---|---|---|
| `id` | int PK | |
| `day_report_id` | int FK → `day_report.id` | |
| `product_id` | int FK → `product.id` | ссылка (для перехода в карточку) |
| `name` | str | снимок названия |
| `unit` | UnitEnum | снимок единицы |
| `quantity_sold` | Decimal | продано единиц за смену |
| `sum_sold` | int | сумма продаж по товару (Σ `ReceiptLine.total`) |
| `cogs` | int | себестоимость проданного по товару |
| `profit` | int | `sum_sold − cogs` — прибыль по товару |

Строки — только по **проданным** товарам (разд. 23.3 — список проданных). Возвраты в
потоварную детализацию не разворачиваем: их вклад учтён в итогах шапки (`returns_total`,
`cogs_returned`) и в доп. фин. информации. Сортировка — по `sum_sold` убыв.

### 4.3 `DayReportNumberCounter` — счётчик номеров
По образцу `ReceiptNumberCounter`/`ReturnNumberCounter`: `id=1`, `last_value`. Инициализация в
`init_db` (`app/database.py`), регистрация в `app/models/__init__.py`.

## 5. Денежная семантика (точная арифметика документа)

Все суммы — `int` копейки. Себестоимость строки — `line_total(buy_price_asof(product_id,
operation.datetime), qty)` (тот же ROUND_HALF_UP-хелпер `money.py`, что и для выручки строки).
Отбор операций смены — по `work_day_id` (разд. 2.2).

- **Валовые продажи:** `sales_total = Σ Receipt.total` по чекам смены (реальные деньги, вкл.
  надбавку округления).
- **Возвраты:** `returns_total = Σ ReturnReceipt.total` по возвратам смены.
- **Итог продаж (нетто):** `net_sales = sales_total − returns_total` — «итоговая сумма продаж
  за день» (разд. 23.3), возвраты вычтены (разд. 2.3).
- **Себестоимость проданного:** по каждой `ReceiptLine` чека смены —
  `line_total(buy_price_asof(product_id, receipt.datetime), qty)`; `cogs_sold` — их сумма.
- **Себестоимость возвращённого:** по каждой `ReturnReceiptLine` возврата смены —
  `line_total(buy_price_asof(product_id, return.datetime), qty)`; `cogs_returned` — их сумма.
- **Чистыми (прибыль):** `net_profit = net_sales − (cogs_sold − cogs_returned)`.
  Смысл: деньги, оставшиеся в кассе, минус себестоимость товара, реально покинувшего склад
  (проданное − возвращённое). Надбавка округления попадает в прибыль естественно (внутри
  `net_sales`).
- **Потоварно** (`DayReportLine`): агрегируем строки чеков смены по `product_id`;
  `quantity_sold` — Σ `ReceiptLine.quantity`, `sum_sold` — Σ `ReceiptLine.total`, `cogs` — Σ
  себестоимостей строк, `profit = sum_sold − cogs`.

Инвариант для теста: `Σ DayReportLine.sum_sold + rounding_total = sales_total`.

## 6. Сервис `app/services/day_report_service.py`

- `compute_day(session, work_day) -> ComputedDay` — чистая агрегация: выбирает
  `Receipt`/`ReturnReceipt` с `work_day_id == work_day.id`, считает все поля разд. 5,
  возвращает dataclass (шапка + список строк). Без записи в БД — тестируется изолированно.
- `close_day_and_report(session) -> DayReport` — **одна транзакция** (правило 2 CLAUDE.md),
  закрытие смены и документ вместе:
  1. `day = get_open_day(session)`; если `None` → `ValueError` («Нет открытого рабочего дня»);
  2. закрыть смену: `day.status = closed`, `day.closed_at = now`;
  3. `computed = compute_day(session, day)`;
  4. инкремент `DayReportNumberCounter` с `with_for_update` (как в `complete_sale`);
  5. вставка `DayReport(work_day_id=day.id, opened_at=day.opened_at,
     closed_at=day.closed_at, …)` + `DayReportLine`; `commit`; `refresh`.
  Закрытие смены и запись документа — в одном commit: не бывает «смена закрыта, документа
  нет». Эта функция **заменяет** прямой вызов `work_day_service.close_day` в роуте
  `/work-day/close` (закрытие смены переезжает внутрь оркестратора 7.1).
- `list_day_reports(session) -> list[DayReport]` — от новых к старым (секция «Чеки за день»).
- `get_day_report(session, id) -> DayReport | None` + загрузка его строк — просмотр
  сохранённого документа.

Смена без единой продажи/возврата закрывается штатно и даёт **нулевой документ** — фиксирует
факт закрытия (двойного учёта нет: следующая смена — другой `work_day_id`).

`work_day_service.close_day` остаётся в кодовой базе (низкоуровневое закрытие/тесты смены),
но рабочий путь кассы идёт через `close_day_and_report`.

## 9. Критерии приёмки (`tests/test_day_report.py`)

**Себестоимость и прибыль (`compute_day`):**
1. Один чек, товар с одной точкой `price_history` → `cogs_sold` = закупка×кол-во,
   `net_profit = net_sales − cogs_sold`.
2. Закупка менялась во времени: продажа берёт закупку, действовавшую **на момент чека**
   (`buy_price_asof`), а не текущую.
3. Весовой товар (`Decimal` кол-во, напр. 1.5 кг) → себестоимость и прибыль дробные,
   округляются `line_total`.

**Возвраты (разд. 2.3):**
4. Возврат смены уменьшает `net_sales` на `returns_total` и `net_cogs` на `cogs_returned`;
   `net_profit` пересчитан корректно.
5. Возврат по товару возвращает его себестоимость в расчёт (знак `cogs_returned`).

**Потоварная детализация:**
6. Продажи одного товара в нескольких чеках агрегируются в одну `DayReportLine`
   (`quantity_sold`, `sum_sold` — суммы), сортировка по `sum_sold` убыв.
7. Инвариант сверки: `Σ DayReportLine.sum_sold + rounding_total = sales_total`.

**Отбор по смене (разд. 2.2):**
8. `compute_day` включает только операции с `work_day_id == day.id`; чеки/возвраты **другой
   смены** и чеки без смены (`work_day_id IS NULL`) не входят.
9. Две последовательные смены дают два независимых документа; операции первой не попадают во
   вторую (нет двойного учёта).

**Атомарность закрытия (`close_day_and_report`):**
10. Успех: смена → `closed` с `closed_at`, `DayReport` (с `work_day_id`, `opened_at`/
    `closed_at` из смены) и все `DayReportLine` сохранены, номер выдан счётчиком
    последовательно; всё в одной транзакции.
11. Нет открытой смены → `ValueError`, ни смена, ни документ не меняются.
12. Смена без продаж/возвратов → закрывается, сохраняется нулевой документ.

**Доп. финансовая информация:**
13. `sales_count`, `returns_count`, `cash_*`/`card_*`, `rounding_total` совпадают с исходными
    чеками/возвратами смены.

**Сохранение и чтение (разд. 23.5):**
14. `get_day_report` возвращает тот же снимок позже; `list_day_reports` содержит документ.

**HTTP (TestClient):**
15. `POST /work-day/close` при открытой смене → 200, смена закрыта, в БД появился `DayReport`;
    ответ ведёт к просмотру документа.
16. `GET /day/reports/{id}` → 200, сохранённый документ; `GET /day/reports` содержит его.

**Регрессия:**
17. Весь существующий набор зелёный — денежный/складской путь продаж, возвратов и жизненный
    цикл смены не затронуты (7.1 только читает их данные и достраивает шаг закрытия).

## 10. Журнал решений (зачем так)

1. **Чистыми — реконструкцией из `price_history`, без снимка в чеке.** Себестоимость на
   момент уже восстановима (`buy_price_asof`, 3.2). Переиспользуем, не правим протестированное
   ядро продажи и не дублируем данные.
2. **День — это смена WorkDay, отбор по `work_day_id`.** Смена уже реализована и метит каждый
   чек; отбор по метке точнее и проще окна по `datetime`. Ранний обходной путь
   (`previous_close` + окно) отменён вместе с появлением таблицы смены.
3. **Возвраты вычитаются из итогов.** Итог = реальные деньги в кассе; себестоимость
   вернувшегося товара возвращается в расчёт прибыли. Валовые и нетто оба видны.
4. **Закрытие смены и документ — одна транзакция.** Правило 2 CLAUDE.md: не бывает «смена
   закрыта, документа нет». Орекстратор 7.1 (`close_day_and_report`) объединяет шаг закрытия
   (был в `work_day_service`) и формирование документа.
5. **Документ иммутабелен, привязан к смене, сохраняется снимком.** Один документ на смену
   (`work_day_id` unique); суммы и строки зафиксированы на момент закрытия, читаются позже без
   пересчёта; месяц (7.2) суммирует сохранённые дни.
6. **Кнопку не дублируем.** «Завершить рабочий день» уже есть; 7.1 наполняет её действие, а не
   плодит второй вход. JS только переключает вкладку (принцип 2.5).
