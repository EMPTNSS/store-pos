ТЕХНИЧЕСКИЙ МАКЕТ СИСТЕМЫ МАГАЗИНА

1. ЭКРАН КАССЫ

1.1 Основная задача

Касса должна уметь:

- считать товар;
- добавить товар в чек;
- убирать товары из чека без сложных действий;
- завершать чек;
- при необходимости формировать накладную при продаже.

1.2 Завершение чека

Чек можно закрыть двумя способами оплаты:

- наличные;
- безналичные.

1.3 Информация в чеке

В чек должна вноситься информация:

- товары;
- итоговая сумма;
- способ оплаты;
- дата и время.

1.4 Накладная при продаже

При завершении покупки в интерфейсе должна быть возможность отметить, нужна ли накладная.

Формат:

- галочка / переключатель "Распечатать накладную".

Если галочка включена:

1. продавец завершает покупку;
2. система сохраняет чек;
3. система печатает чек;
4. система дополнительно формирует накладную;
5. система отправляет накладную на печать.

Если галочка выключена:

- система печатает только обычный чек.

1.5 Ограничение по денежной части

Деньги напрямую к системе не привязываются.

Система не управляет:

- открытием кассы / денежного ящика;
- банковским терминалом;
- онлайн-банкоматом;
- подтверждением оплаты.

Способ оплаты фиксируется только как информация внутри чека.

1.6 Электронное сохранение чека

После завершения покупки и печати чека система должна сохранять электронную версию чека.

Электронный чек должен сохраняться:

- на компьютере;
- или в базе данных;
- или в виде отдельного файла.

Электронная версия чека должна содержать:

- список товаров;
- сумма по каждому товару;
- итоговая сумма;
- способ оплаты;
- дата и время покупки.


2. СКАНЕР

2.1 Формат устройства

Желательный вариант:

- сканер в форме пистолета;
- желательно беспроводной;
- допустимый вариант — проводной сканер с длинным кабелем;
- сканер должен быть удобен для свободного перемещения возле кассы.

2.2 Задача сканера

Основная задача:

- считать QR-код;
- передать считанный код в систему;
- система по этому коду должна определить товар;
- найденный товар должен автоматически добавиться в текущий чек.

2.3 Связь со сценарием кассы

Сценарий:

1. Продавец наводит сканер на QR-код товара.
2. Сканер считывает код.
3. Система получает код.
4. Система ищет товар по коду.
5. Если товар найден — добавляет его в чек.


3. ПОИСК ТОВАРОВ ДЛЯ ЧЕКА

3.1 Назначение

Поиск товаров для чека нужен для случаев, когда продавец не сканирует код товара, а добавляет товар вручную.

3.2 Способы поиска

Продавец может искать товар по:

- названию;
- числовому коду;
- другому идентификатору товара.

3.3 Использование найденного товара

После нахождения товара продавец может:

- добавить товар в текущий чек.

3.4 Отображение статуса товара

В поиске товаров для чека должен быть виден статус товара.


4. ПОИСК ТОВАРОВ ДЛЯ КАРТОЧКИ

4.1 Назначение

Поиск товаров для карточки нужен для просмотра детальной информации о товаре без добавления его в чек.

4.2 Способы поиска

Продавец может искать товар по:

- названию;
- числовому коду;
- другому идентификатору товара;
- QR-коду / штрихкоду.

4.3 Использование найденного товара

После нахождения товара продавец может:

- открыть карточку товара;
- посмотреть информацию о товаре.

4.4 Информация в поиске

Поиск должен показывать:

- есть ли товар в системе;
- есть ли товар в наличии;
- цену товара;
- дополнительную информацию о товаре.


5. КАРТОЧКА ТОВАРА

5.1 Назначение

В системе должен быть способ открыть карточку товара с детальной информацией.

5.2 Идентификация товара

Карточка товара должна содержать:

- название товара;
- артикул;
- числовой код;
- QR-код / штрихкод;
- другие идентификаторы товара.

5.3 Информация в карточке товара

Карточка товара должна показывать:

- наличие товара;
- текущую цену продажи;
- разницу в процентах;
- количество товара;
- цену закупки;
- единицу измерения;
- минимальный остаток;
- статус товара;
- динамику цены закупки;
- динамику цены продажи;
- дополнительную информацию о товаре.

5.4 История и статистика товара

Карточка товара должна показывать:

- сколько товара было продано за всё время;
- сколько товара прибыло по датам;
- сколько товара убыло по датам;
- сколько чистой прибыли товар сделал за всё время.

5.5 Редактирование информации

В карточке товара должна быть возможность изменять часть информации о товаре:

- изменить цену товара;
- изменить количество товара;
- изменить единицу измерения;
- изменить минимальный остаток;
- изменить статус товара;
- изменить дополнительную информацию о товаре.

5.6 Динамика цен

Динамика цены закупки и цены продажи должна отображаться в виде графика / кривой.

5.7 История движений товара

В карточке товара должно быть отдельное окно с движением товара.

В истории движений товара должно отображаться:

- дата движения;
- количество;
- пометка операции.

На данном этапе достаточно отображать движение товара по датам и количеству.


6. СОЗДАНИЕ КАРТОЧКИ ТОВАРА

6.1 Назначение

В системе должна быть возможность создавать новые карточки товаров.

6.2 Когда создаётся карточка товара

Карточка товара создаётся, если товара ещё нет в базе данных.

6.3 Способы перехода к созданию карточки

Создание карточки товара может выполняться:

- через поиск карточек товара;
- при ручном добавлении товара из накладной;
- при автоматизированном добавлении товара из накладной, если система не нашла товар в базе.

6.4 Данные при создании карточки товара

При создании карточки товара вносится информация, которая нужна для карточки товара и базы данных товаров.

К ней относится:

- название товара;
- артикул;
- числовой код;
- QR-код / штрихкод;
- другие идентификаторы товара;
- наличие товара;
- текущая цена продажи;
- количество товара;
- цена закупки;
- единица измерения;
- минимальный остаток;
- статус товара;
- дополнительная информация о товаре.

6.5 Результат

После создания карточки товара новый товар добавляется в базу данных товаров.


7. БАЗА ДАННЫХ ТОВАРОВ

7.1 Назначение

В системе должна быть база данных товаров.

База данных товаров хранит информацию по каждому товару.

7.2 Содержание базы данных товаров

В базе данных товаров должна храниться вся информация, которая используется в карточке товара.

К ней относится:

- название товара;
- артикул;
- числовой код;
- QR-код / штрихкод;
- другие идентификаторы товара;
- наличие товара;
- текущая цена продажи;
- разница в процентах;
- количество товара;
- цена закупки;
- единица измерения;
- минимальный остаток;
- статус товара;
- динамика цены закупки;
- динамика цены продажи;
- дополнительная информация о товаре;
- количество проданного товара за всё время;
- приход товара по датам;
- убытие товара по датам;
- чистая прибыль товара за всё время.


8. СТАТУСЫ ТОВАРОВ

8.1 Назначение

У товаров должны быть статусы.

Статусы нужны для разделения товаров по актуальности и доступности.

8.2 Список статусов

Конкретный список статусов будет определён позже.

8.3 Изменение статуса

Статус товара можно менять в карточке товара.

8.4 Влияние на поиск

Статус товара должен влиять на порядок отображения товаров в поиске.

Примерная логика отображения:

1. активные товары;
2. товары, которых нет в наличии;
3. архивные товары.

Точная логика будет определена позже.

8.5 Отображение в поиске для чека

В поиске товаров для чека должен быть виден статус товара.


9. ЕДИНИЦЫ ИЗМЕРЕНИЯ

9.1 Назначение

У каждого товара должна быть указана единица измерения.

9.2 Расположение

Единица измерения добавляется в карточку товара.

9.3 Возможные единицы измерения

В карточке товара можно указать единицу измерения, например:

- штука;
- килограмм;
- метр;
- литр;
- упаковка;
- комплект.

9.4 Использование

Единица измерения используется:

- при продаже товара;
- при отображении количества;
- при приходе товара;
- при движении товара;
- в чеках;
- в отчётах.

9.5 Пример

Если товар продаётся по весу:

- единица измерения: кг;
- количество в чеке: 1.5 кг.

Если товар продаётся поштучно:

- единица измерения: шт;
- количество в чеке: 3 шт.


10. МИНИМАЛЬНЫЙ ОСТАТОК

10.1 Назначение

Минимальный остаток нужен для контроля товаров, которые заканчиваются.

10.2 Расположение

Минимальный остаток указывается в карточке товара.

10.3 Изменение минимального остатка

Минимальный остаток можно менять в карточке товара.

10.4 Сценарий работы

1. У товара задан минимальный остаток.
2. Система отслеживает текущее количество товара.
3. Если количество товара достигает минимального остатка, система фиксирует это.
4. Система отправляет уведомление.
5. Система добавляет товар в заявки.

10.5 Связь с карточкой товара

В карточке товара должно быть видно:

- текущий остаток;
- минимальный остаток;
- достигнут ли минимальный остаток.


11. ЗАЯВКИ

11.1 Назначение

Заявки нужны для фиксации товаров, которые нужно заказать или пополнить.

11.2 Источник заявки

Товар попадает в заявки, если:

- товар достиг минимального остатка;
- товар закончился.

Также товар можно добавить в заявки вручную.

11.3 Хранение заявок

Заявки должны сохраняться в системе.

Формат хранения будет определён позже:

- в базе данных;
- или в виде отдельных файлов;
- или другим способом.

11.4 Разделение заявок по магазинам

В системе нужно предусмотреть разделение заявок по магазинам.

Каждая заявка должна быть связана с конкретным магазином.

11.5 Содержание заявки

В заявке должно быть видно:

- магазин;
- товар;
- текущее количество товара;
- минимальный остаток товара;
- нужное количество для заказа;
- комментарий / примечание;
- дата создания заявки;
- статус заявки.

11.6 Ручное редактирование заявки

В заявке должна быть возможность вручную:

- добавить товар;
- указать нужное количество;
- изменить нужное количество;
- добавить комментарий;
- удалить товар из заявки.

11.7 Закрытие заявки

При заказе товара заявка закрывается.

11.8 Сохранение закрытых заявок

После закрытия заявка не удаляется.

Закрытая заявка сохраняется в системе и должна быть доступна для просмотра позже.


12. ДОБАВЛЕНИЕ ТОВАРОВ ИЗ НАКЛАДНЫХ

12.1 Назначение

После приезда товара его нужно добавить в базу данных товаров.

12.2 Варианты добавления

В системе должно быть два варианта добавления товара из накладных:

- ручной;
- автоматизированный.

12.3 Ручной вариант

Ручное добавление товара из накладной выполняется через поиск карточек товара.

Если товар уже есть в базе:

1. Продавец ищет карточку товара.
2. Поиск карточки может выполняться по:
   - названию;
   - QR-коду / штрихкоду;
   - другому идентификатору.
3. Продавец открывает карточку товара.
4. Продавец добавляет нужное количество.
5. Продавец при необходимости изменяет цену.
6. Система обновляет информацию по товару.

Если товара нет в базе:

1. Продавец создаёт новую карточку товара.
2. Продавец вносит нужную информацию по товару.
3. Система добавляет новый товар в базу данных.

12.4 Автоматизированный вариант

Статус пункта:

- под вопросом.

Сценарий:

1. Приходит накладная.
2. Продавец сканирует накладную.
3. Система распознаёт товары из накладной.
4. Система показывает позиции накладной в виде таблицы.
5. В таблице отображается старая цена товара.
6. Перед добавлением продавец может изменить цену товара.
7. Если старая цена ниже расчётной рекомендуемой цены, система подсвечивает такую позицию.
8. Подсветка означает, что цену желательно проверить или изменить.
9. Товары, которые уже есть в базе, поочерёдно добавляются с проверкой продавцом.
10. Продавец подтверждает добавление позиций через интерфейс.
11. Для подтверждения может использоваться значок добавления.
12. Товары, которых нет в базе, создаются через карточку товара.

12.5 Принцип подтверждения

Автоматизированное добавление не должно происходить полностью без проверки продавцом.

Продавец должен видеть позиции накладной и подтверждать добавление товаров.

12.6 Проверка цены перед добавлением

Перед добавлением товара из распознанной накладной система должна показывать:

- название товара;
- количество;
- цену из накладной;
- старую цену товара;
- расчётную рекомендуемую цену;
- поле для изменения цены.

Если старая цена ниже расчётной рекомендуемой цены, позиция должна подсвечиваться.

Расчёт рекомендуемой цены выполняется по формуле, которая будет определена позже.


13. ИНВЕНТАРИЗАЦИЯ

13.1 Назначение

Инвентаризация нужна для сверки фактического количества товара с количеством в системе.

13.2 Формат работы

В системе должна быть возможность вручную изменить фактический остаток товара.

13.3 Сценарий

1. Продавец открывает товар.
2. Продавец проверяет фактическое количество товара.
3. Продавец вводит фактический остаток.
4. Система обновляет количество товара.
5. Система фиксирует операцию в движении товара.

13.4 Отображение в движении товара

После изменения фактического остатка в истории движения товара должна появляться отметка, что была выполнена операция корректировки / инвентаризации.

13.5 Содержание отметки

В движении товара должно быть видно:

- дата операции;
- количество;
- пометка операции.


14. ВОЗВРАТ ТОВАРА

14.1 Назначение

В системе должна быть функция возврата товара.

14.2 Запуск возврата

В системе должна быть отдельная кнопка для возврата товара.

Точное расположение кнопки будет определено позже.

14.3 Сценарий возврата

1. Продавец нажимает кнопку возврата.
2. Продавец выбирает товар для возврата.
3. Система оформляет возврат.
4. Возвращённый товар добавляется обратно в количество товара.
5. Система создаёт чек о возврате.

14.4 Документ возврата

После возврата система должна сформировать чек о возврате.

Чек возврата должен сохраняться в системе.


15. ОТМЕНА / ИСПРАВЛЕНИЕ ЧЕКА

15.1 Отмена до завершения покупки

До завершения покупки текущий чек можно закрыть / отменить.

Если покупка ещё не завершена:

- чек можно очистить;
- товары можно убрать из чека;
- количество товаров можно изменить;
- продажа не сохраняется как завершённая.

15.2 Исправление после завершения покупки

После завершения покупки чек напрямую не исправляется.

Если в завершённом чеке есть ошибочный товар, исправление выполняется через возврат.

15.3 Сценарий исправления после покупки

1. Продавец обнаруживает ошибку в завершённом чеке.
2. Ошибочный товар оформляется через возврат.
3. Товар возвращается обратно в количество.
4. Система создаёт чек о возврате.
5. Исправление фиксируется в системе через возврат.

17. ОТРИЦАТЕЛЬНЫЕ ОСТАТКИ

Статус:

- решить позже.

17.1 Назначение

Пункт описывает сценарий, когда товар физически есть, но в системе его остаток равен нулю или становится отрицательным после продажи.

17.2 Предупреждение

Система должна показывать предупреждение, если продажа товара может привести к отрицательному остатку.

17.3 Что нужно решить позже

- как это отображается в карточке товара;
- как это отображается в движении товара.


18. ПРИНТЕР ЧЕКОВ И НАКЛАДНЫХ

18.1 Назначение

После завершения продажи система должна печатать чек.

Если при продаже отмечена накладная, система должна дополнительно печатать накладную при продаже.

18.2 Устройство

Необходимо устройство для печати чеков.

Рабочее название:

- принтер чеков;
- чековый принтер;
- термопринтер чеков.

18.3 Момент печати

Печать чека происходит после нажатия кнопки завершения продажи.

Сценарий:

1. Продавец формирует чек.
2. Продавец выбирает способ оплаты.
3. Продавец нажимает завершение продажи.
4. Система сохраняет чек.
5. Система отправляет чек на печать.

18.4 Содержание чека

На чеке должно отображаться:

- блок магазина / рекламный блок;
- список товаров;
- сумма по каждому товару;
- итоговая сумма;
- способ оплаты;
- дата и время покупки.

18.5 Печать накладной при продаже

Если при завершении покупки включена галочка "Распечатать накладную", система дополнительно формирует и печатает накладную.

18.6 Содержание накладной при продаже

Накладная должна формироваться на основе текущего чека.

В накладной должно отображаться:

- список товаров;
- количество товаров;
- цена по каждому товару;
- сумма по каждому товару;
- итоговая сумма;
- дата и время покупки.

18.7 Электронное сохранение накладной

Если накладная была сформирована, система должна сохранить её электронную версию.

Накладная должна сохраняться:

- на компьютере;
- или в базе данных;
- или в виде отдельного файла.

18.8 Возможность дополнения

Содержание чека и накладной может быть дополнено позже.


19. ПЕЧАТЬ QR / ШТРИХКОДОВ

19.1 Назначение

В системе должна быть возможность печатать QR-коды / штрихкоды для товаров.

19.2 Устройство

Для печати кодов нужно отдельное устройство:

- принтер этикеток;
- принтер QR-кодов / штрихкодов.

19.3 Когда нужна печать кодов

Печать кодов нужна:

- для товаров без заводского идентификатора;
- для товаров на полках;
- для тетрадок / каталогов с товарами, чтобы товары было удобно пробивать.

19.4 Что печатается на этикетке

На этикетке должно быть:

- код;
- цена;
- название товара.

19.5 Использование

После печати код может использоваться:

- на самом товаре;
- на полке;
- в тетрадке / каталоге товаров.


20. ЭКРАН ДЛЯ ПОКУПАТЕЛЯ

20.1 Назначение

С обратной стороны кассы должен быть экран для покупателя.

Экран нужен, чтобы покупатель видел информацию по текущему чеку и ощущал контроль над покупкой.

20.2 Отображение во время продажи

Во время формирования чека экран должен показывать коротко и крупными символами:

- добавленные товары;
- количество;
- цену;
- итоговую сумму.

20.3 Требования к отображению

Информация должна быть:

- крупной;
- короткой;
- понятной;
- без лишних деталей.

20.4 Отображение после завершения покупки

После завершения покупки на экране должно отображаться сообщение:

- "Спасибо за покупку"


21. ТЕРМИНАЛ ДЛЯ БЕЗНАЛИЧНОГО РАСЧЁТА

21.1 Назначение

Для безналичной оплаты должен использоваться отдельный платёжный терминал.

21.2 Формат работы

Терминал работает отдельно от системы.

Сценарий:

1. Продавец видит итоговую сумму в чеке.
2. Продавец вручную вводит сумму на терминале.
3. Покупатель оплачивает безналично.
4. В системе чек закрывается со способом оплаты "безналичные".

21.3 Ограничение

Терминал не привязывается к системе.

Система не должна:

- передавать сумму в терминал автоматически;
- получать подтверждение оплаты от терминала;
- управлять терминалом;
- сверять оплату с терминалом.

Способ оплаты уже фиксируется в чеке внутри системы.


22. ВЕСЫ

22.1 Назначение

Весы используются для товаров, которые продаются по весу.

22.2 Основной формат работы

Весы не привязываются к системе.

22.3 Сценарий работы

1. Продавец взвешивает товар на отдельных весах.
2. Продавец вручную вводит вес в поле количества.
3. Система рассчитывает сумму по товару.

22.4 Требование к весам

Для весов нужна специальная чаша.

22.5 Возможное развитие

23. ЗАВЕРШЕНИЕ РАБОЧЕГО ДНЯ

23.1 Кнопка завершения дня

В системе должна быть кнопка:

- "Завершить рабочий день"

23.2 Назначение

После нажатия система должна сформировать детальный документ по рабочему дню.

23.3 Содержание документа

Документ завершения дня должен содержать:

- итоговую сумму продаж за день;
- сумму чистыми;

- список всех проданных товаров за день;
- количество проданных единиц по каждому товару;
- сумму продаж по каждому товару.

23.4 Детализация

Документ должен показывать информацию:

- общую по дню;
- детальную по товарам.

23.5 Сохранение документа завершения дня

После формирования документ завершения рабочего дня должен сохраняться.

Документ должен сохраняться:

- на компьютере;
- или в базе данных;
- или в виде отдельного файла.

Сохранённый документ должен быть доступен для просмотра позже.

24.1 Формирование месячного отчёта

После завершения последнего рабочего дня месяца система должна формировать отчёт за месяц.

24.2 Источник данных

Месячный отчёт должен формироваться на основе сохранённых данных рабочих дней.

24.3 Содержание месячного отчёта

Месячный отчёт должен содержать:

- итоговую сумму продаж за месяц;
- сумму чистыми за месяц;
- дополнительную финансовую информацию;
- самый продаваемый товар за месяц.

24.4 Ограничение детализации

Месячный отчёт не должен содержать детальный потоварный анализ по всем товарам.

24.5 Сохранение месячного отчёта

После формирования месячный отчёт должен сохраняться.

Отчёт должен сохраняться:

- на компьютере;
