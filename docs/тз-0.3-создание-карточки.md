# ТЗ — Создание карточки товара

> Этап **0.3** roadmap (создание карточки товара).
> Макет: разд. 6 (создание), 5 (карточка), 7 (БД товаров), 9 (единицы), 10 (мин. остаток).
> Опирается на 0.2 (`docs/тз-база-товаров.md`): таблицы `product`, `movement`,
> `price_history`, `ProductCodeCounter`.
> Источник истины — `docs/макет-системы.txt`. При противоречии — сверяться с макетом.

---

## 1. Назначение

Первая операция записи в базу. Дать минимальный способ завести новый товар: форма →
эндпоинт → сервис, создающий товар со всей сопутствующей историей **одной транзакцией**.

В 0.2 построены только таблицы. Здесь впервые пишется бизнес-логика: генерация числового
кода, начальное движение-приход, первая запись истории цен. Готовых сервисов нет — строим тут.

Результат (макет 6.5): новый товар появляется в базе и пригоден для продажи в этапе 1.

---

## 2. Граница этапа

### Входит
- Кнопка/страница «Создать товар» → форма создания.
- Заполнение полей продавцом + сохранение.
- Сервис создания: товар + числовой код + движение-приход + история цен, атомарно.
- Валидация входных данных на границе (Pydantic).

### НЕ входит (отложено)
- **Поиск карточек** — этап 3. Точка входа в 0.3 — простая форма, без поиска.
- **Редактирование** созданного товара — этап 3.1.
- **Приём из накладной** (макет 6.3) — этап 6. Та же форма переиспользуется оттуда из ветки
  «товар не найден», но сценарий приёмки строим не здесь.
- **Авто-рекомендация цены продажи** (макет 12.6) — формула отложена, цену вводит продавец.
- **Уведомление при мин. остатке** (макет 10.4) — этап 5.2. Число храним, ничего не срабатывает.
- **Генерация/печать QR-штрихкода** (макет 19) — этап 7.4. Поле `qr_code` опционально, не генерируем.
- **Полный список статусов** (макет 8.2) — этап 8. Хватает `активный`/`архивный` из 0.2.

---

## 3. Точка входа

Простая страница с формой «Создать товар», открывается по кнопке/ссылке. Поиска нет.

- `GET /products/new` — отдаёт HTML-форму (Jinja2 + HTMX).
- `POST /products` — принимает данные, создаёт товар, возвращает результат.

Имена роутов и полей — английские. Текст формы для продавца — русский.

---

## 4. Поля формы

Два класса: что вводит продавец и что ставит система. Продавец видит и заполняет только левый блок.

### 4.1 Заполняет продавец

| Поле | Тип | Обяз. | Примечание |
|---|---|---|---|
| `name` | str | да | Название товара. Непустое. |
| `article` | str | нет | Артикул поставщика. Nullable, **без `UNIQUE`** (пересекается у разных поставщиков). Нужен для сопоставления с накладной в этапе 6. |
| `price_buy` | int (копейки) | да | Цена закупки. Целое в копейках (100.00 ₽ = `10000`). ≥ 0. |
| `price_sell` | int (копейки) | да | Цена продажи. Целое в копейках. > 0. Вводится руками. |
| `quantity` | Decimal | да | Начальный остаток. **> 0** — это начальный приход. |
| `unit` | enum | да | Единица измерения, выпадающий список из 7 значений (п. 5). |
| `min_stock` | Decimal | нет | Мин. остаток. По умолчанию `0` (= без отслеживания). |
| `qr_code` | str | нет | Заводской QR/штрихкод, если есть. Иначе пусто. |

### 4.2 Ставит система (продавец не трогает)

| Поле | Значение |
|---|---|
| `id` | автоинкремент |
| `numeric_code` | генерируется из счётчика (п. 6) |
| `status` | `активный` по умолчанию |
| `quantity_current` | выставляется через движение-приход, = `quantity` |
| `created_at` | текущая дата/время |

Статус в форме не показываем: новый товар всегда `активный`. Перевод в архив — редактирование (этап 3.1).

Наличие не вводится и не выбирается: «нет в наличии» вычисляется из нулевого остатка,
отдельным полем не хранится (решено в 0.2).

---

## 5. Единицы измерения

`UnitEnum` — **семь** значений:

```python
class UnitEnum(str, Enum):
    piece = "шт"
    kg = "кг"
    meter = "м"
    liter = "л"
    pack = "упак"
    pair = "пара"
    bottle = "бутылка"
```

Значения справа попадают в базу, чеки и отчёты. В форме — выпадающий список, выбор обязателен.
Граммы отдельной единицей не вводим: вес ведём в кг, `Decimal(10,3)` держит граммовую точность
(`0.250` кг).

---

## 6. Числовой код (`numeric_code`)

Генерируется системой при создании, в **той же транзакции**, что создаёт товар.

- Источник — таблица-счётчик `ProductCodeCounter` (засеяна в `init_db()` из 0.2).
- При создании: взять текущее значение счётчика **с блокировкой строки**
  (`with_for_update()`), нарастить, присвоить товару, сохранить счётчик — всё в одной
  транзакции. Блокировка исключает выдачу одного кода двум товарам при параллельном создании.
- Код — строка (сохранять ведущие нули). Формат значения стабильный.
- На поле стоит `UNIQUE` (из 0.2) — вторая линия защиты на уровне БД.

> SQLite не даёт реального `FOR UPDATE`, но при одной кассе запись идёт в одной транзакции, и
> WAL + сериализация записи это закрывают. Код пишем через `with_for_update()` — при переезде
> на Postgres он заработает как настоящая блокировка, без переписывания (не привязываемся к диалекту).

## 7. Что происходит при сохранении — одна транзакция

Сервис создания (`app/services/`, отдельно от роута) выполняет **атомарно** (правило 2 — либо
всё, либо ничего):

1. Сгенерировать `numeric_code` (счётчик + блокировка, п. 6).
2. Создать строку `product`. `status = активный`, `created_at = now`.
3. Создать запись `movement` — **приход** на `quantity` (п. 8.1).
4. Выставить `product.quantity_current = quantity` (кэш остатка, той же транзакцией).
5. Создать одну запись `price_history` с обеими ценами (п. 8.2).
6. Commit.

Ошибка на любом шаге → rollback: в базе не остаётся ни товара, ни движения, ни израсходованного
кода счётчика. Полусозданных состояний быть не должно.

Поскольку `quantity > 0` всегда (товар заводят, когда он физически приехал), ветки «создать без
движения» нет: движение-приход пишется всегда.

## 8. Структура движения и истории цен

### 8.1 Запись движения (`movement`)

Начальный приход:

| Поле | Значение |
|---|---|
| `product_id` | id созданного товара |
| `operation_type` | `приход` (`OperationType.income`) |
| `quantity` | `+quantity` (со знаком плюс) |
| `datetime` | дата/время |

Движение хранит только дату, количество и тип операции (как в макете 5.7 — «достаточно
отображать движение по датам и количеству»). **Цену закупки в движение НЕ пишем** — снимок
закупочной цены на момент прихода фиксируется в `price_history` (п. 8.2). Дублировать в
`movement` не нужно: прибыль (этап 3) считается из снимка закупочной цены в строке чека
(этап 1.3), а не из движения; историю движений цена не использует. Модель `Movement` остаётся
без изменений.

Тип движения — enum `OperationType`: `приход`, `продажа`, `возврат`, `инвентаризация` (как в
коде). В 0.3 используется только `приход`, остальные подключаются в своих этапах.

> Партионный учёт (отдельная таблица поставок с ценой, поставщиком, накладной, сроком годности)
> — к нему вернёмся в этапе 6, если приёмка накладных это потребует. Для v1 партионная
> себестоимость не ведётся: используем снимок цены карточки в момент продажи.

### 8.2 Запись истории цен (`price_history`)

Схема «широкая»: одна строка хранит **обе** цены на момент времени. При создании пишем **одну**
строку — origin-точку обеих кривых для будущих графиков (макет 5.6, этап 3.3):

| Поле | Значение |
|---|---|
| `product_id` | id товара |
| `datetime` | дата/время |
| `price_buy` | `price_buy`, копейки |
| `price_sell` | `price_sell`, копейки |

Пишем сразу, а не на первом редактировании: начальные цены — первая точка обеих кривых. Без неё
график в этапе 3 начнётся «из ниоткуда». Стоит дёшево (одна вставка в той же транзакции).

> Эволюция (не сейчас): когда на приёмке меняется только цена закупки, новая строка пишется с
> повторённой ценой продажи — лишняя точка на кривой продажи, безвредна. «Длинная» схема (тип
> цены + значение) — только если понадобится идеальная чистота. Для v1 хватает «широкой».

## 9. Валидация на границе (Pydantic)

Все входные данные проверяются Pydantic-моделью **до** сервиса (правило 3):

- `name` — непустая строка.
- `quantity` — `Decimal`, **строго > 0**. Пусто/0/отрицательное → отказ.
- `price_sell` — `int` ≥ 1 (копейки). `price_buy` — `int` ≥ 0.
- `unit` — обязательно одно из `UnitEnum`. Иначе отказ.
- `min_stock` — `Decimal` ≥ 0. Не задано → `0`.
- `article` — необязательное, без проверки уникальности.
- `qr_code` — необязательное; если задано, по нему `UNIQUE`, дубль → отказ.
- Деньги приходят с формы как валюта («100.50») и конвертируются в копейки (`int`) на границе.
  Внутрь логики float не проникает.

## 10. Эндпоинты

| Метод | Путь | Назначение |
|---|---|---|
| `GET` | `/products/new` | Отдать HTML-форму создания |
| `POST` | `/products` | Принять данные, создать товар, вернуть результат |

Роут (`app/routes/`) тонкий: принять → провалидировать Pydantic-моделью → вызвать сервис →
вернуть ответ. Вся логика создания — в сервисе (`app/services/`), не в роуте.

- Успех: редирект или HTMX-замена с подтверждением (показать `numeric_code` нового товара).
- Ошибка валидации: вернуть форму с сообщением, введённые данные не теряются.

## 11. Критерии приёмки (pytest)

Денежный и складской путь покрыть тестами.

**Создание**
1. Валидный POST создаёт товар; строка `product` в БД с верными полями.
2. `numeric_code` сгенерирован, непустой, `UNIQUE`; при создании двух подряд код растёт.
3. `status` нового товара = `активный`.

**Движение и остаток**
4. Создаётся ровно одна `movement` типа `приход` (`OperationType.income`); `quantity` = введённому, цены в движении нет.
5. `product.quantity_current` равен введённому количеству.

**История цен**
6. В `price_history` одна запись с верными `price_buy`, `price_sell` (копейки) и `datetime`.

**Валидация**
7. `quantity` = 0 или отрицательное → отказ, в БД ничего не создано.
8. Пустое `name` → отказ.
9. `unit` не из enum → отказ.
10. Пропущенные `article`/`min_stock` → товар создаётся; `min_stock` = 0.

**Атомарность**
11. Искусственный сбой любого шага сервиса (например, записи движения) → полный rollback: ни товара, ни движения, ни израсходованного значения счётчика.

**Счётчик**
12. `ProductCodeCounter` инкрементится в той же транзакции; значение берётся через `with_for_update()`.

## 12. Журнал решений (зачем так)

1. **`article` — необязательный, без `UNIQUE`.** Поиск закрыт `numeric_code` и штрихкодом;
   артикул есть не у всех и пересекается у разных поставщиков. Оставлен ради сопоставления с
   накладной в этапе 6. *Изменение против 0.2: `str` → `Optional[str]`.*
2. **`min_stock` — необязательный, дефолт 0.** Не каждому товару нужен порог пополнения.
   *Изменение против 0.2: добавлен `default=Decimal("0")`.*
3. **`price_history` пишем при создании** — одна строка-origin для графиков.
4. **`extra_info` не добавляем** — исключено из формы и из макета (решение владельца).
5. **`created_at` добавлено** в `product`.
6. **`UnitEnum` — 7 значений** (убран `компл`, добавлены `пара`, `бутылка`).
7. **Количество обязательно и > 0** — товар заводят только по факту прихода.

## 13. Состояние моделей (на момент ТЗ — готово)

- `Product`: `article` → `Optional`/без unique ✅; `min_stock` → `default=Decimal("0")` ✅;
  `extra_info` не добавляем; `created_at` добавлено ✅.
- `UnitEnum`: 7 значений ✅.
- `Movement`: остаётся **без изменений** (`product_id`, `datetime`, `quantity`, `operation_type`).
  Цену закупки в движение не добавляем — снимок живёт в `price_history` (см. п. 8.1).
- `PriceHistory`: «широкая» схема (`datetime`, `price_buy`, `price_sell`) ✅.
- Миграции не нужны: товаров и движений в базе ещё нет.

> Синхронизировать ТЗ 0.2 и макет: изменённая обязательность `article`/`min_stock`, удаление
> `extra_info` — чтобы источники истины не разъехались.
