# ТЗ — Экран кассы

> Этап **1.1** roadmap (экран кассы).
> Макет: разд. 1.1–1.3 (задача кассы, завершение, состав чека), 15.1 (отмена до завершения).
> Опирается на 0.2/0.3 (`docs/тз-0.2-база-товаров.md`, `docs/тз-0.3-создание-карточки.md`):
> таблица `product` с полями `numeric_code`, `name`, `unit`, `price_sell`, `quantity_current`.
> Источник истины — `docs/Макет системы.txt`. При противоречии — сверяться с макетом.

---

## 1. Назначение

Первый кусок главного денежного пути. Дать экран кассы, который ведёт **текущий
(незавершённый) чек**: добавить товар, изменить количество, убрать товар, видеть итоговую
сумму, очистить/отменить чек.

Ключевое: на этом этапе чек — это **живое состояние до завершения продажи**. Он
**не сохраняется** в базу и **не трогает остаток** товара. Сохранение чека, оплата, списание
остатка и запись движения — следующий этап (1.3). Здесь строится только механика корзины и
экран вокруг неё.

Результат: на экране кассы можно собрать чек и увидеть корректный итог; склад при этом не
меняется.

---

## 2. Граница этапа

### Входит
- Экран кассы (`GET /cashier`).
- Добавление товара в чек по **числовому коду** (точное совпадение `numeric_code`).
- Объединение одинаковых товаров в одну строку с суммированием количества.
- Изменение количества строки (целое и дробное — весовые товары).
- Удаление строки.
- Отмена/очистка всего незавершённого чека (макет 15.1).
- Подсчёт суммы строки и итоговой суммы (деньги — `int` в копейках).
- Валидация количества на границе (Pydantic).

### НЕ входит (отложено)
- **Поиск товара по названию и UX сканера** — этап 1.2. Здесь только минимальный ввод по
  числовому коду; полноценный поиск и сценарий сканера достраиваются поверх этой механики.
- **Завершение продажи** — этап 1.3: способ оплаты, сохранение чека, списание остатка,
  движение товара — всё одной транзакцией. Модель `Receipt`/`ReceiptLine` вводится там, не здесь.
- **Печать чека и накладной** (макет 1.4, 18) — этап 2.
- **Экран покупателя** (макет 20) — этап 2.3.
- **Поведение «товар не найден»** (макет 16) — отложено. Пока просто короткое сообщение во
  фрагменте, без эффектов на БД.
- **Правила отрицательных остатков** (макет 17) — отложено. Остаток в 1.1 не проверяется и не
  меняется, поэтому ветка не возникает.

## 3. Где живёт чек

Одна касса = один активный чек. Состояние корзины держим **в памяти процесса**
(модуль-синглтон в `app/services/cart.py`). Новых таблиц в БД нет.

- Плюс: не плодим черновики в базе, остаток и движения не затрагиваются до завершения (1.3);
  модель `Receipt` приходит ровно тогда, когда нужна.
- Минус: незавершённый чек теряется при перезапуске приложения. Для **незавершённой** продажи
  это приемлемо.
- При появлении нескольких касс на одну базу глобальный синглтон станет точкой переделки
  (состояние чека привяжется к кассе/сессии). Для v1 (одна касса) — достаточно.

## 4. Модель корзины

`CartLine` — строка чека, снимок товара на момент добавления:

| Поле | Тип | Примечание |
|---|---|---|
| `line_id` | int | Инкрементный id строки внутри чека |
| `product_id` | int | Ссылка на товар (для объединения дублей) |
| `name` | str | Снимок названия |
| `unit` | `UnitEnum` | Снимок единицы измерения |
| `price_sell` | int (копейки) | **Снимок цены продажи** на момент добавления |
| `quantity` | Decimal | Количество (целое или дробное) |
| `total` (вычисляется) | int (копейки) | Сумма строки = `line_total(price_sell, quantity)` |

**Снимок цены.** Цена берётся из `Product.price_sell` при добавлении и дальше не
пересчитывается, даже если карточку товара отредактируют. Чек считается по цене на момент
пробития.

`CartView` — то, что отдаётся в шаблон: список строк + `subtotal`, `rounding`, `grand_total`
(см. п. 5.2).

---

## 5. Деньги — округление

Деньги — `int` в копейках, никогда float (правило 1). Два уровня округления:

### 5.1 Сумма строки — до целой копейки

- Хелпер `line_total(price_kopecks: int, quantity: Decimal) -> int` в `app/services/money.py`.
- Округление **ROUND_HALF_UP до целой копейки** — то же правило, что при разборе цен
  (`_parse_kopecks` в 0.3). Обязательное: суммы хранятся в копейках.
- Строки остаются точными до копейки (49.995 → 49.99/50.00).

### 5.2 Итог чека — вверх до целой ₽

Открытый вопрос из CLAUDE.md (округлять весовую сумму до копейки или до целой валюты)
**решён**: округляем **итог чека вверх до целой денежной единицы (₽)**.

- Хелпер `round_total_up(kopecks: int) -> int` в `app/services/money.py` — ceil до 100 копеек.
- Покупатель платит ровную сумму без копеечной сдачи; переплата **≤ 1 ₽ на чек** и всегда
  в пользу магазина.
- Округляется **итог**, а не каждая строка → переплата ограничена ~1 ₽ на весь чек, а не
  накапливается по позициям.
- Разница `grand_total - subtotal` выносится в чек отдельной строкой **«Округление»**, чтобы
  итог сходился со строками.

`CartView` отдаёт три суммы: `subtotal` (точная сумма строк), `rounding` (надбавка, ≥ 0),
`grand_total` (итог к оплате = `subtotal`, округлённый вверх до ₽). Оба хелпера переиспользуются
в 1.3 при сохранении чека.

> Для 1.3: при сохранении чека `subtotal`, `rounding` и `grand_total` (а также `price_sell` и
> `total` каждой строки) записываются как **зафиксированные** значения, а не пересчитываются
> при печати/просмотре. Иначе смена правила округления или правка карточки «поедут» по уже
> завершённым чекам (правило 4 — завершённый чек не редактируется).

## 6. Поведение операций

| Операция | Поведение |
|---|---|
| Добавить товар | Если товар уже в чеке — увеличить количество его строки; иначе новая строка. |
| Изменить количество | Заменить количество строки. Должно быть **> 0**. |
| Убрать строку | Удалить строку из чека. |
| Очистить чек | Удалить все строки (отмена незавершённой продажи, макет 15.1). |

Дубликаты объединяются: повторный ввод того же числового кода (или повторный «скан» в 1.2)
наращивает количество существующей строки — привычное поведение кассы.

## 7. Добавление товара (минимальный ввод)

Полноценный поиск и сканер — этап 1.2. В 1.1 — минимум, достаточный, чтобы экран можно было
пощупать:

- Поле «числовой код товара» + кнопка «Добавить».
- Поиск **точным совпадением**: `select(Product).where(Product.numeric_code == code)`.
- Найден → добавить в чек (с объединением дублей).
- Не найден → короткое сообщение во фрагменте, чек не меняется (макет 16 отложен).

Сканер «печатает» код в это же поле и шлёт Enter (HID-клавиатура) — отдельной интеграции не
требует; полноценный сценарий сканера оформляется в 1.2.

## 8. Эндпоинты

Роут (`app/routes/cashier.py`) тонкий: принять → (для количества) провалидировать Pydantic →
вызвать сервис корзины → вернуть HTML. Логика корзины — в сервисе, не в роуте.

| Метод | Путь | Назначение | Ответ |
|---|---|---|---|
| `GET` | `/cashier` | Экран кассы целиком | `cashier/index.html` |
| `POST` | `/cashier/items` | Добавить товар по `numeric_code` (Form) | фрагмент `cashier/_cart.html` |
| `POST` | `/cashier/items/{line_id}/quantity` | Изменить количество строки (Form) | фрагмент |
| `POST` | `/cashier/items/{line_id}/delete` | Убрать строку | фрагмент |
| `POST` | `/cashier/clear` | Очистить чек | фрагмент |

HTMX: каждая мутация возвращает перерисованный фрагмент корзины (`hx-target="#cart"`,
`hx-swap="innerHTML"`). Имена роутов/полей — английские, текст для продавца — русский.

## 9. Интерфейс (HTMX + Jinja2)

- `templates/cashier/index.html` (наследует `base.html`): поле «числовой код» + «Добавить»,
  контейнер `#cart`, в который инклюдится фрагмент.
- `templates/cashier/_cart.html`: таблица строк (название, цена, количество с inline-формой
  изменения + единица, сумма строки, кнопка «убрать»), блок итогов (подытог и строка
  «Округление» показываются только при ненулевом округлении, иначе один «Итого»), кнопка
  «Очистить чек», опциональное сообщение об ошибке добавления. Пустой чек — подсказка «Чек пуст».
- Деньги выводятся через Jinja-фильтр `money` (копейки → `148.50`).

Минимум JS: только HTMX, без JS-фреймворков (CLAUDE.md).

## 10. Валидация на границе (Pydantic)

Все входные данные проверяются до сервиса (правило 3):

- `quantity` (изменение количества) — схема `CartQuantity`, поле `PositiveDecimal`,
  **строго > 0**. Пусто/0/отрицательное/не число → отказ, сообщение во фрагменте, чек не меняется.
- `numeric_code` — строка; пустая → сообщение «введите код»; ненайденный код → сообщение
  «не найден».
- Дробное количество допускается (весовые товары, макет 22). Округление денег — в `line_total`.

Переиспользуется `PositiveDecimal` из `app/schemas/product.py` (0.3).

## 11. Критерии приёмки (pytest)

Денежный и складской путь покрыть тестами.

**Деньги (`line_total`)**
1. Штучное целое: 3 × 20.00 ₽ = 60.00 ₽ (`6000`).
2. Весовое с округлением вверх: 1.5 × 33.33 ₽ = 49.995 → 50.00 ₽ (`5000`).
3. Весовое ровное: 1.2 × 10.10 ₽ = 12.12 ₽ (`1212`). Результат — `int`.

**Округление итога (`round_total_up`)**
3a. Кратное ₽ не меняется (`4000` → `4000`).
3b. Вверх до целой ₽ даже на копейку: `8601`/`8650`/`8699` → `8700`. Ноль → ноль.

**Корзина (сервис)**
4. Добавление нового товара создаёт строку; количество по умолчанию 1; итог верный.
5. Повторное добавление того же товара объединяет в одну строку, количество суммируется.
6. Разные товары — отдельные строки.
7. `set_quantity` меняет количество и итог; дробное количество считается верно.
8. `set_quantity` ≤ 0 → отказ (`ValueError`).
9. `remove` убирает строку; `clear` опустошает чек.
10. **Снимок цены**: после изменения `price_sell` товара сумма строки не меняется.
11. `subtotal` равен сумме `total` всех строк; `grand_total` — округлён вверх до ₽,
    `rounding` = разнице (напр. `8650` → `8700`, `rounding` `50`). Пустой чек — все нули.

**HTTP-слой**
12. `GET /cashier` → 200, есть «Касса».
13. Добавление по `numeric_code` возвращает фрагмент со строкой и итогом.
14. Несуществующий код → сообщение «не найден», корзина не падает (остаётся «Чек пуст»).
15. **Склад не трогается**: после операций кассы `product.quantity_current` не изменился.
16. `clear` опустошает чек («Чек пуст»).

## 12. Журнал решений (зачем так)

1. **Чек — в памяти, не в БД.** Незавершённая продажа не должна плодить черновики и трогать
   склад. Модель `Receipt` вводится в 1.3, когда продажа становится постоянной.
2. **Дубликаты объединяются.** Привычное поведение кассы: проскан/ввёл дважды — количество 2.
3. **Минимальный ввод по числовому коду.** 1.1 даёт механику корзины; поиск по названию и
   сканер — 1.2 поверх неё, без переписывания.
4. **Снимок цены при добавлении.** Чек считается по цене на момент пробития; правка карточки
   не задним числом меняет уже собранный чек.
5. **Округление.** Строки — ROUND_HALF_UP до копейки (согласовано с `_parse_kopecks`). Итог
   чека — **вверх до целой ₽** (разница в строке «Округление»). Округляем итог, а не строки:
   переплата ≤ 1 ₽ на чек, не накапливается. Закрывает открытый вопрос CLAUDE.md.
6. **Склад и движение не трогаем.** Подтверждается тестом (`quantity_current` неизменен) —
   граница с 1.3 проведена явно.
